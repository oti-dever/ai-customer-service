## 一、软件初期范围
**实现功能**：

+ 简单登录/注册（自定义用户名+密码，本地 SQLite 存储，用户量不大，暂不用 MySQL）  
+ 主窗口：左侧平台列表，右侧聚合主工作区  
+ 将 **外部客服窗口（千牛PC客户端、拼多多网页等）嵌入到本软件主窗口中**  
+ 有客户咨询时，**将各平台会话聚合到“聚合对话接待”界面，人工查看并手动回复**（暂不接 AI 自动回复）

**技术可行性结论**：

+ 登录/注册、主窗口布局、平台列表、聚合对话界面等：**Qt 6 可实现**。
+ 用户/会话/消息等数据持久化：**SQLite + QtSql 足够**。
+ 外部窗口嵌入（千牛等PC客户端）：在 Windows 上通过 **Win32 窗口句柄 + Qt 的 **`QWindow::fromWinId`** / **`QWidget::createWindowContainer`** 等机制可实现**。
+ “消息实时推送到聚合对话接待”：**在技术上可行，但是复杂度最高的部分**，需要按平台分别实现：
    - 优先使用 **官方开放接口（HTTP/WebSocket）** 的平台；
    - 不提供接口的平台，则需要 **RPA/Windows UI Automation/OCR 等方式“读屏+模拟操作”**，属于中高难度、对维护成本要求较高。
+ 总体来看，**当前技术选型不是限制因素，真正的难点在于“各电商平台的开放程度和对接策略”**。初期先做一个“人工回复 + 聚合窗口”的 MVP，再逐步接 AI 和复杂平台。

**使用场景说明（内部阶段）**：

本软件在当前阶段定位为**公司团队内部使用的效率工具**，主要服务对象是少量客服同事。每位同事平时需要同时使用千牛、拼多多等平台（各自名下的店铺账号）接待客户，存在频繁切换窗口、难以统一查看待处理会话的问题。本系统的目标是在不改变原有店铺账号体系的前提下，将这些外部客服窗口与消息“集中到一个桌面应用里”，帮助同事更高效地处理咨询。短期内不对外售卖，也不承载第三方商家数据，后期如要演进为对外 SaaS（通过互联网提供的软件服务模式，用户无需安装和维护本地软件，只需按需访问，核心理念：用户不用买软件，不用管服务器，按月/年交费，像用电一样随时随地使用软件），再根据实际情况升级为企业/服务商开发者并补充合规要求。

## 二、软件功能范围界定
**初期只做下面这些**：

+ **登录/注册**
    - 用户名 + 密码注册、登录；
    - 用户信息保存在本地 SQLite。
+ **主窗口与平台列表**
    - 左侧：**在线平台**、**离线平台**列表（千牛、京麦、抖店、小红书商家、得物商家等，先展示为“可接入项”）。
    - 中间：默认“系统就绪”页，提示用户从左侧选择平台或通过按钮启动平台应用。
    - 顶部：标签页（多接待窗口）可先简化为 1–2 个标签。
+ **外部客服窗口嵌入**
    - 已安装并运行的 **千牛 / 拼多多商家工作台** 等 PC 客户端窗口，能够 **嵌入到主窗口的工作区中**；
    - 支持：从左侧点击某平台 → 如果未运行则启动平台客户端 → 找到其窗口 → 嵌入到当前标签页。
+ **聚合对话接待（人工模式）**
    - 独立的“聚合接待”界面：
        * 左侧：会话列表（来自各平台）。
        * 中间：当前会话消息气泡列表。
        * 右侧：客户信息/订单简要信息（初期可简化为占位）。
    - 当前阶段：**只实现人工查看 + 手动输入文本回复，发送到对应平台**。
    - AI 自动回复、机器人管理、知识库管理等放在后续阶段。

## 三、整体架构设计
### 3.1 分层结构
+ **UI 层（**`src/ui/`**）**
    - `LoginWindow`：登录/注册窗口。
    - `MainWindow`：主窗口，包含左侧平台列表、顶部标签栏、中间嵌入区域、状态栏。
    - `AggregateChatWindow`：聚合对话接待界面。
+ **业务核心层（**`src/core/`**）**
    - `AuthManager`：用户登录/注册逻辑。
    - `PlatformManager`：管理支持的平台列表及其状态（在线/离线、已绑定账号等）。
    - `ConversationManager`：统一管理所有会话、消息、未读数等。
    - `MessageRouter`：根据平台、会话路由消息收发。
+ **平台适配层（**`src/services/platforms/`**）**
    - 定义一个统一接口 `IPlatformAdapter`，不同平台（千牛、京麦、抖店等）有各自实现：
        * `connect()` / `disconnect()`
        * `startListening()` / `stopListening()`
        * `sendMessage(conversationId, text)`
        * 事件：`onIncomingMessage`、`onConversationUpdated` 等。
+ **数据层（**`src/data/`**）**
    - `Database`：SQLite 连接与初始化。
    - DAO 类：`UserDao`、`ConversationDao`、`MessageDao`、`PlatformDao` 等。
+ **工具层（**`src/utils/`**）**
    - `Win32WindowHelper`：Windows 句柄查找、窗口嵌入、尺寸同步。
    - `CryptoUtil`：密码哈希与校验。
    - 日志工具封装 `Logger`（基于 spdlog）。

### 3.2 运行形态
+ 单进程 Windows 桌面应用：
    - **UI 主线程**：Qt 主线程。
    - **平台监听线程/任务**：使用 `QtConcurrent` 或 `QThread` 为平台适配器做 IO 和轮询。
    - 所有与 Win32 API 交互（窗口嵌入、窗口查找）应在合适的线程中进行并通过信号槽与 UI 通信。

### 3.3 消息推送服务与「桌面 + 服务端」部署形态
当采用**官方 API（如淘宝 TOP TMC）**对接平台消息时，推荐引入**常驻消息推送服务（服务端）**，与桌面应用配合形成「桌面 + 服务端」的部署形态：

+ **消息推送服务（服务端）**：
    - 负责与各平台开放接口（如 TOP `taobao.tmc.messages.consume`）建立长轮询/长连接，持续拉取消息。
    - 将平台原始消息转换为内部统一结构（如 `PlatformMessage`），写入服务端数据库（`conversations` / `messages`），并作为**会话与消息的权威数据源**。
    - 通过 WebSocket 或 HTTP 将新消息、发送结果等推送给已连接的桌面客户端。
    - 桌面端发起的“发送消息”请求由服务端接收后，再调用对应平台的发送 API（如旺旺发送接口），密钥（`app_secret`、`session`）仅保存在服务端。
+ **桌面客户端**：
    - 不直接连接平台开放接口，仅与「我方消息推送服务」通信：建立 WebSocket 收推送、通过 HTTP/WebSocket 提交发送请求。
    - 桌面端可保留本地 SQLite 用于用户登录、窗口配置等；会话与消息数据以服务端为准，是否在桌面做本地缓存或全量镜像由《消息推送与回写设计方案》约定（见该文档中的桌面端持久化与同步策略）。

此种形态下，**平台适配器**可存在于服务端（如 `TaobaoTopAdapter` 跑在 server 进程），桌面端对这类平台仅保留“与服务端通信的客户端”角色，不实现直连平台的 `IPlatformAdapter`。

## 四、技术方案
### 4.1 登录/注册与用户管理
**数据设计（SQLite）**：

+ 表 `users`：
    - `id` INTEGER PRIMARY KEY AUTOINCREMENT
    - `username` TEXT UNIQUE NOT NULL
    - `password_hash` TEXT NOT NULL
    - `salt` TEXT NOT NULL
    - `created_at` DATETIME
    - 预留字段：`display_name`、`role`（以后可用）

**密码方案**：

+ 使用 **不可逆哈希**，例如：
    - Qt 内置或结合 OpenSSL，做 `SHA-256( salt + password )`；
    - 存储 `salt` + `hash`，禁止明文。
+ 登录时：取用户记录 → 重新计算 hash → 对比。
+ 补充：初期对密码做个简单异或加密操作就行。

**登录流程**：

1. 启动应用 → 显示 `LoginWindow`。
2. 用户选择“注册”：输入用户名+密码 → 校验 → 写入 `users` 表 → 返回登录。
3. 用户登录成功后，加载用户上下文并打开 `MainWindow`。

> 这一部分完全在 Qt + SQLite 能力范围内，无额外依赖。
>

### 4.2 主窗口与平台列表
**UI 结构（Qt Widgets）**：

+ `MainWindow` 结构建议：
    - 左侧：`QTreeWidget` 或 `QListWidget` 实现 **“在线平台”、“离线平台”** 树形结构；
    - 顶部：`QTabWidget` 用于多个接待标签页；
    - 中间：`QStackedWidget`，每个 tab 内部有一个中心容器（用于嵌入外部窗口或加载 `AggregateChatWindow`）；
    - 底部状态栏：显示“自动回复状态、自动同步状态”等占位信息。

**逻辑**：

+ 左侧点击一个平台：
    - 若是“聚合接待”条目 → 显示 `AggregateChatWindow`。
    - 若是具体平台（千牛/京麦/抖店等） → 尝试启动/附着外部客户端窗口（见下一节）。

### 4.3 外部客服窗口嵌入方案
**期望效果**：当电脑上已经运行了千牛客户端，当点击主窗口左上方的绿色加号按钮（添加新窗口功能），弹出一个置顶的对话框（界面如下图所示），自动搜索窗口并展示在搜索框下方的列表中，当点击选择其中一个窗口项时，为我们自动填写窗口相关信息（除窗口句柄外，其他信息可编辑），再点击确认关闭“添加新窗口”对话框后，离线平台列表中的“千牛”一项被去除，并显示在了在线平台列表下（平台项名称和窗口信息中的“平台名”相同），点击在线平台列表下的“千牛”一项时，千牛客户端窗口被嵌入主窗口中，并可跟随主窗口移动而移动，可以去除千牛客户端窗口自己的某些特性（如最大化/最小化），在视觉效果上看起来像成为了主窗口的一部分，当主窗口最大或最小化时，被嵌入的窗口随之自适应缩放，当主窗口关闭时，被嵌入的窗口恢复为独立窗口，千牛等窗口继续存在，仍可继续使用千牛。

**窗口聚合效果**：

对于不在离线平台列表下的平台项，期望效果：当点击“添加新窗口”对话框后，选择了窗口列表中的一项后，比如“微信”，并点击确认关闭对话框后，“微信”一项被加入到在线平台列表下（平台项名称和窗口信息中的“平台名”相同），此时“微信”还没有被嵌入主窗口，再点击在线列表下的“微信”一项时，“微信”才真正被嵌入到主窗口中。

对于已在离线平台列表下的平台项，期望效果：当点击“添加新窗口”对话框后，选择了窗口列表中的一项后，比如“千牛”，并点击确认关闭对话框后，离线平台列表下的“千牛”一项被去除（界面也对应更新，比如原来离线平台列表下有“千牛、拼多多、抖店”，把“千牛”去掉了，当然每次程序重新运行，离线平台列表下还是有“千牛、拼多多、抖店”），“千牛”一项被加入到在线平台列表下，再点击在线列表下的“千牛”一项时，“千牛”才真正被嵌入到主窗口中。

当在线平台列表下加入了多个平台项时，期望效果：比如在线平台下有“微信、千牛”，没有点击其中任何一项时，谁都不会被嵌入到主窗口中，当点击如“微信”一项，此时“微信”被嵌入到主窗口中，当点击“千牛”，此时切换成“千牛”被嵌入到主窗口中，但“微信”没有被解除嵌入，再点击“微信”一项，再换成“微信”显示，“千牛”隐藏。支持平台项删除，如鼠标右键在线平台下的某个平台项，显示“删除选项”，如果该平台项没有被嵌入，就直接删除平台项，如果已被嵌入，点击删除后，解除嵌入，恢复为独立窗口。当关闭主窗口时，所有被嵌入的窗口均解除恢复为独立窗口。

但并非所有窗口被嵌入后都可正常显示，不是程序逻辑问题，应该和这些窗口的底层实现（如界面渲染逻辑）有关，比如浏览器，浏览器中有多个页面时，再把浏览器嵌入到主窗口中后，浏览器显示会出现黑屏；当“QQ”被嵌入后，QQ界面显示全灰，包括Typora也是，无法正常显示界面。对于这种情况，我们可以对这些窗口不做强制嵌入，也不对这些窗口做过滤操作，依然显示在窗口列表中，只是不做嵌入操作，改为其他的显示方式，比如对于浏览器，可以只是调整浏览器窗口大小，让窗口显示在主窗口右侧中心区域之上，并且可以随主窗口移动而移动，随主窗口最大/最小化自适应缩放，与“被嵌入”的区别是，看起来像是一个随主窗口移动而移动的固定窗口。但也将其加入到在线平台列表下，可与其他平台项切换显示。另一个想法是，通过一个“添加新标签页”按钮（放在顶部栏），在主窗口右侧中心区域内置打开浏览器，也就是在主窗口中使用浏览器，打开浏览器之后，也将浏览器加入到在线平台列表项中，可与其他平台项切换显示。

#### 4.3.1 需求分析细化
+ **用户角色**：客服人员，需在同一界面内同时操作多个电商平台客服客户端。
+ **使用场景**：
    - 已运行千牛、京麦等客户端，希望将其窗口“收进”聚合客服主窗口内统一操作；
    - 通过“添加新窗口”从系统当前已打开窗口中挑选并配置，再在“在线平台”中点击该项完成嵌入。
+ **功能清单**：
    - **发现并列举当前系统顶层窗口**：支持按标题/进程名搜索或过滤；
    - **选择后自动填充并支持编辑**：平台名、进程名、窗口标题、窗口类名、窗口句柄（句柄只读）；
    - **保存配置**：将当前选中的窗口信息保存为“平台”，并从离线列表移至在线列表；
    - **嵌入展示**：点击在线平台项时，将对应外部窗口嵌入主窗口中央容器；
    - **随主窗口联动**：主窗口移动、最大化、最小化、调整大小时，嵌入窗口同步；
    - **关闭联动**：主窗口关闭时，被嵌入窗口变回独立顶层窗口。
+ **非功能需求**：枚举窗口时界面不卡顿（可异步或工作线程）；兼容千牛、京麦等常见客户端；Win32 相关逻辑集中封装（如 `Win32WindowHelper`），便于维护与替换。

#### 4.3.2 交互与数据设计
**“添加新窗口”弹窗流程**：

1. 用户点击主窗口“添加新窗口”（如绿色加号）→ 弹出置顶对话框。
2. 弹窗打开时自动“刷新”：枚举系统顶层窗口，展示在列表中（可选：搜索框按标题/进程名过滤）。
3. 用户点击列表中某一项 → 自动填充“窗口信息”区域（平台名、进程名、窗口标题、窗口类名、窗口句柄）。
4. 用户可编辑除句柄外的字段；平台名将用于显示在在线平台列表中。
5. 用户点击“确定” → 保存该窗口配置（与 `platforms` 表关联），关闭弹窗；左侧“离线平台”中对应项移除（若存在），在“在线平台”下新增一项（显示名为所填“平台名”）。
6. 用户点击“取消” → 不保存，关闭弹窗。

**平台列表状态**：离线平台仅展示可接入的类型；用户添加窗口并确定后，该窗口以“平台名”出现在在线平台列表；点击在线平台项 → 主内容区执行嵌入逻辑。

**数据结构**：窗口信息至少包含：平台名、进程名、窗口标题、窗口类名、窗口句柄（HWND 数值）。与 `platforms` 表的对应关系：可持久化平台名、进程名、类名、标题，用于下次启动时按“进程名+类名+标题”再次查找窗口（句柄在进程重启后失效，仅对当前运行实例有效）。

#### 4.3.3 技术实现流程
以下按“步骤 + 目的 + 所用技术 + 注意点”说明，便于实现与排错。

1. **枚举系统顶层窗口**
    - **目的**：得到当前系统中所有顶层窗口列表，供“添加新窗口”对话框展示。
    - **技术**：Win32 `EnumWindows` 遍历顶层窗口；在回调中对每个句柄用 `GetWindowThreadProcessId` 得到进程 ID，再用 `OpenProcess` + `GetModuleBaseName`（或 `QueryFullProcessImageName`）得到进程名；用 `GetClassName`、`GetWindowText` 得到类名和标题。
    - **注意**：枚举可放在工作线程，结果通过信号槽回传 UI 线程，避免卡顿；过滤掉本进程窗口、不可见窗口等。
2. **展示列表并选择后填充表单**
    - **目的**：用户从列表选一项，自动填“平台名、进程名、窗口标题、窗口类名、窗口句柄”。
    - **技术**：用 Qt 控件（`QListWidget`）展示；选中项对应一个句柄，用上述 Get* 函数补全信息并写入表单（平台名可默认用窗口标题）。
3. **保存窗口配置**
    - **目的**：将“平台名 + 进程名 + 窗口类名 + 窗口标题 + 句柄”持久化，并加入“在线平台”列表。
    - **技术**：写入 SQLite（`platforms`表）。句柄在进程重启后会失效，需说明“仅对当前运行实例有效”，下次启动需重新添加或通过“进程名+类名+标题”再次查找。
4. **获取 Qt 容器句柄**
    - **目的**：得到主窗口中央用于嵌入的控件的 Windows 句柄（HWND），作为 `SetParent` 的父窗口。
    - **技术**：Qt 中 `QWidget::winId()` 得到 `WId`（在 Windows 上即 HWND）；或先 `QWindow::fromWinId(hwnd)` 再用 `QWidget::createWindowContainer` 包装。需在控件已显示后调用 `winId()`，否则可能无效。
5. **执行嵌入（重父化）**
    - **目的**：把外部窗口“挂”到我们容器的内部，视觉上变成主窗口的一部分。
    - **技术**：`SetParent(外部窗口句柄, 容器句柄)`；可选：`SetWindowLong` 修改样式（如去掉 WS_MAXIMIZEBOX、WS_MINIMIZEBOX，这是窗口样式，最大化/最小化）。
    - **注意**：`SetParent` 后窗口坐标变为相对父客户区，需立即用 `SetWindowPos` 调整位置和大小，使嵌入窗口填满容器客户区。
6. **尺寸与位置同步**
    - **目的**：主窗口或中央容器 resize 时，嵌入窗口始终填满容器。
    - **技术**：在 Qt 中重写容器的 `resizeEvent`，在回调里用 `SetWindowPos(hwnd, 0, 0, width, height, SWP_NOZORDER)` 等将嵌入窗口设为 (0,0) 到 width×height。
7. **焦点与最小化/最大化/关闭**
    - **目的**：点击嵌入区域时焦点交给子窗口；主窗口最小化/最大化/关闭时嵌入窗口行为正确。
    - **技术**：焦点可通过 `SetFocus` 传给子窗口；主窗口最小化/最大化时，通常子窗口会随父窗口一起被系统处理；主窗口关闭时，先 `SetParent(嵌入句柄, NULL)` 再关闭主窗口。
8. **通过“平台名”再次查找窗口**
    - **目的**：用户重启本软件后，之前保存的句柄已失效，可通过“进程名+类名+标题”再次枚举找到目标窗口。
    - **技术**：与步骤 1 相同枚举，按保存的进程名/类名/标题过滤，取第一个匹配的 HWND，再执行步骤 4～7。

#### 4.3.4 相关技术说明
简要说明本方案涉及的概念与接口。

+ **什么是 API**  
API 可以理解为“别人已经写好的函数/接口”：你按约定传入参数、调用它，就能完成特定功能（例如“枚举所有窗口”“把 A 窗口设为 B 的子窗口”），而不必自己从零实现。
+ **什么是 Win32 API**  
Win32 API 是 Windows 操作系统提供的一套 C 语言接口，用来操作窗口、进程、文件、注册表等。例如“找窗口”“改窗口父子关系”“改窗口大小和样式”都是通过调用这些函数完成的。本方案用到的就是 Win32 里和“窗口”相关的部分。
+ **在 Qt 中能使用 Win32 API 吗？**  
可以。Qt 程序运行在 Windows 上时，本质上就是一个 Windows 进程，可以像普通 C++ 程序一样包含 `<windows.h>` 并调用 Win32 函数。Qt 的窗口在底层也有一个 Windows 句柄（HWND），Qt 提供 `QWidget::winId()` 把这个句柄暴露出来，这样就能把“外部窗口”设为我们 Qt 控件的子窗口，实现嵌入。
+ **本方案用到的 Win32 API 简要说明**：
    - `EnumWindows`：遍历当前所有顶层窗口，对每个窗口调用你提供的回调函数；
    - `GetWindowThreadProcessId`：根据窗口句柄得到所属进程 ID；
    - `GetClassName` / `GetWindowText`：根据窗口句柄得到窗口类名和标题文字；
    - `FindWindow`：按类名和/或标题查找一个窗口（可选，枚举也能实现查找）；
    - `SetParent`：把某个窗口设为另一个窗口的子窗口（即“重父化”，嵌入的核心）；
    - `SetWindowLong`：修改窗口样式（如去掉最大化/最小化按钮）；
    - `SetWindowPos`：设置窗口位置和大小（嵌入后用来把子窗口铺满容器）。
+ **Qt 侧用到的接口**：
    - `QWidget::winId()`：获取该 Qt 控件对应的 Windows 句柄（HWND）；
    - `QWindow::fromWinId(WId)`：用已有的原生句柄构造一个 QWindow（用于包装外部窗口）；
    - `QWidget::createWindowContainer(QWindow*)`：把 QWindow 包成 QWidget，放进布局里，这样外部窗口就能随布局自动调整大小。

补充：Qt 6中，`fromWinId` 等可能位于 `QNativeInterface::QWindow` 或类似命名空间，避免与 Qt5 混用。

#### 4.3.5 外部窗口嵌入：PC 客户端与网页的异同
从“嵌入”这一步讲，**PC 客户端与浏览器窗口没有本质区别**，都是对“窗口句柄”（HWND）做文章。**PC 客户端**（如千牛）：独立进程，有一个或多个顶层窗口；我们通过 Win32 枚举/查找该窗口，用 `SetParent(目标句柄, 容器句柄)` 把它嵌到主窗口里。**用户用浏览器打开拼多多商家后台等网页时**：我们嵌入的是**浏览器进程的窗口**（如 Chrome/Edge 的某个标签页对应的 HWND），操作同样为“找到该窗口的 HWND，再 SetParent 到我们容器”，与嵌入千牛一致；区别仅在于要枚举/识别的窗口不同（千牛主窗口 vs 浏览器窗口）。若我们在应用内用 **QWebEngineView** 自己加载该网页，则是“本进程内嵌浏览器控件显示网页”，不涉及把外部进程的窗口嵌进来，不用 SetParent 外部 HWND，与“嵌入外部窗口”是两回事。因此：**只要嵌入的是外部进程的窗口（千牛或浏览器），技术上都是“找 HWND → SetParent”，嵌入操作相同。**

---

**小结**：

在当前 **MSVC + Qt 6 + Windows 10/11** 环境下，可以采用“**Win32 句柄重父化 + Qt 容器控件**”实现嵌入。

**关键思路**：

+ 使用 Win32 API：
    - `FindWindow` / `EnumWindows`：按窗口标题、类名、进程名搜索目标窗口；
    - `SetParent`：将外部窗口的父窗口设置为我们 Qt 中间容器的原生窗口句柄；
    - 辅助：`SetWindowLong`、`SetWindowPos` 控制嵌入窗口的样式和大小。
+ 在 Qt 侧：
    - 获取用于承载外部窗口的控件的 `HWND`（如通过 `winId()` / `QNativeInterface::QWindow::fromWinId` 等）；
    - 使用 `QWidget::createWindowContainer(QWindow *foreignWindow)` 将外部窗口包装成 Qt 控件，以便随布局自动调整。

**嵌入流程**（以千牛为例）：

1. 用户点击左侧“千牛”平台：
    - 若本地未检测到 `AliIM.exe` 进程，则使用 `QProcess::startDetached()` 启动千牛客户端；
2. 通过轮询（或短延迟）使用 `EnumWindows` + `GetWindowThreadProcessId` 查找属于该进程且符合窗口标题/类名的主窗口 `HWND`；
3. 获取 `MainWindow` 中心容器的原生 `HWND`；
4. 调用 `SetParent(aliHwmd, containerHwmd)`，同时设置样式去除最大化/最小化按钮（根据需要）；
5. 在 Qt 中监听容器大小变化（`resizeEvent`），同步调用 `SetWindowPos` 调整嵌入窗口大小；
6. 处理焦点切换（点击嵌入窗口时，将焦点传递给子窗口）。

> 风险提示：个别平台客户端（尤其是基于 Electron 的）在被重父化后可能行为不完全稳定（如弹窗位置、菜单显示等），需要逐个平台测试和调整，但这属于**平台兼容问题**，不是技术栈问题。

### 4.4 消息实时推送到“聚合对话接待”
**期望效果**：有客户咨询时，消息实时推送到聚合对话接待，将各平台会话聚合到“聚合对话接待”界面，人工查看并手动回复，“全部会话”列表中展示所有会话，“待处理”列表中展示还没有处理回复的会话，每次已回复的会话将转移到“全部会话”列表中。

**概念澄清：消息推送——PC 客户端与网页的差异**

区别主要在**“怎么拿到消息数据”**，不在“嵌的是 PC 还是网页”。**若有官方 API/WebSocket**：平台提供客服消息接口时，我们统一用 HTTP/WebSocket 收消息，与用户使用 PC 客户端还是网页无关，消息推送实现方式一致。**若没有官方接口**：**PC 客户端**（如千牛）没有 DOM、没有我们可控制的页面脚本，只能通过 RPA / UI Automation / OCR 读窗口上的文字来“看到”新消息，或模拟键盘在输入框发送；**嵌入的是外部浏览器窗口**时，我们同样拿不到网页内部 DOM 或网络请求，要“读消息”只能靠 RPA/读屏或截屏+OCR，与 PC 客户端类似；**若我们用自己的 QWebEngineView 加载该网页**，则可在我们控制的页面里注入 JS，监听网络或 DOM 变化，从而拿到结构化消息（在平台未强加密的前提下）。具体对接策略见 4.4.5。

**补充**：

只要平台（如淘宝、京东、抖音）提供了官方的客服消息接口（无论是HTTP回调还是WebSocket），无论对方是在PC客户端、网页还是App上登录，消息都会流经平台服务器。我们的服务端统一接入即可，与前端形态无关。需要为每个平台的接口（如千牛开放平台、抖音开放平台）开发适配器（Adapter）。虽然实现方式一致，但不同平台的接口鉴权、数据格式、心跳机制不同，需要一个统一的“消息中台”来抹平这些差异。无论采用哪种方式获取消息，“消息去重”和“多端同步”的逻辑会变得复杂。例如，用户同时在手机和PC上登录，你在RPA端抓到了消息，但用户已经在手机上回复了，系统需要有能力判断这条消息是否已被处理，避免重复推送或重复回复。

RPA/UI Automation（UI自动化）的难点：

+ 句柄识别： 像千牛这样的客户端，通常使用Qt或Duilib等自绘UI框架。传统的Windows消息机制可能无法获取到具体的文本内容，或者控件的“句柄”是不存在的。这时，UI Automation（UIA）有时能识别出元素，但稳定性远低于Web的DOM。
+ OCR的局限性： OCR是兜底方案，但存在误报、emoji表情识别乱码、消息顺序错乱等问题，且对性能消耗较大。
+ 输入模拟： 模拟键盘输入相对容易，但难点在于“焦点管理”——必须确保打字的时候光标确实在输入框里，不能被弹窗或其他程序打断。

#### 4.4.1 需求分析细化
+ **用户角色**：客服人员，需在一处查看多平台会话并统一回复。
+ **使用场景**：各平台有客户咨询时，消息汇总到“聚合接待”；客服在待处理/全部会话中选会话、看消息、输入回复并发送到对应平台。
+ **功能清单**：
    - 新消息检测与入库（来自各平台适配器，根据各平台如千牛、拼多多适配）；
    - 会话列表（待处理 / 全部）与未读标识；
    - 选中会话后展示消息气泡；
    - 输入回复并发送到对应平台；
    - 区分平台图标与来源。
+ **非功能需求**：实时性（轮询或 WebSocket 频率可配置）；不阻塞 UI（监听在工作线程）；平台适配可插拔（`IPlatformAdapter` 抽象，便于按平台扩展）。

#### 4.4.2 交互与数据设计
**新消息到聚合接待的流程**：平台有消息 → Adapter 检测到 → 发出 `incomingMessage` 等信号 → `MessageRouter` 写库（`conversations`/`messages`）并更新会话 → `ConversationManager` 发信号通知 UI → 聚合接待列表刷新、待处理数量更新。

**待处理 / 全部会话**：待处理为未回复或未关闭的会话；全部为所有会话；点击某会话则加载该会话的消息列表到中间对话区。

**发送回复流程**：用户在输入框输入文本并点击发送 → `MessageRouter` 根据当前会话找到对应 `IPlatformAdapter` → 调用 `sendMessage(conversationId, text)`；若为 API 则发 HTTP/WebSocket 请求，若为 RPA 则模拟输入到嵌入窗口；成功后写入 `messages` 表并刷新 UI。

#### 4.4.3 数据模型（SQLite 表简要设计）
+ 表 `platforms`
    - `id`, `name`, `type`（taobao/jd/douyin/…），`status`（online/offline）
+ 表 `conversations`
    - `id`，`platform_id`，`platform_conversation_id`（平台原始会话 ID）
    - `customer_name`，`last_message`，`last_time`，`unread_count`，`status`（open/closed）
+ 表 `messages`
    - `id`，`conversation_id`，`direction`（in/out）
    - `content`（文本，后续可扩展多媒体）
    - `created_at`，`sender`（customer/robot/agent），`raw_payload`（JSON）

#### 4.4.4 平台适配接口设计
定义 `IPlatformAdapter` 接口（伪接口示意）：

```cpp
class IPlatformAdapter : public QObject {
    Q_OBJECT
public:
    virtual void connect() = 0;
    virtual void disconnect() = 0;
    virtual void startListening() = 0;
    virtual void stopListening() = 0;
    virtual void sendMessage(const QString &conversationId,
                             const QString &text) = 0;

signals:
    void incomingMessage(PlatformMessage msg);     // 新消息
    void conversationUpdated(ConversationInfo c);  // 会话状态变化
};
```

`PlatformManager` 维护一个 `QMap<PlatformType, IPlatformAdapter*>`，`MessageRouter` 订阅所有 adapter 的信号，将消息写入 SQLite，并发信号给 UI。

#### 4.4.5 平台对接策略
**策略 A：官方 API / Web 接口（优先）**

+ 若平台提供客服会话相关 API 或 WebSocket：
    - 使用 `Qt Network` / `QWebSocket` 实现登录、订阅会话、发送消息；
    - 数据直接是结构化 JSON → 可轻松映射到 `conversations`/`messages` 表。
+ 优点：稳定、可维护、对 UI 布局不敏感；
+ 缺点：需要平台开放权限、签名等，对个人开发者可能有门槛。

**千牛平台示例：基于消息推送 API 获取客服消息**

1. 在千牛开放平台注册并完成开发者认证，创建一个应用（通常为服务端应用）。
2. 在该应用中申请“消息推送 API”等与消息相关的权限，说明使用场景并等待审核通过。
3. 在开发者后台获取 `AppKey` / `AppSecret`，并配置“消息推送回调地址”，该地址指向本系统提供的 HTTP(S) 接口，同时选择需要推送的消息类型（如客服消息、订单消息等）。
4. 在本系统中，由千牛平台的 `IPlatformAdapter`（API 版本）实现回调处理：接收千牛推送的 JSON 数据，解析为内部的 `PlatformMessage`，交给 `MessageRouter` 写入 `conversations` / `messages` 表，再由 `ConversationManager` 通知聚合接待 UI（整体链路见 4.4.6）。

上述方案属于本节的“策略 A：官方 API / Web 接口”，在条件允许时应优先采用；更多细节可参考千牛相关介绍（例如：[千牛如何 API 对接获取消息](https://docs.pingcode.com/baike/3391714)）。

**策略 B：本地客户端 RPA / UIAutomation / OCR（备选，中长期）**

+ 不提供接口的平台（例如某些版本的千牛）：
    - 使用 **Windows UI Automation API（IAccessible/UIAutomation）** 读取窗口控件树，从会话列表与消息区提取文本；
    - 必要时结合 OCR（调用本地/云 OCR API）从截图中提取文本；
    - 使用键盘输入模拟（`SendInput` 等）在客户端输入框中“代打字+回车”发送消息。
+ 这类实现需要：
    - 针对每个平台编写专门的 UI 解析逻辑；
    - 随平台版本变化不断维护；
    - 对产品迭代资源要求高。

**初期建议**：

+ 为了先把完整链路跑通（“客户咨询 → 聚合对话接待 → 人工输入 → 返回平台”）：
    - 可以 **先选一个对接“成本最低”的平台或先做一个“模拟平台”**：
        * 如：自建一个简单的 Web 聊天服务，作为“电商平台模拟器”，对接只用 HTTP/WebSocket；
        * 或选择 API 文档较好、你掌控力更强的一个平台作为首个 PoC。
    - 对于千牛、拼多多等：
        * **初期先实现“窗口聚合 + 快速启动”，不强行做消息采集**，避免项目被高难度部分阻塞；
        * 等“聚合接待 + AI 方案”验证价值以后，再投入精力做平台级 RPA 对接。

**开放平台与账号使用策略（内部阶段）**：

+ 开放平台开发者账号建议**集中使用 1 个账号**（初期可以是个人身份入驻的开放平台账号），统一负责创建应用、申请权限以及维护回调地址等配置，避免每位客服同事单独建应用、增加管理复杂度。
+ 每位客服同事继续使用自己原有的店铺账号（千牛、拼多多商家后台等）登录平台，通过平台提供的“授权应用”机制（如 OAuth 授权、扫码授权等）把店铺授权给本系统对应的开放平台应用。本系统只在获得授权的前提下，访问该店铺的客服消息。
+ 在当前“只给内部同事用”的阶段，数据范围限定为**本团队成员自有店铺**，不为外部商家提供服务；如未来需要为他人店铺提供聚合客服能力，再评估升级为企业/服务商开发者，补充相应资质与合规说明。

**接入优先级与阶段性目标**：

+ **阶段 1：验证技术链路（1 平台 + 1 店铺）**  
选定 1 个平台和 1 个店铺（例如“千牛 + 某位同事的主店铺”），目标是打通“平台消息 → 本系统 → 聚合对话界面 → 人工回复 → 回写平台”完整链路，仅支持人工模式，不引入 AI 自动回复。
+ **阶段 2：扩展到多店铺、多平台**  
在阶段 1 稳定后，再逐步为更多同事的店铺开通同一应用的授权，并按需接入拼多多等其他平台；对每个新平台优先查官方 API 文档，没有可用接口时只做窗口聚合，不强行做消息抓取。
+ **阶段 3：评估对外服务与 AI 能力**  
内部使用验证价值后，再评估是否需要升级为企业/服务商身份，将“聚合客服 + AI 回复”对外提供给其他商家，并在文档中单独补充合规与运营策略。

#### 4.4.6 技术实现流程
以下按“步骤 + 目的 + 技术 + 注意点”说明，便于实现与排错。

1. **建立连接与监听**
    - **目的**：让各平台适配器与数据源建立连接并开始接收新消息。
    - **技术**：Adapter 的 `connect()` / `startListening()`；API 方式用 `QNetworkAccessManager` 或 `QWebSocket` 与平台服务器通信；RPA 方式用定时器 + Windows UI Automation 或截屏+OCR 轮询嵌入窗口内容。
    - **注意**：网络与 RPA 逻辑放在工作线程或异步任务中，避免阻塞 UI；通过信号槽将“新消息”传回主线程。
2. **接收并解析消息**
    - **目的**：把平台侧原始数据转成我们统一的 `PlatformMessage` / 会话与消息结构。
    - **技术**：API 方式解析 JSON 响应或 WebSocket 帧，映射到 `conversationId`、`content`、`sender` 等字段；RPA 方式从控件树或 OCR 结果中提取会话列表与消息文本，按约定格式组装。
    - **注意**：不同平台字段名与格式不同，各 Adapter 内部做转换，对外统一发 `incomingMessage` 等信号。
3. **落库与通知 UI**
    - **目的**：持久化会话与消息，并让聚合接待界面及时刷新。
    - **技术**：`MessageRouter` 在收到 Adapter 信号后，写入或更新 `conversations`、`messages` 表（无对应会话则先创建），再通过 `ConversationManager` 发信号；`AggregateChatWindow` 订阅该信号，在主线程更新会话列表与未读数。
    - **注意**：数据库写入可在工作线程完成，但发往 UI 的信号必须在主线程处理（Qt 信号槽跨线程会自动排队到主线程）。
4. **发送回复**
    - **目的**：用户点击发送后，把回复内容送到对应平台。
    - **技术**：API 方式：构造 HTTP POST 或通过 WebSocket 发消息到平台接口；RPA 方式：对嵌入窗口调用 `SetFocus` 聚焦到输入框，再用 `SendInput` 模拟键入文本和回车。
    - **注意**：发送前根据当前会话确定使用哪个 Adapter；发送成功后在本地的 `messages` 表插入一条“out”方向记录并刷新 UI。
5. **线程与安全**
    - **目的**：避免界面卡顿，并保证窗口/句柄操作在正确线程执行。
    - **技术**：网络请求、WebSocket、RPA 轮询均在工作线程或 Qt 异步机制中执行；与 Win32 窗口句柄相关的操作（如 RPA 中 SetFocus、SendInput 的目标窗口）需注意线程归属，必要时通过信号把“要在主线程执行的窗口操作”抛回主线程（参见 4.3 中窗口嵌入的线程说明）。

#### 4.4.7 相关技术说明
简要说明本方案用到的技术概念及在 Qt 中的使用方式。

+ **HTTP / WebSocket**
    - **HTTP**：客户端向服务器发请求（如“获取会话列表”“发送一条消息”），服务器返回响应（如 JSON）。适合“请求-响应”式的拉取与提交。
    - **WebSocket**：建立长连接后，服务器可主动向客户端推送数据（如新消息），适合实时推送。Qt 提供 `QNetworkAccessManager`（HTTP）、`QWebSocket`（WebSocket），可与平台提供的 API 对接。
+ **Windows UI Automation**
    - Windows 提供的一套接口，让程序能够**读取或操作其他窗口上的控件**（按钮、编辑框、列表等）以及其中的文字。在没有官方 API 时，可用它从千牛、浏览器等窗口的会话列表和消息区域“读屏”式地提取文本，或模拟点击、输入。需在工程中链接 Windows SDK，在 C++ 中调用；通常在工作线程中轮询，将结果通过信号传回 UI。
+ **OCR（光学字符识别）**
    - 对**截图**中的图像进行识别，得到可编辑的文本。当 UI Automation 无法稳定获取某些控件文字（如自定义绘制、图片里的字）时，可对嵌入窗口或区域截图，再调用本地或云端 OCR API 得到文本，作为补充手段。
+ **SendInput**
    - Win32 API，用于**模拟键盘（或鼠标）输入**。在 RPA 发送回复时，先让目标输入框获得焦点，再通过 `SendInput` 模拟用户打字和回车，从而把回复内容“填进”平台客户端的输入框并发送。需在拥有焦点与目标窗口的线程中调用，通常在主线程或与嵌入窗口同线程。
+ **在 Qt 中的使用**
    - 网络部分直接使用 Qt 的 `Qt::Network` 模块（`QNetworkAccessManager`、`QWebSocket` 等）。UI Automation、OCR、SendInput 等为 Windows 或第三方能力，在 Qt 项目中通过 `#include` 相应头文件、链接库，在 worker 或主线程中调用即可；与 Qt 的集成方式为：在子线程或异步回调中完成“取数/发请求”，通过 `QObject::signal` 把结果传回 UI 线程，由槽函数更新界面。

**操作建议（内部探索阶段）**：

+ **先从 1 人 1 店铺开始验证**：选择一位愿意配合测试的客服同事及其主力店铺，优先在该店铺完成开放平台授权与消息链路验证，避免一开始就大面积申请权限带来沟通和维护压力。
+ **集中管理密钥与配置**：AppKey、AppSecret、回调地址等敏感信息统一由技术负责人维护，可存放在本地加密配置文件或私有配置仓库中，禁止硬编码进源码仓库。
+ **为每个平台单独记录“对接笔记”**：建议在后续创建 `docs/platforms/` 子目录，按平台记录已申请的权限名称、开放平台文档链接、回调地址示例、测试过程中的问题与解决方案，方便日后维护和排错。
+ **暂缓重投入的 RPA 实现**：在尚未完全确认官方接口情况前，文档与实现层面都应优先尝试 API 方案，仅在确认某平台长期无接口或权限难以获取时，才考虑针对该平台投入 RPA/读屏方案，以控制长期维护成本。

### 4.5 聚合对话接待界面与手动回复
#### 4.5.1 需求分析细化
+ **用户角色**：
    - 电商客服同事：日常需要在多个平台间切换，查看待处理会话并回复。
+ **使用场景**：
    - 工作时间，客服打开本软件 → 从“聚合接待”查看所有平台的**待处理**会话；
    - 根据紧急程度或平台优先级选择会话，阅读上下文后手动输入回复；
    - 对已完成会话可以标记为“已处理/关闭”，转移到“全部会话”列表，减少待处理列表压力，也支持删除会话列表中的会话。
+ **功能需求（初期）**：
    - 展示汇总的会话列表，支持按“待处理 / 全部会话”切换和搜索；
    - 选中会话后，展示完整消息气泡历史；
    - 支持文本输入（后续完善图片、文件传输等）和发送按钮，将消息通过适配器发回对应平台；
    - 显示基础客户信息和订单概览（可先占位，后续逐步完善）；
    - 支持键盘快捷键（如 Ctrl+Enter 发送）与简单的发送失败提示。
+ **非功能需求**：
    - 界面在大量会话和消息时保持流畅（列表需分页或懒加载）；
    - 新消息到来时不抢焦点，仅在左侧会话列表显示未读标记和排序调整；
    - 故障安全：平台适配器异常（网络错误等）时，界面给出清晰提示，不崩溃。

#### 4.5.2 界面与交互设计
整体采用三栏布局：左侧会话列表、中间对话区、右侧客户信息面板。

+ **左侧会话列表**：
    - 顶部：
        * Tab 切换：“待处理 / 全部会话”；
        * 搜索框（按客户名、会话内容关键字过滤）。
    - 列表项字段：平台图标、客户名、最近一条消息摘要、时间、未读数徽标。
    - 交互：
        * 点击一项切换当前会话，在中间和右侧显示对应内容；
        * 通过右键菜单或按钮将会话标记为“已处理/关闭”，从“待处理”列表中移除（仍保留在“全部会话”中）。
+ **中间对话区**：
    - 上部为滚动消息气泡区域，区分“客户消息”“人工回复”（未来可留出 AI 回复样式位置）；
    - 下部为输入区：多行文本输入框（如 `QPlainTextEdit`）+ 发送按钮；
    - 交互：
        * Ctrl+Enter 发送，Enter 换行；
        * 发送过程中可在输入框下方或状态栏显示“正在发送…”、“发送失败，请重试”等提示；
        * 支持快捷键从会话列表切换上一条/下一条会话（可选）。
+ **右侧客户信息面板**：
    - 顶部显示：客户昵称、平台、店铺、简单标签（新客/老客等，初期可以为空）；
    - 中部展示近几笔订单摘要（若暂未打通订单数据，则显示“暂无订单数据”，保留结构）；
    - 底部预留“备注、内部标记”等区域，供后续扩展。

可以在实现阶段使用 Qt 的 `QSplitter` 或布局管理器实现上述三栏布局，文档此处仅描述结构与交互。

#### 4.5.3 消息流转与状态管理
从“客服看到什么变化”的角度，再次描述消息处理流程（对应技术细节见 4.4.6）。

+ **新消息到来**：
    - 平台端有新消息 → 对应 `IPlatformAdapter` 检测到并发出 `incomingMessage` → `MessageRouter` 写入 `messages` 表，必要时新建 `conversation` 记录；
    - `ConversationManager` 更新会话的最后一条消息、时间、未读数等，并发信号给 UI；
    - 左侧“待处理”列表中出现（或刷新）该会话，排序靠前，显示未读数红点，但不会主动抢占当前已打开会话的焦点。
+ **未读 / 待处理逻辑**：
    - 当会话有未读消息且未被标记为关闭时，出现在“待处理”列表中；
    - 客服选中会话并将对话区滚动到最新消息后，未读数清零，但仍保留在“待处理”列表中，直到人工确认“已处理/关闭”；
    - 点击“标记为已处理/关闭”后，会话从“待处理”列表中移除，只保留在“全部会话”列表，用于后续查询与统计。
+ **发送回复**：
    - 客服在输入框中输入内容并点击“发送”（或使用快捷键）；
    - UI 调用 `MessageRouter` 的发送接口，后者根据当前会话找到对应的 `IPlatformAdapter` 并调用 `sendMessage(conversationId, text)`；
    - 发送成功后，`MessageRouter` 在 `messages` 表中插入一条“out”方向记录，对话区即时出现新的“人工回复”气泡；
    - 若发送失败，UI 给出红色错误提示，并尽量保留输入内容，方便客服修改后重试。

#### 4.5.4 技术实现要点
实现聚合对话界面时建议采用的技术组合。

+ **界面技术栈**：
    - 使用 Qt Widgets 实现三栏布局（如 `QSplitter` + 若干 `QWidget`），左侧会话列表可用 `QListView` / `QTableView` + `QAbstractListModel`，中间消息区可以是自绘的消息气泡列表或基于 `QListView` 的自定义委托，右侧信息面板为普通表单布局。
+ **数据绑定与刷新**：
    - 会话列表对应 `ConversationModel`，订阅 `ConversationManager` 发出的信号，在有新会话或会话状态变化时更新模型数据，由 Qt 自动刷新视图；
    - 消息列表对应 `MessageModel`，在当前会话切换时重新加载该会话的最近 N 条消息，并支持向上滚动时懒加载更早的历史记录；
    - 通过 Qt 的 Model/View 机制，让“数据变化 → 模型通知 → 视图刷新”形成标准流程，而不是在业务逻辑里直接操作每一行 UI 控件。
+ **交互逻辑**：
    - 发送按钮和快捷键通过信号槽连接到 `onSendClicked()` 槽函数，内部负责读取输入框内容、调用 `MessageRouter` 并处理返回结果；
    - 会话列表点击信号连接到 `onConversationSelected()`，在其中切换当前 `conversationId`，刷新中间对话区和右侧客户信息面板。
+ **持久化与查询**：
    - 复用前文设计的 `conversations` 与 `messages` 表以及相应 DAO（如 `ConversationDao::listOpenConversations()`、`MessageDao::listMessages(conversationId, limit, offset)`）；
    - “待处理 / 全部会话”以及搜索过滤通过不同 SQL 条件实现（如 `status = 'open'` 或模糊匹配客户名、最近一条消息内容等）；
    - 对会话列表和消息列表均建议加 `LIMIT`/`OFFSET` 或其他分页方式，避免一次性加载过多数据导致界面卡顿。

#### 4.5.5 相关技术说明（界面与线程）
简要解释实现聚合接待界面时常用的几个概念。

+ **Qt Widgets 与布局**  
Qt 提供了一整套现成的界面控件（窗口、按钮、列表、文本框等）和布局管理器。通过 `QSplitter`、`QWidget`、`QLayout` 等组件，可以把窗口自然地分成左中右三栏，而不需要手动计算坐标和尺寸。
+ **Model/View 模式**  
会话列表和消息列表不是“画死”的控件，而是由“数据模型 + 视图”组成：模型负责存储和提供数据，视图负责展示。数据一旦有增删改，模型发出信号，视图自动刷新，这种方式更易于维护，也便于后续增加筛选、排序等功能。
+ **信号与槽**  
Qt 中的信号/槽可以类比为“事件/回调”：例如“收到新消息”是一个信号，界面层的槽函数连接到这个信号后，一旦有新消息发出信号，就会自动触发对应的刷新逻辑，无需在业务代码中手动查找所有需要刷新的地方。
+ **线程与响应性**  
为了避免界面卡死，耗时操作（如网络请求、数据库大查询）应放在工作线程或异步任务中执行；但所有界面更新都必须在 UI 主线程中完成。因此常见做法是：工作线程完成任务后发出信号，把结果传回主线程，由主线程中的槽函数更新会话列表和对话区。4.4.6 中描述的平台消息监听同样遵循这一原则。

## 五、数据结构（初期核心表简要设计）
+ `users`：用户登录信息  
    - `id`, `username`, `password_hash`, `salt`, `created_at`
+ `platforms`：支持的电商平台  
    - `id`, `name`, `type`（taobao/jd/douyin/xhs/dewu/wechat 等）、`status`（online/offline）
+ `platform_accounts`（可选，后续支持多账号）  
    - `id`, `platform_id`, `account_name`, `owner_user_id`, `extra_config`（JSON）
+ `conversations`：会话  
    - `id`, `platform_id`, `platform_conversation_id`（平台原始会话 ID）, `customer_name`, `last_message`, `last_time`, `unread_count`, `status`（open/closed）
+ `messages`：消息  
    - `id`, `conversation_id`, `direction`（in/out）、`content`（文本，后续可扩展多媒体）, `sender`（customer/robot/agent）, `created_at`, `raw_payload`（JSON）

## 六、开发阶段划分与风险提示
### 6.1 初期开发步骤
1. **阶段 0：项目初始化**
    - 搭建 Qt + CMake 工程骨架，接入 SQLite、日志；
    - 实现基础 `Database` 封装。
2. **阶段 1：登录/注册**
    - `users` 表设计与迁移；
    - `LoginWindow` + `AuthManager`；
    - 登录后进入 `MainWindow`。
3. **阶段 2：主窗口 + 平台列表**
    - 实现主窗口基本布局、左侧平台树；
    - 实现“系统就绪”默认页与简单状态栏。
4. **阶段 3：外部窗口嵌入**
    - 实现 `Win32WindowHelper`；
    - 完成至少一个平台（如千牛）的：启动客户端 + 窗口查找 + 嵌入 + 跟随布局调整。
5. **阶段 4：聚合对话接待（本地模拟/可控平台）**
    - 完成 `conversations`、`messages` 表及 DAO；
    - 实现 `AggregateChatWindow` 的会话列表 + 对话区 + 发送框；
    - 用一个**模拟平台**或 API 简单的平台打通“新消息 → 聚合界面 → 人工回复 → 回写平台”。
6. **阶段 5：逐步接入真实电商平台**
    - 按平台一一评估对接方案（API / RPA）；
    - 优先选择对业务最关键、技术难度相对可控的平台。

### 6.2 主要风险与对策
+ **风险 1：电商平台缺乏开放接口**
    - 对策：初期用模拟平台或 API 更友好的平台打通链路；千牛/拼多多先只做“窗口聚合”。
+ **风险 2：RPA/读屏方案易碎**
    - 对策：将平台适配逻辑集中在 `services/platforms`，通过接口隔离；对 UI 元素使用相对稳健的定位方式（Automation Id、控件层次），并做好版本检查。
+ **风险 3：嵌入外部窗口带来的兼容性问题**
    - 对策：采用可回退方案：在嵌入失败时，至少提供“快速启动 + 置顶显示”模式，保证客服工作不被阻断。
