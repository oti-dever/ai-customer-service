# 默认同机服务主库模式落地记录

本文记录本阶段围绕“单机客服工作台”产品形态的落地内容。目标是把客户端和 Python 服务同机运行时的业务事实源收敛到本机统一库 `database/app_data.db`，避免 C++ 客户端和 Python 服务同时写会话、消息、清空、删除等业务数据。

## 0. 最新产品形态决策：只做单机客服工作台主线

更新时间：2026-06-16

结合当前产品使用场景和 RPA 技术现实，本阶段不再把“默认模式、兼容模式、远程模式”作为并列产品形态推进。项目主线统一收敛为：

```text
每个客服电脑独立部署完整项目
客户端 + Python 服务 + 平台窗口 + database/app_data.db 均在同一台电脑
Python 服务操作本机微信/千牛等真实窗口
Python 服务主写业务事实表
C++ 客户端直接读取本机 app_data.db 展示会话和消息
C++ 客户端只写 UI 状态、AI 辅助记录、用户资料等客户端私有表
```

调整原因：

- 当前项目是前台 RPA 模拟人工操作，不是纯后台协议接入。Python 服务需要依赖本机平台窗口、焦点、剪贴板、控件树、截图和发送框状态。
- 客户端与 Python 服务跨机器部署后，真实平台窗口仍在 Python 服务所在电脑，客户端无法直接掌控窗口状态，反而会增加网络、路由和状态同步成本。
- 一台电脑集中监听多个客服账号、多微信窗口、多千牛窗口，会增加窗口切换、焦点争抢和响应延迟。
- 多台 Python 服务共写外部 MySQL 需要额外处理客服、设备、平台、账号、窗口、会话、消息、发送任务和 RPA 状态归属，现阶段收益不足。
- 单机部署的数据隔离和故障边界更清楚，一台客服电脑异常只影响自己。

因此后续落地原则调整为：

- 默认只维护单机客服工作台主链路。
- `database/app_data.db` 是本机统一事实库。
- 不再为 `client_cache_db` 设计新的业务能力；旧缓存库只作为历史代码、测试数据或必要时的临时迁移来源。
- 不再推进 `remote_service` 作为近期目标；远程/中心化服务只保留为后期重新设计方向，不影响当前实现。
- 后续代码可逐步移除运行时模式分层，让主链路更直接。

## 1. 背景

项目打包给最终用户使用时，客户端和 Python 服务通常运行在同一台电脑。继续维护“客户端缓存库”和“Python 服务事实库”两套业务事实，会带来几个问题：

- 同一个业务动作可能被 C++ 和 Python 各写一次。
- 两边删除语义不同：Python 服务端倾向软删除，客户端缓存层存在硬删除。
- 服务端采集、发送、清空、删除和客户端本地 DAO 写入之间容易出现竞态。
- 用户只关心界面操作，不应该理解“先启动服务端、再启动客户端、再手动同步缓存”的技术细节。

因此本阶段将产品形态定为：

```text
单机客服工作台
客户端 + Python 服务同机运行
database/app_data.db 是共享业务事实库
Python 服务主写业务事实表
C++ 客户端直接读取业务事实表，只写 UI 状态和客户端私有表
Python 服务启动后负责监听平台、写库并推送变化通知
```

## 2. 单机客服工作台统一运行形态

历史上新增过运行模式工具：

- `src/utils/runtimemode.h`

但从产品方向看，当前阶段不再继续扩展“运行模式配置”这条主线。历史代码中曾出现过三类模式：

| 历史模式 | 当前处理 |
| --- | --- | --- |
| `single_host_service_db` | 作为当前唯一主线语义保留，后续可进一步改名或内化为默认行为 |
| `client_cache_db` | 不再新增业务能力，仅作为旧链路、测试或临时迁移来源处理 |
| `remote_service` | 不作为近期目标，不再影响默认实现设计 |

当前统一运行形态：

- 客户端启动时默认不连接 Python 服务。
- 聚合界面启动后直接读取本机 `database/app_data.db` 中的会话和消息。
- Python 服务由用户在聚合界面手动启动。
- Python 服务启动后负责监听本机平台窗口、写入 `app_data.db` 并向客户端推送变化通知。
- 客户端收到变化通知后重新从 `app_data.db` 读取对应会话和消息。
- 发送、清空、删除等业务 mutation 仍通过 Python 服务执行，避免 C++ 直接写业务事实表。

后续代码层面的目标是逐步减少 `RuntimeMode` 对主链路的影响。凡是只服务于旧缓存或远程预留的分支，都不再作为新功能设计依据。

## 3. Python 服务生命周期

客户端聚合界面左侧已有 Python 服务按钮，用于启动/停止 Python 服务。默认启动模式为 `debug`。

当前设计含义：

- Python 服务可以由客户端后台启动。
- 服务端通过 `--parent-pid` 支持父进程退出后自动结束。
- 客户端启动时不再默认连接 Python 服务，避免服务未启动时产生误导性的失败日志。
- 用户通过界面按钮启动服务，而不是手动执行脚本。

该部分对应代码：

- `src/services/app/pythonservicecontroller.*`
- `src/ui/aggregatechatform.*`
- `src/main.cpp`

## 4. 清空聊天记录和删除会话

### 4.1 旧行为

旧链路中，聚合界面右键会话后：

1. C++ 尝试调用 Python 服务端 mutation。
2. 如果服务端不可用，直接跳过服务端同步。
3. C++ 继续执行本地 DAO 清空或删除。

这会导致默认同机服务主库模式下出现“双写”和“服务失败但本地已删”的问题。

### 4.2 新行为

默认 `single_host_service_db` 下：

- 清空聊天记录必须调用 Python 服务端。
- 删除会话必须调用 Python 服务端。
- Python 服务不可用时，客户端阻止操作并提示先启动服务。
- Python 服务返回失败时，客户端不执行本地删除。
- Python 服务返回成功后，客户端派发服务端返回的事件，走现有事件链路刷新 UI 和本地展示缓存。

相关客户端代码：

- `src/ui/aggregatechatform.cpp`
  - `applyServiceConversationMutation()`
  - `onConversationListContextMenu()`
- `src/ipc/ipcservice.*`
  - `clearConversationMessages()`
  - `deleteConversationOnService()`
  - `dispatchPlatformEvent()`

相关服务端代码：

- `python/service/server.py`
  - `/api/conversations/clear_messages`
  - `/api/conversations/delete`
- `python/service/rpa_bridge.py`
- `python/service/truth_store.py`

### 4.3 表影响

服务端事实库中：

- `conversation_mutations`：记录 mutation。
- `messages`：清空/删除时删除该会话消息。
- `conversations`：
  - 清空聊天记录：保留会话，清空摘要和未读数。
  - 删除会话：软删除，设置 `deleted_at` 和 `status='deleted'`。
- `rpa_events`：写入 `conversation_messages_cleared` 或 `conversation_deleted`。

客户端缓存库中：

- 默认模式下不再由右键操作直接写本地 DAO。
- 本地缓存只通过服务端事件、replay、snapshot 同步展示结果。
- 兼容模式 `client_cache_db` 仍保留旧的本地清空/删除行为。

## 5. 发送消息

### 5.1 旧行为

旧链路中，`MessageRouter::sendMessage()` 会先在客户端缓存库创建一条 pending outbound 消息，然后再调用平台适配器发给 Python 服务。

但 Python 服务端在处理 `send_message` 命令时也会记录 pending outbound，并在后续 `message_sent` 或 `send_failed` 事件中更新状态。

这意味着默认同机服务主库模式下同一条出站消息有两边同时创建 pending 的风险。

### 5.2 新行为

默认 `single_host_service_db` 下：

- `MessageRouter::sendMessage()` 不再先写客户端 pending。
- C++ 直接把发送命令委托给平台适配器。
- 平台适配器通过 Python 命令 WebSocket 发送 `send_message`。
- Python 服务端记录 pending outbound。
- 发送成功/失败后，Python 服务端更新同一条服务端消息。
- C++ 通过事件和 snapshot/backfill 更新展示。

相关客户端代码：

- `src/core/messagerouter.cpp`
- `src/services/platforms/wechatrp_adapter.cpp`
- `src/services/platforms/qianniurp_adapter.cpp`
- `src/ui/aggregatechatform.cpp`

### 5.3 媒体消息字段

Python 服务端 `persist_outbound_command()` 已补齐媒体字段，不再把所有出站命令都当 text 写入：

- `content_type`
- `file_path`
- `file_name`
- `mime_type`
- `size_bytes`
- `content_image_path`
- `evidence_ref`

相关服务端代码：

- `python/service/truth_store.py`

## 6. 服务端回填和展示缓存

当前会话列表仍依赖客户端本地缓存和本地会话 ID，因此本阶段没有直接移除 `client_cache.db`。

默认模式下的同步策略是：

1. Python 服务写事实库。
2. Python 服务推送实时事件。
3. C++ 收到事件后调度 backfill。
4. C++ 通过 replay/snapshot 更新本地展示缓存。
5. UI 用本地缓存承载会话列表、本地 ID、筛选、草稿和右栏状态。

聚合界面新增了节流回填调度：

- 发送消息后延迟触发 backfill。
- 收到 `message_observed`、`message_sent`、`send_failed` 后延迟触发 backfill。
- 避免每个事件都立即全量同步。

相关代码：

- `src/ui/aggregatechatform.cpp`
  - `backfillFromPythonService()`
  - `schedulePythonServiceBackfill()`
  - `applyCacheSnapshotToLocalCache()`

## 8. 当前边界

本阶段没有把“会话列表读取”API 化。

原因：

- 聚合界面当前会话列表依赖客户端本地自增 ID。
- 选中态、最近选中会话、草稿、附件草稿、右栏客户信息和筛选逻辑都围绕本地 ID 工作。
- 直接把列表改成服务端 ID 会牵涉较大范围迁移。

因此当前边界是：

- 会话列表：仍通过本地展示缓存读取。
- 当前会话消息：默认模式下优先服务端 API 读取。
- 高风险写操作：清空、删除、发送，默认模式下由 Python 服务主写。
- 客户端本地库：继续作为展示缓存和 UI 状态承载。

## 9. 后续计划

后续如果继续推进，可以按以下顺序：

1. 给客户端缓存表补充稳定的服务端映射字段，例如 `server_conversation_id`、`server_message_id`。
2. 会话列表 API 化，但保持本地 shadow id 映射，避免 UI 大面积改造。

5. 在 `remote_service` 模式下评估是否需要本地轻量缓存或完全无缓存。

## 11. 新统一数据库方案

当前 `client_cache.db` 和 `service.db` 的表结构、迁移历史和删除语义都不完全一致，因此不适合直接把两个旧库合并成同一个 SQLite 文件。

新的方向是创建一个全新的统一数据库：

```text
database/app_data.db
```

旧库暂时不动：

- `database/service.db`：保留为旧服务事实库和迁移来源。
- `client_cache.db`：保留为旧客户端展示缓存和兼容模式来源。

### 11.1 基本原则

统一库不是“客户端和 Python 随便都写同一批表”，而是“同一个 SQLite 文件内，表级所有权明确”。

核心原则：

- 业务事实表由 Python 服务唯一主写。
- 客户端可以读取业务事实表或只读 view。
- 客户端只写 UI 状态、草稿、窗口布局、最近选择等客户端私有表。
- 同一张业务表不允许 C++ 和 Python 同时主写。
- 跨进程业务动作仍优先走 Python 服务 API，而不是让客户端直接写业务任务表。

### 11.2 表所有权

建议所有权如下：

| 表/对象 | 主写方 | 读取方 | 说明 |
| --- | --- | --- | --- |
| `conversations` | Python 服务 | Python / C++ | 统一会话事实 |
| `messages` | Python 服务 | Python / C++ | 统一消息事实 |
| `wechat_conversations` | Python 服务 | Python / C++ | 微信会话扩展 |
| `wechat_messages` | Python 服务 | Python / C++ | 微信消息扩展 |
| `qianniu_conversations` | Python 服务 | Python / C++ | 千牛会话扩展 |
| `qianniu_messages` | Python 服务 | Python / C++ | 千牛消息扩展 |
| `rpa_events` | Python 服务 | Python / C++ | 事实事件日志 |
| `conversation_mutations` | Python 服务 | Python / C++ | 清空/删除等 mutation |
| `ui_state` | C++ 客户端 | C++ | 通用 UI 状态 |
| `ui_conversation_state` | C++ 客户端 | C++ | 会话选中、折叠、未保存 UI 标记 |
| `ui_conversation_drafts` | C++ 客户端 | C++ | 文本草稿 |
| `ui_compose_attachments` | C++ 客户端 | C++ | 附件草稿 |
| `ui_window_layout` | C++ 客户端 | C++ | 窗口和分栏布局 |

### 11.3 客户端禁止直接写的字段

即便客户端能连接 `app_data.db`，也不应直接写以下业务字段：

- `conversations.last_message`
- `conversations.unread_count`
- `conversations.status`
- `conversations.deleted_at`
- `messages.status`
- `messages.error_reason`
- `messages.deleted_at`
- `messages.platform_message_id`
- `messages.client_message_id`

这些字段由 Python 服务根据采集、发送、回执、清空、删除等业务事件维护。

### 11.4 只读 view

为了降低客户端理解服务端细节的成本，后续可以在统一库中增加只读 view：

```sql
active_conversations
conversation_messages
conversation_last_messages
```

客户端列表和消息区优先读取 view，而不是直接拼接服务端业务表。

当前阶段暂不强制客户端直接读 view，因为聚合界面的会话列表仍依赖客户端本地 ID、筛选状态和草稿状态。

### 11.5 第一阶段实现边界

第一阶段先做三件事：

1. 新增统一库 schema 文件，路径为 `database/app_data_schema.sql`。
2. Python 服务默认事实库从 `database/service.db` 切到 `database/app_data.db`。
3. 保留环境变量覆盖能力，便于回退和调试。

第一阶段不做：

- 不把 C++ 客户端业务 DAO 直接指向 `app_data.db`。
- 不让 C++ 直接写 `conversations/messages`。
- 不迁移旧 `service.db` 和旧 `client_cache.db` 的历史数据。
- 不把会话列表读取 API 化。

这样做的原因是：如果直接把当前客户端 DAO 指到新库，客户端收到实时事件后仍会写 `conversations/messages`，会把双写问题原样带进新库。

### 11.6 路径和环境变量

默认路径：

```text
database/app_data.db
```

环境变量优先级建议：

```text
AI_CUSTOMER_SERVICE_APP_DB
AI_CUSTOMER_SERVICE_SERVER_DB
AI_CUSTOMER_SERVICE_SNAPSHOT_DB
AI_CUSTOMER_SERVICE_DB
```

含义：

- `AI_CUSTOMER_SERVICE_APP_DB`：统一新库路径，优先级最高。
- `AI_CUSTOMER_SERVICE_SERVER_DB`：兼容旧服务端事实库配置。
- `AI_CUSTOMER_SERVICE_SNAPSHOT_DB`：兼容 snapshot 调试配置。
- `AI_CUSTOMER_SERVICE_DB`：兼容旧 RPA DB 配置。

### 11.7 迁移计划

后续迁移建议分阶段进行：

1. Python 服务新数据写入 `app_data.db`。已完成。
2. 提供一次性迁移工具，把旧 `service.db` 的业务事实导入 `app_data.db`。已完成第一版。
3. 客户端连接服务后通过 API/snapshot 从 `app_data.db` 回填展示缓存。
4. 客户端 UI 状态逐步迁到 `ui_*` 表。
5. 客户端会话列表读取再评估是否直接读 view 或继续通过 API。
6. 稳定后，`service.db` 和 `client_cache.db` 降级为历史兼容库。

### 11.8 旧服务库导入统一库

本次新增一次性迁移入口：

```powershell
cd python
python -m service.migrate_legacy_databases --target app-data
```

默认行为：

- 迁移来源：`database/service.db`
- 迁移目标：`database/app_data.db`

## 12. 开发阶段继续迁移讨论：客户端和 Python 服务共用 app_data.db

当前已经进入“统一库事实源 + 客户端 shadow cache”的中间态。下一阶段目标可以更明确地改成：

```text
开发阶段允许清空 service.db / client_cache.db / app_data.db 重测
客户端和 Python 服务默认都使用 database/app_data.db
Python 服务主写业务事实表
C++ 客户端读业务事实表或服务 API，只写 ui_* 和客户端私有表
逐步取消 client_cache.db 作为运行时 shadow cache 的必要性
```

因为现在三个库里的数据都是实机测试数据，可以随时清空重测，所以后续不必把“历史数据无损迁移”作为第一约束。更重要的是把读写边界整理干净，避免把旧 `client_cache.db` 的双写语义带进 `app_data.db`。

### 12.1 当前代码状态

Python 服务侧：

- `python/rpa/db/connection.py`
  - `resolved_default_db_path()` 默认返回 `database/app_data.db`。
- `python/service/cache_snapshot.py`
  - `DEFAULT_SERVICE_DB_PATH` 默认是 `database/app_data.db`。
  - `/api/cache/snapshot`、`/api/conversations/list`、`/api/conversations/messages` 都读取统一库。
- `python/service/truth_store.py`
  - `persist_event()`、`persist_outbound_command()`、`clear_conversation_messages()`、`delete_conversation()` 都通过 `resolved_snapshot_db_path()` 写统一库。

C++ 客户端侧：

- `src/data/database.cpp`
  - `defaultDatabasePath()` 仍返回 AppData 目录下的 `client_cache.db`。
  - `Database::runMigrations()` 仍按旧客户端库模型创建/迁移表。
- 大量 DAO 仍通过 `Database::getInstance().connection()` 读写旧客户端库：
  - `ConversationDao`
  - `MessageDao`
  - `MessageSendEventDao`
  - `AiRequestEventDao`
  - `CustomerProfileDao`
  - `UserDao`
  - `AiAssistantDao`
- `src/data/appdatauistatedao.*`
  - 已经单独连接 `database/app_data.db`。
  - 当前只写 `ui_conversation_drafts` 和 `ui_state`。

聚合界面当前状态：

- 会话列表：优先 `/api/conversations/list` 读取 `app_data.db`，然后调用 `ConversationDao::upsertSnapshotCacheConversation()` 写入 `client_cache.db` shadow cache，UI 使用本地自增 ID。
- 当前会话消息：优先 `/api/conversations/messages` 读取 `app_data.db`，失败回退 `client_cache.db`。
- 清空聊天记录、删除会话、发送消息：默认模式下都已经走 Python 服务，不再让 C++ 主写业务事实。
- 草稿和最近选中：已开始写 `app_data.db.ui_*`，同时保留旧本地回退。

### 12.2 不能直接把 client_cache.db 路径改成 app_data.db 的原因

虽然开发阶段数据可以清空，但仍不建议只做一行改动：

```cpp
// 错误方向：直接让 Database::defaultDatabasePath() 返回 database/app_data.db
```

原因是当前 C++ `Database::runMigrations()` 不是“只创建客户端私有表”，而是会主动创建/迁移业务表：

- `conversations`
- `messages`
- `wechat_conversations`
- `wechat_messages`
- `qianniu_conversations`
- `qianniu_messages`

并且 C++ 旧缓存 schema 还包含服务端统一 schema 不需要的展示缓存字段：

- `cache_scope`
- `cache_origin`
- 一些旧兼容字段和 optional migration

与此同时，Python `truth_store.py` 也有自己的 schema 维护逻辑，并且会清理旧缓存字段，例如 `cache_scope/cache_origin`。如果 C++ `Database` 直接打开 `app_data.db` 并运行旧迁移，就会出现：

- C++ 给业务表加客户端缓存字段。
- Python 又把这些字段当旧字段清掉。
- C++ DAO 仍可能更新 `conversations.last_message/status/unread_count` 等业务字段。
- Python 服务也在维护同一批业务字段。

这会把旧双写问题从“两套库”搬到“同一个 SQLite 文件里的同一批表”，不是目标状态。

### 12.3 开发阶段推荐的新目标状态

目标不是“C++ 和 Python 都随便写同一个库”，而是：

```text
同一个 SQLite 文件
不同表/字段所有权
业务动作仍通过 Python 服务 API
C++ 只读业务事实，主写 UI/客户端私有状态
```

建议所有权调整为：

| 数据类型 | 表 | 主写方 | C++ 行为 |
| --- | --- | --- | --- |
| 会话事实 | `conversations` | Python 服务 | 读，或通过 API 读 |
| 消息事实 | `messages` | Python 服务 | 读，或通过 API 读 |
| 平台扩展 | `wechat_*` / `qianniu_*` | Python 服务 | 读 |
| 清空/删除记录 | `conversation_mutations` | Python 服务 | 读 |
| 事件日志 | `rpa_events` | Python 服务 | 读 |
| 草稿 | `ui_conversation_drafts` | C++ 客户端 | 写 |
| 最近选中/通用状态 | `ui_state` | C++ 客户端 | 写 |
| 附件草稿 | `ui_compose_attachments` | C++ 客户端 | 待迁移 |
| 窗口布局 | `ui_window_layout` | C++ 客户端 | 待迁移 |
| AI 请求过程 | 可继续用客户端私有表，或迁到 `ui_*`/`client_*` 命名表 | C++ 客户端 | 写 |
| 客户画像 | 需要单独决定是业务事实还是 UI 派生缓存 | 待定 | 待定 |
| 登录用户/本地用户 | 客户端私有表 | C++ 客户端 | 写 |

### 12.4 后续实现顺序建议

因为开发数据可以清空，后续可以比之前更直接，但仍建议分小步：

1. 新增 C++ 统一库连接层
   - 不复用当前 `Database::runMigrations()` 的业务表迁移。
   - 可以新增 `AppDataDatabase` 或扩展 `AppDataUiStateDao`，专门打开 `database/app_data.db`。
   - 只负责确保 `ui_*` 和客户端私有表，不创建/修改业务事实表。

2. 让 C++ 业务列表读取取消 shadow cache 写入
   - 当前 `refreshConversationList()` 是：

```text
/api/conversations/list
-> ConversationDao::upsertSnapshotCacheConversation()
-> client_cache.db.conversations
-> UI 使用本地 id
```

   - 下一步可以改成：

```text

-> 直接构造 ConversationInfo

```

   - 开发阶段允许清空重测，因此可以接受本地 ID 语义从 `client_cache.db.id` 切到 `app_data.db.conversations.id`。

3. 当前会话消息完全以服务端消息为准
   - `messagesForDisplay()` 已经服务端优先。
   - 下一步可以移除或弱化本地消息回退。
   - 消息区使用服务端 `messages.id`。

4. 禁止默认模式下 C++ 业务 DAO 写业务表

   - 还需要继续梳理以下路径：
     - `ConversationManager::selectConversation()` 当前会写 `setStatus()` / `clearUnread()`。
     - `ConversationManager::clearConversationMessages()` 兼容模式仍会本地清空。
     - `ConversationManager::deleteConversation()` 兼容模式仍会本地删除。

5. 迁移 UI 私有表
   - 已完成：
     - `ui_conversation_drafts`
     - `ui_state.aggregate/last_selected_conversation`
   - 待迁移：
     - 附件草稿 -> `ui_compose_attachments`
     - 窗口布局/分栏 -> `ui_window_layout`
     - 会话级 UI 状态 -> `ui_conversation_state`

6. 决定客户端私有业务辅助表命名
   - 当前 C++ 还有：
     - `users`

     - `client_users`
     - `client_ai_request_events`

   - 避免它们和 Python 服务业务事实表混在一个语义层里。

7. 移除运行时对 `client_cache.db` 的依赖
   - 当会话列表、当前消息、草稿、最近选中、右栏状态、AI 过程记录都可从 `app_data.db` 或服务 API 获取后：

```text

```

### 12.5 可以利用“开发阶段可清空数据”的地方

既然当前数据都可清空重测，后续可以简化：

- 不必做 `client_cache.db -> app_data.db` 的完整历史迁移。
- 不必维护本地 shadow id 和服务端 id 的长期映射。
- 可以直接约定 UI 使用 `app_data.db.conversations.id`。
- 可以清空 `app_data.db` 后由 Python 服务重新创建 schema 并重新采集。
- 可以在切换阶段删除 `client_cache.db`，用失败尽早暴露残留依赖。

但仍需要保留一个原则：

```text
数据可以清空，读写边界不能含糊。
```

也就是说，开发阶段可以不照顾旧数据，但不能让 C++ 和 Python 同时主写 `conversations/messages`。

### 12.6 下一步最小落地点

建议下一步先不改 Python 服务，而是改 C++ 数据层边界：

1. 新增 `AppDataDatabase` 或扩展现有 `AppDataUiStateDao` 为通用 app data 连接。
2. 增加一个只读/轻写的 `AppDataConversationReader`：
   - 默认同机模式直接读取 `app_data.db` 的 `active_conversations` / `conversation_messages` 或等价查询。
   - 输出 `ConversationInfo`，其中 `id = app_data.conversations.id`。
3. 聚合界面列表直接使用该结果，不再调用 `upsertSnapshotCacheConversation()` 写 `client_cache.db`。
4. 当前消息默认同机模式直接读取 `app_data.db.messages`，不再把 `/api/conversations/messages` 作为主链路。
5. 把默认模式下 `ConversationManager::selectConversation()` 对业务表的 `setStatus()/clearUnread()` 改为不写业务表，必要时后续增加 Python API。

这样做后，`client_cache.db` 在聚合界面主链路中的作用会明显降低：

```text
会话列表：app_data.db 直接读
当前消息：app_data.db 直接读
草稿：app_data.db.ui_conversation_drafts
最近选中：app_data.db.ui_state
业务写操作：Python 服务 API
```

剩余的 `client_cache.db` 依赖就可以集中处理 AI 过程记录、客户画像、本地用户和兼容模式。

### 12.7 本轮落地：默认模式切到 app_data.db 并停止 shadow cache 主链路

本轮按 12.6 的最小落地点推进。默认 `single_host_service_db` 不再把 `client_cache.db` 作为聚合界面会话主链路。

已完成的 C++ 侧改动：

1. `Database::open()` 默认模式改为打开统一库：
   - 默认模式和 `remote_service` 打开 `AppDataUiStateDao::resolvedAppDataDbPath()`，也就是默认 `database/app_data.db`。
   - `client_cache_db` 兼容模式仍打开 AppData 目录下的 `client_cache.db`。
   - 测试或显式传入路径时仍按旧本地库迁移逻辑执行，避免影响现有数据层测试。

2. `Database::runMigrations()` 职责拆分：
   - 新增 `runClientPrivateMigrations()`，默认统一库模式只创建 C++ 私有表。
   - 默认统一库模式不再创建/ALTER `conversations`、`messages`、`wechat_*`、`qianniu_*`。
   - 业务事实表继续由 `database/app_data_schema.sql` 和 Python `ensure_app_database_schema()` 负责。

3. 聚合界面会话列表不再写 shadow cache：
   - `/api/conversations/list` 返回后直接构造 `ConversationInfo`。
   - `ConversationInfo.id` 直接使用 `app_data.db.conversations.id`。
   - 不再调用 `ConversationDao::upsertSnapshotCacheConversation()` 把服务端列表写回本地缓存。
   - `last_direction` 继续使用服务端返回值，用于“待处理/已处理”筛选。

4. 默认模式下禁用旧本地业务写路径：
   - `ConversationManager::selectConversation()` 不再执行 `clearUnread()` / `setStatus("active")`。
   - `ConversationManager::clearConversationMessages()`、`deleteConversation()`、`closeConversation()` 在服务主库模式下拒绝本地业务表写入。
   - `MessageRouter` 收到平台观察、入站消息、清空、删除事件时，默认模式不再写本地业务表；清空/删除只发 UI 事件刷新。
   - `schedulePythonServiceBackfill()` 和 `applyCacheSnapshotToLocalCache()` 在默认模式跳过，不再把 snapshot 写回旧缓存。

5. 草稿只写统一库 UI 表：
   - 默认模式下聚合输入框草稿只写 `app_data.db.ui_conversation_drafts`。
   - 不再写旧 `conversation_drafts`。

6. `ConversationDao` 读取统一库时兼容缺失旧缓存字段：
   - `cache_scope/cache_origin` 不存在时按 `service_db/python_service_db` 处理。
   - 这样 `ConversationDao::findById()` / `findByPlatformId()` 可以读取 `app_data.db.conversations`，供发送链路查平台和会话 key。

本轮之后的实际默认主链路：

```text
C++ Database 默认连接 -> database/app_data.db
会话列表 -> 当前实现仍是服务可用时 /api/conversations/list，失败回退 app_data.db 读取
当前消息 -> 当前实现仍是服务可用时 /api/conversations/messages，失败回退 app_data.db 读取
发送/清空/删除 -> Python 服务 API / command websocket
草稿/最近选中 -> app_data.db.ui_*
client_cache.db -> 仅兼容模式和显式旧链路
```

需要继续处理的遗留点：

- `MessageDao` 当前仍承担读取 `app_data.db.messages` 的回退和 AI 上下文读取；默认模式下应继续避免调用其写入方法。
- `ai_request_events`、`conversation_customer_profiles`、`message_send_events` 仍沿用旧表名，后续可评估是否迁到 `client_*` 命名。
- 附件草稿仍是聚合界面内存态，尚未迁到 `ui_compose_attachments`。
- 旧章节中关于“默认 backfill 写入本地缓存”“会话列表写 shadow cache”的描述是历史阶段记录；以本节为当前状态。

### 12.8 新决策：默认同机模式下列表和消息直接读 app_data.db

本次继续讨论后，默认 `single_host_service_db` 的目标再收窄一步：聚合界面的历史会话列表和历史聊天消息不需要优先走 Python HTTP API。

原因如下：

1. Python 服务是用户在客户端界面里手动启动的。
2. 客户端启动和打开聚合界面时，Python 服务通常还没有启动。
3. 如果会话列表/消息展示先请求 Python API，就会变成“预期失败后再回退本地库”的复杂链路。
4. 默认同机模式下客户端和 Python 服务访问的是同一个 `database/app_data.db`，历史列表和历史消息直接读库即可。
5. Python 服务启动后的核心价值是监听平台、写入新业务事实、把变化通知推送给客户端，而不是给同机客户端转发同一个 SQLite 文件里的历史查询结果。

因此后续默认同机模式的目标主链路调整为：

```text
客户端启动
-> C++ Database 打开 database/app_data.db
-> 聚合会话列表直接读 app_data.db.conversations / active_conversations
-> 当前会话消息直接读 app_data.db.messages / conversation_messages

用户点击启动 Python 服务
-> Python ensure_app_database_schema()
-> Python 监听平台
-> Python 写 app_data.db.conversations/messages/wechat_*/qianniu_*
-> Python 通过事件/WebSocket 通知 C++ 有变化
-> C++ 收到通知后刷新列表或当前会话，并重新从 app_data.db 读取
```

新的读写边界：

| 数据/动作 | 默认同机模式主链路 |
| --- | --- |
| 会话列表展示 | C++ 直接读 `app_data.db` |
| 当前会话消息展示 | C++ 直接读 `app_data.db` |
| 历史消息查询 API | 不作为默认主链路，可保留为调试/远程模式能力 |
| 会话列表查询 API | 不作为默认主链路，可保留为调试/远程模式能力 |
| 平台监听新消息 | Python 服务监听平台并写 `app_data.db` |
| 新消息通知 UI | Python 推送事件，C++ 收到后读库刷新 |
| 发送消息 | C++ 发命令给 Python，Python 写业务事实并执行发送 |
| 清空聊天记录 | C++ 调 Python mutation，Python 写业务事实 |
| 删除会话 | C++ 调 Python mutation，Python 写业务事实 |
| 草稿/最近选中 | C++ 写 `ui_conversation_drafts` / `ui_state` |

需要保留的原则：

- C++ 可以直接读 `conversations/messages/wechat_*/qianniu_*`。
- C++ 默认模式下仍不直接写这些业务事实表。
- `ConversationDao` / `MessageDao` 的读方法可以成为默认展示主链路。
- `ConversationDao` / `MessageDao` 的写方法不能因为默认连接指向 `app_data.db` 就被正常业务路径调用。
- 如果 `app_data.db` 初次运行还没有业务表，C++ 读取应表现为空列表/空消息，而不是崩溃。业务表 schema 仍由 Python `ensure_app_database_schema()` 和 `database/app_data_schema.sql` 负责。

后续实现顺序相应调整为：

1. 默认模式下 `refreshConversationList()` 不再优先调用 `/api/conversations/list`，直接走本地 `app_data.db` 读取。
2. 默认模式下 `messagesForDisplay()` 不再优先调用 `/api/conversations/messages`，直接走本地 `app_data.db` 读取。
3. Python 推送的会话/消息/清空/删除事件只作为“数据已变化”的通知，客户端收到后刷新 UI 并重新读库。
4. `/api/conversations/list` 和 `/api/conversations/messages` 保留为远程模式、调试工具或兼容能力，不再是默认同机模式主链路。
5. 继续清理默认模式下所有 C++ 业务写路径，保证 `app_data.db` 中业务事实仍由 Python 服务主写。

### 12.9 本轮落地：默认列表和消息改为直接读 app_data.db

本轮按 12.8 的新决策推进，先改 C++ 聚合界面展示读链路，不改 Python 服务 API。

已完成：

1. `AggregateChatForm::refreshConversationList()`：
   - 默认 `single_host_service_db` 下不再调用 `/api/conversations/list`。
   - 直接走 `ConversationManager::allConversations()`，最终通过 `ConversationDao::listCachedConversations()` 读取当前默认连接。
   - 当前默认连接已经是 `database/app_data.db`，因此列表事实来自统一库。
   - `remote_service` 仍保留服务 API 读取能力。
   - `client_cache_db` 仍走本地兼容缓存库。

2. `AggregateChatForm::messagesForDisplay()`：
   - 默认 `single_host_service_db` 下不再调用 `/api/conversations/messages`。

```text

-> C++ Database 默认打开 database/app_data.db
-> 会话列表直接读 app_data.db.conversations
-> 当前消息直接读 app_data.db.messages

Python 服务启动
-> 监听平台
```

仍需继续处理：

- `MessageDao::messageSelectProjection()` 当前仍假设 `wechat_messages` / `qianniu_messages` 存在；首次空库时因为 `messages` 不存在可正常返回空，但如果后续出现只有 `messages`、缺少扩展表的半初始化库，还需要进一步增强只读查询兼容性。

## 13. 2026-06-16 后续收敛方向：移除多模式设计负担

基于新的产品形态判断，后续迁移不再以“默认模式、兼容模式、远程模式”三套运行形态为目标，而是以单机客服工作台为唯一主线。

当前判断：

```text
主线：单机客服工作台
库：database/app_data.db
业务事实写入：Python 服务
业务事实读取：C++ 客户端直接读 app_data.db
业务 mutation：C++ 调 Python 服务，Python 写 app_data.db
旧 client_cache.db：不再扩展，只作为历史迁移/测试来源
remote_service：暂不推进，不影响当前主线
```

后续代码清理建议：

1. `RuntimeMode` 相关分支逐步收敛。
   - 新代码不再增加 `client_cache_db` / `remote_service` 分支。
   - 已存在的分支按风险逐步移除或改成内部兼容代码。

2. `Database::open()` 最终可以直接打开 `database/app_data.db`。
   - 不再把 `client_cache.db` 作为运行时可选主库。
   - 测试需要临时库时继续通过显式路径注入。

3. C++ 业务 DAO 保留读能力，写能力从默认主链路移除。
   - `ConversationDao::listAll()`、`findById()`、`findByPlatformId()` 可继续服务 UI 展示和发送前查询。
   - `ConversationDao::create/upsert/remove/updateLastMessage/setStatus/clearUnread` 等业务写方法不再作为客户端主链路能力。
   - `MessageDao::listByConversation()`、`lastMessageForConversation()` 可继续用于展示和 AI 上下文。
   - `MessageDao::create/updateDeliveryState/clearAllForConversation` 等业务写方法不再作为客户端主链路能力。

4. 聚合界面继续直接读 `app_data.db`。
   - 会话列表直接读 `conversations` 或后续只读 view。
   - 当前会话消息直接读 `messages` + 平台扩展表。
   - Python 推送事件只作为“库已变化”的通知。

5. 清空、删除、发送继续通过 Python 服务。
   - Python 服务未启动时禁止这些业务 mutation。
   - Python 服务负责写 `messages`、`conversations`、`conversation_mutations`、`rpa_events` 和平台扩展表。

6. UI/AI/用户私有数据继续由 C++ 写入 `app_data.db` 中的客户端私有表。
   - `ui_state`
   - `ui_conversation_drafts`
   - `users`
   - `ai_request_events`
   - `ai_request_stage_events`
   - `conversation_customer_profiles`
   - `ai_assistant_sessions`
   - `ai_assistant_messages`

现存历史接口处理：

- `/api/conversations/list` 和 `/api/conversations/messages` 还保留在 Python 服务中，后续可明确标记为调试接口，或在确认不再需要远程形态后移除。

这个收敛方向的核心目标是减少不必要的架构分支：先把“每个客服电脑独立运行”的版本做扎实，再考虑是否需要管理端统计、导出、备份或后期重新设计远程集中形态。
