# 微信平台启动发送与消息存储演进方案

## 1. 背景

当前微信 RPA 已经从旧 OCR / session 方案逐步切到对齐 `D:/wechat` 的 UIA / Win32 / 后台操作路线。近期讨论中明确了三个新的工程约束：

- 微信 RPA 不应在客户端进程刚启动时无条件拉起，至少应等客户端登录后再启动。
- 微信发送链路应对齐 `D:/wechat`：后台优先，前台只作为失败兜底，并且需要把实际命中的发送策略回传给 C++。
- 微信平台消息后续可能需要单独的平台消息表，以承接微信特有字段、原始观测数据、角色判断证据和发送诊断信息。

本文档用于沉淀这三个方向的设计边界，后续实现按小步迁移推进。

## 2. 设计结论

### 2.1 启动链路先做小改

当前主程序启动后会初始化平台适配器，微信适配器会启动 Python sidecar 并尝试连接微信自动化链路。短期先不大改平台生命周期，只做一层启动门控：

- 客户端未登录时，不启动微信 RPA 监听。
- 登录成功后，再初始化或启动微信平台适配器。
- 微信未启动、微信窗口最小化、微信未登录、UIA 控件不可用时，只标记微信平台不可用或降级，不让客户端异常终止。
- Python sidecar 返回结构化健康状态，C++ 展示平台状态并阻止无效发送。

推荐状态码：

| 状态码 | 含义 | C++ 行为 |
|---|---|---|
| `client_not_logged_in` | 客户端未登录 | 不启动微信 RPA |
| `wechat_not_running` | 未发现微信主窗口 | 平台降级，提示用户打开微信 |
| `wechat_minimized` | 微信窗口最小化或不可操作 | 平台降级，提示恢复窗口 |
| `wechat_not_logged_in` | 微信未登录或主界面不可用 | 平台降级 |
| `uia_unavailable` | UIA 控件树不可用 | 平台降级并记录诊断 |
| `ready` | 微信链路可用 | 允许扫描未读和发送 |

### 2.2 发送链路改为后台优先、前台兜底可配置

发送策略应对齐 `D:/wechat` 的思路：优先不抢前台、不移动鼠标、不依赖人工当前焦点。

推荐策略顺序：

1. 后台窗口消息发送：`SendMessage`
2. 后台窗口消息发送：`PostMessage`
3. UIA `InvokePattern`
4. 可配置前台兜底：UIA 控件点击
5. 可配置前台兜底：前台鼠标点击

默认行为：

- 聚合界面 C++ 下发 `send_message` 时默认传入 `allow_foreground_fallback=true`。
- Python 侧先尝试后台发送。
- 只有后台策略失败时，才允许前台兜底。
- 前台兜底可以在配置中关闭，但不是默认行为。
- 每次发送都返回 `send_method`、`foreground_used`、`target_verified`、`error_code`，写入发送事件或消息 metadata。

长期目标：

- 前台兜底可以在配置中显式关闭。
- 聚合界面可显示“后台发送成功 / 前台兜底发送 / 发送失败”。
- 发送失败不导致客户端崩溃，只更新消息为 `failed` 并保留诊断。

### 2.3 微信消息表采用“统一核心表 + 微信扩展表”

不建议直接把所有消息完全按平台拆开后让上层分别查询。聚合界面、未读数、全局搜索、AI 上下文和跨平台统计都需要稳定的统一入口。

推荐结构：

```text
conversations          # 统一会话主表，聚合界面和状态机使用
messages               # 统一消息主表，聚合界面、AI 上下文、搜索使用
wechat_conversations   # 微信会话扩展表
wechat_messages        # 微信消息扩展表
message_send_events    # 发送过程审计，保持通用
rpa_events             # RPA 原始事件，可按需保留或归档
```

统一表只放跨平台稳定字段，微信扩展表放微信特有字段和诊断信息。这样既保留聚合能力，也避免把微信字段污染到所有平台。

## 3. 启动链路设计

### 3.1 当前问题

当前平台启动更接近：

```text
客户端进程启动
  -> 初始化 ConversationManager / MessageRouter
  -> 注册平台适配器
  -> 适配器 connectPlatform
  -> 适配器 startListening
  -> 微信 RPA 启动 Python sidecar 并 connect
```

这个流程没有等待客户端登录态，也没有把微信不可用视作普通平台状态。

### 3.2 短期改造

短期目标是低风险：

```text
客户端进程启动
  -> 初始化核心对象
  -> 不启动微信 RPA
用户登录成功
  -> 启动微信平台适配器
  -> Python health_check
  -> ready 则监听未读
  -> 非 ready 则平台降级
```

关键点：

- 微信平台适配器注册可以保留，但 `startListening` 延后。
- 如果登录态退出，应停止微信 RPA 监听。
- Python sidecar 本身可以由 AI 服务复用，但微信自动化能力不要在未登录时开始。
- 健康检查失败是业务状态，不是进程错误。

### 3.3 后续演进

后续可将平台启动改为显式生命周期：

```text
registerAdapter
  -> adapter exists but inactive
loginSuccess
  -> activate platform
aggregateViewOpened
  -> start listen
logout
  -> stop listen
```

这样可以减少启动成本，也避免未进入聚合界面时就驱动真实平台。

## 4. 发送链路设计

### 4.1 命令字段

C++ 下发给 Python 的 `send_message` 建议包含：

```json
{
  "conversation_key": "wechat:local_wechat:张三",
  "display_name": "张三",
  "text": "你好",
  "require_target_verification": true,
  "prefer_background": true,
  "allow_foreground_fallback": true,
  "confirm_token": "manual_confirmed_by_agent"
}
```

其中：

- `require_target_verification=true`：必须确认当前目标会话正确。
- `prefer_background=true`：优先使用不抢前台的后台策略。
- `allow_foreground_fallback=true`：后台失败后允许前台兜底。

### 4.2 返回字段

Python 返回建议包含：

```json
{
  "status": "sent",
  "send_method": "win32_send_message",
  "foreground_used": false,
  "target_verified": true,
  "error_code": "",
  "debug_artifact": ""
}
```

失败时：

```json
{
  "status": "failed",
  "send_method": "",
  "foreground_used": false,
  "target_verified": false,
  "error_code": "target_session_not_found",
  "debug_artifact": "debug/wechat/send_failed_20260529_153000.png"
}
```

### 4.3 C++ 状态更新

C++ 不应因为发送失败退出或崩溃。

推荐状态流：

```text
用户点击发送
  -> messages 插入 outbound pending
  -> UI 立即显示待发送气泡
  -> 下发 send_message
  -> Python 返回 sent
      -> message status = sent
  -> Python 返回 failed
      -> message status = failed
      -> errorReason 写入失败原因
```

## 5. 微信消息扩展表设计

### 5.1 保留统一消息主表

`messages` 继续作为聚合界面和 AI 上下文的主入口。建议保持字段稳定：

| 字段 | 说明 |
|---|---|
| `id` | 内部消息 ID |
| `conversation_id` | 内部会话 ID |
| `direction` | `in` / `out` / `system` |
| `content_type` | 文本、图片、文件等 |
| `content` | 展示文本或摘要 |
| `sender` | `customer` / `agent` / `system` |
| `sender_name` | 展示名 |
| `platform_msg_id` | 平台消息 ID 或合成 ID |
| `sync_status` | 发送 / 观测状态 |
| `source_type` | 来源类型 |
| `confidence` | 可信度 |
| `created_at` | 系统落库时间 |

### 5.2 新增微信消息扩展表

推荐新增 `wechat_messages`：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `message_id` | 关联 `messages.id` |
| `wechat_account_id` | 微信账号标识 |
| `wechat_conversation_key` | 微信会话 key |
| `wechat_display_name` | 微信显示名 |
| `raw_control_name` | UIA 原始控件文本，可脱敏或 hash |
| `raw_control_type` | UIA 控件类型 |
| `role_method` | 角色判断方法 |
| `role_confidence` | 角色判断置信度 |
| `bubble_rect` | 气泡矩形 JSON |
| `message_list_rect` | 消息列表矩形 JSON |
| `observation_method` | `uia` / `win32` / `screenshot_assisted` |
| `evidence_ref` | 诊断截图或证据引用 |
| `raw_payload_json` | Python 原始事件 payload |
| `created_at` | 创建时间 |

### 5.3 新增微信会话扩展表

推荐新增 `wechat_conversations`：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `conversation_id` | 关联 `conversations.id` |
| `wechat_account_id` | 微信账号标识 |
| `wechat_conversation_key` | 微信会话 key |
| `display_name` | 微信显示名 |
| `session_control_hash` | 会话控件标识 hash |
| `last_unread_badge` | 最近观测到的未读数 |
| `last_observed_at` | 最近观测时间 |
| `last_health_status` | 最近平台状态 |
| `raw_payload_json` | 原始会话观测 payload |

### 5.4 为什么不直接按平台完全拆表

完全拆成 `wechat_messages`、`qianniu_messages`、`pdd_messages` 后，如果没有统一主表，会带来几个问题：

- 聚合会话列表需要跨表 `UNION`。
- 未读数和最后一条消息更新逻辑会分散到多个 DAO。
- AI 上下文要为每个平台写一套查询。
- 全局搜索、历史记录、统计报表会变复杂。
- UI 需要知道平台表结构，破坏平台适配器边界。

因此推荐“统一核心表 + 平台扩展表”，扩展表只增强平台能力，不取代统一主表。

## 6. 迁移阶段

### P1：启动门控

目标：

- 客户端登录后才启动微信 RPA。
- 微信不可用只进入降级状态。
- 聚合界面可感知微信平台是否 ready。

验收：

- 未登录客户端时不会启动微信自动化。
- 微信未打开时客户端不崩溃。
- 微信最小化或未登录时，发送按钮应阻止或返回明确失败。

### P2：发送策略收敛

目标：

- Python 默认后台优先。
- 前台只作为最后兜底。
- 返回实际发送方法和是否使用前台。

验收：

- 后台成功时不抢鼠标、不切前台。
- 后台失败时允许一次前台兜底。
- 失败消息在聚合界面显示为 failed。
- 发送诊断可追踪。

### P3：微信扩展表

目标：

- 新增微信会话和消息扩展表。
- `messages` 继续作为聚合主表。
- Python 原始事件落入扩展表或 metadata，便于排查。

验收：

- 聚合界面无需关心微信扩展表。
- AI 上下文仍从统一消息表读取。
- 微信角色判断、控件证据、发送策略结果可从扩展表追踪。

### P4：多平台存储模板

目标：

- 总结微信扩展表经验。
- 为千牛、拼多多预留同样的扩展表模式。
- 不把任何单个平台字段塞进统一主表。

## 7. 实现注意事项

- 不要恢复旧微信 OCR 主链路。
- 不要让 Python 侧直接决定 AI 自动回复。
- 不要让 Python 侧成为业务消息最终事实来源，最终仍由 C++ 统一落库和驱动 UI。
- 不要为了微信扩展表破坏聚合界面的统一模型。
- 发送失败、健康检查失败、窗口不可用都应是可恢复状态。
- 日志和诊断文本统一使用 UTF-8，避免再次出现源码或日志乱码。

## 8. 当前建议

下一步建议按顺序实现：

1. 先做登录后启动微信 RPA 的小改，降低启动期异常风险。
2. 再把发送链路改为后台优先、前台兜底，并补全返回字段。
3. 最后引入微信扩展表，先只写诊断和原始 payload，不急着把查询链路全部迁过去。
