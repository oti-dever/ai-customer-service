# 平台接入 PoC 方案

## 1. PoC 目标

PoC 的目的不是完成完整平台接入，而是验证在**拿不到官方 API / SDK** 的前提下，是否能通过受控技术手段获得一条可用、可降级、可审计的真实消息接入链路。

PoC 需要回答的核心问题是：

1. 能不能稳定识别平台可用状态
2. 能不能拿到当前会话或可见消息
3. 能不能把平台观察结果映射到统一数据模型
4. 能不能把 AI 建议带回到人工确认发送流程
5. 能不能在失败时安全降级

## 2. PoC 选择原则

### 2.1 不从高风险路径起步

PoC 不建议从个人微信 Hook、DLL 注入、未授权抓包或中间人代理开始。这些方式风险高、排查困难，也不符合 MVP 的最小可用原则。

### 2.2 优先选择可见内容较稳定的平台

优先选择：

- Web 化平台
- 界面结构相对稳定的平台
- 允许人工登录并持续可见会话状态的平台

在当前项目背景下，最优先的 PoC 候选是：

1. 拼多多商家后台网页版
2. 千牛 PC 客户端
3. 个人微信 PC 客户端

### 2.3 PoC 只验证一条主链路

PoC 只需要验证一条平台接入链路，不要求同时覆盖多个平台。

## 3. 推荐 PoC 路线

### 3.1 第一 PoC 推荐：拼多多商家后台网页版

推荐原因：

- Web 路线更容易用受控浏览器和 DOM 监听验证
- 页面可见内容更适合统一模型采集
- 更便于后续和浏览器自动化、页面审计结合

### 3.2 备选 PoC：千牛 PC 客户端

适合作为桌面可见内容采集的验证对象，重点验证：

- 无障碍接口
- 窗口状态
- 消息列表识别
- OCR 辅助读取

### 3.3 不作为第一 PoC 的对象：个人微信 PC 客户端

原因：

- 风险更高
- 采集和稳定性更难验证
- 不适合在 MVP 第一阶段作为主要验证目标

## 4. 技术手段组合

PoC 不是单一技术手段，而是组合使用。

### 4.1 Web 平台组合

适用于拼多多商家后台网页版：

- 受控浏览器实例
- DOM 监听
- 页面路由观察
- 可见消息文本提取
- 截图校验
- 人工确认发送

### 4.2 PC 客户端组合

适用于千牛、微信等桌面客户端：

- 窗口识别
- 系统通知辅助
- 无障碍接口
- OCR
- 剪贴板辅助
- UI 自动化
- 人工确认发送

### 4.3 高风险实验链路

以下手段只作为受控实验，不作为 MVP 默认主链路：

- Hook / 注入
- 流量抓包 / 中间人代理

## 5. PoC 范围

### 5.1 必须验证的内容

| 项目 | 要求 |
|---|---|
| 账号状态识别 | 能判断平台当前是否可用 |
| 会话识别 | 能识别当前会话或会话列表 |
| 消息采集 | 能拿到一条可见消息 |
| 数据归一 | 能映射成统一 `Conversation` / `Message` |
| AI 建议 | 能返回回复建议 |
| 草稿回填 | 能填入输入框或复制到剪贴板 |
| 人工确认发送 | 必须由人确认后发送 |
| 异常降级 | 失败时系统不崩溃、不误发 |

### 5.2 不要求验证的内容

- 全量历史消息补拉
- 全平台统一接入
- 自动批量发送
- 完整插件体系
- 全量联系人画像
- 完整工单闭环

## 6. PoC 阶段划分

### 6.1 阶段 0：环境准备

目标：

- 准备测试账号
- 准备浏览器或桌面客户端环境
- 准备日志和截图存储
- 准备人工确认流程

产出：

- PoC 环境说明
- 测试账号清单
- 风险边界说明

### 6.2 阶段 1：可用状态识别

目标：

- 判断平台是否登录
- 判断窗口或页面是否处于可交互状态
- 判断账号状态是否正常

产出：

- `PlatformAccountStatus`
- 基础健康检查结果

### 6.3 阶段 2：消息采集

目标：

- 获取当前会话可见消息
- 获取会话标题、未读数或页面状态
- 生成统一 `ConversationEvent`

产出：

- 一条或多条 `Message`
- 一条 `Conversation`

### 6.4 阶段 3：AI 建议回填

目标：

- 将采集到的消息传给 Python 服务
- 生成 1-3 条建议
- 回填到输入框或复制到剪贴板

产出：

- `AISuggestion`
- `SendMessageCommand` 草稿状态

### 6.5 阶段 4：人工确认发送

目标：

- 由人工确认最终发送
- 记录发送结果
- 回写消息状态

产出：

- `manual_confirmed` 事件
- `sent` / `failed` / `unknown` 状态

### 6.6 阶段 5：降级与恢复

目标：

- 验证识别失败、页面变更、登录失效时的降级
- 验证恢复后能重新进入正常状态

产出：

- `degraded` / `offline` / `unknown` 状态流转

## 7. 成功标准

PoC 成功不代表平台被完整接入，而是代表这条技术路线值得继续投入。

### 7.1 技术成功标准

- 能稳定识别账号状态
- 能稳定拿到可见消息
- 能稳定回填 AI 建议
- 能稳定进行人工确认发送
- 失败时不会误发或崩溃

### 7.2 工程成功标准

- 适配器接口可复用
- 采集结果可归一到统一模型
- 日志和证据可追踪
- 失败可降级
- 代码不会把平台细节写死在 UI 层

## 8. 风险与约束

### 8.1 主要风险

- 页面结构变化
- 窗口状态不稳定
- OCR 识别错误
- 登录态失效
- 自动化行为被限制
- 误发风险
- 采集结果可信度不足

### 8.2 风险控制

- 只做人工确认发送
- 所有操作可关闭
- 高风险链路与主进程隔离
- 采集数据必须带来源和可信度
- 失败优先降级，不盲目重试

### 8.3 退出条件

PoC 若满足以下情况，应停止继续投入当前路线：

- 连续多轮验证无法稳定拿到可见消息
- 发送结果无法可靠确认
- 风险和维护成本明显高于收益
- 平台环境变化过快，无法维持最小可用能力

## 9. PoC 输出物

PoC 结束后应输出：

- 平台接入评估结论
- 技术手段组合建议
- 数据可信度结论
- 是否进入 MVP 的建议
- 风险清单和降级建议
- 下一步工程任务拆分建议

## 10. 推荐结论

对于当前项目，PoC 的正确姿势不是先证明“我能不能绕过平台”，而是先证明“我能不能在不依赖官方 API 的前提下，把可见内容、人工确认发送和统一模型跑通”。如果这条链路可行，再考虑更稳定的辅助手段和后续扩展。
