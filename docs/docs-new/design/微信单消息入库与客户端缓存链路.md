# 微信单消息入库与客户端缓存链路

本文结合当前代码实现，梳理微信平台收到一条未读消息时，Python 服务端事实库和 C++ 客户端缓存库的写入顺序、关键字段生成、表之间关联。

场景假设：

- 微信未读会话显示名：`邬鸿涛`
- 对方发给我一条文本消息：`你好`
- 服务端数据库中原本没有该会话，也没有该消息
- 客户端缓存库中原本也没有该会话和消息
- C++ 客户端连接 Python 服务时，`WechatRPAAdapter` 使用默认 `accountId()`，因此常见 `account_id = wechat`；如果直接跑 Python sidecar，未传账号时默认值是 `local_wechat`

## 一、事件生成

当前主链路是 `scan_unread_and_fetch`：

1. `python/rpa/platforms/wechat/adapter.py` 扫描未读会话。
2. 选择会话 `邬鸿涛` 后，调用 `read_visible_messages()` 读取当前聊天区。
3. 先生成并 `append` 一个 `conversation_observed` 事件。
4. 再把读取到的消息样本转成 `message_observed` 事件，经过 `filter_observed_message_events()` 过滤后逐条 `append`。

因此，本场景通常会先入库会话事件，再入库消息事件。

关键字段生成规则：

- `account_id`：来自平台命令 payload；C++ 客户端发起连接时通常是 `wechat`
- `conversation_key`：`_conversation_key(account_id, display_name)` 生成，形如 `wechat:wechat:邬鸿涛`
- `occurred_at`：`_now_iso()` 生成的本机时区 ISO 毫秒时间
- `event_id`：`evt_` + `sha1(event_type|conversation_key|occurred_at|payload)[:24]`
- `platform_msg_id`：消息事件中生成，规则是：

```text
raw_id = wechat|<account_id>|<conversation_key>|<direction>|<content_type>|<content>
platform_msg_id = wechat_ + sha1(raw_id)[:24]
```

对这条消息来说：

- `direction = in`
- `content_type = text`
- `content = 你好`
- `sender_role = customer`
- `sender_name = 邬鸿涛`

## 二、服务端事实库写入

服务端入口是 `python/service/rpa_bridge.py` 的 `RpaEventStore.append()`。它会先调用 `PythonServiceTruthStore.persist_event()` 持久化，持久化成功后才把事件放入内存队列并广播给客户端。

### 1. conversation_observed 写入顺序

`persist_event()` 对 `conversation_observed` 的写入顺序是：

1. `_ensure_schema()`
2. `_should_accept_observed_event()`
3. `_append_event_log()`
4. `_upsert_conversation()`
5. `_upsert_platform_conversation()`
6. `commit()`
7. `RpaEventStore.append()` 广播事件给客户端

涉及表：

#### rpa_events

先插入事件日志：

- `event_id`：事件生成时的 `evt_xxx`
- `event_type = conversation_observed`
- `platform = wechat`
- `account_id = wechat`
- `conversation_key = wechat:wechat:邬鸿涛`
- `occurred_at`：事件时间，转成 SQLite 时间文本
- `payload_json`：conversation payload
- `raw_event_json`：完整事件 JSON，后续 replay 依赖它还原事件
- `source_role = python_service_truth`

`event_id` 有唯一约束，重复 append 同一事件时 `INSERT OR IGNORE` 不会重复写日志。

#### conversations

因为该会话不存在，插入一条服务端统一会话记录：

- `id`：服务端自增主键，记为 `server_conv_id`
- `platform = wechat`
- `platform_conversation_id = wechat:wechat:邬鸿涛`
- `account_id = wechat`
- `customer_name = 邬鸿涛`
- `last_message = ''`
- `last_time = occurred_at`
- `unread_count = 0`
- `status = active`
- `updated_at = occurred_at`
- `created_at = CURRENT_TIMESTAMP`

唯一键是 `(platform, platform_conversation_id)`。服务端统一会话 ID 和客户端缓存库会话 ID 是两个不同数据库里的自增 ID，不要求相等。

#### wechat_conversations

随后插入微信扩展会话记录：

- `conversation_id = server_conv_id`
- `wechat_account_id = wechat`
- `wechat_conversation_key = wechat:wechat:邬鸿涛`
- `display_name = 邬鸿涛`
- `last_unread_badge = payload.unread_count`，没有则为 `0`
- `last_observed_at = occurred_at`
- `last_health_status = payload.status`，本场景通常为空
- `raw_payload_json = conversation payload`

`conversation_id` 唯一，指向服务端 `conversations.id`。

### 2. message_observed 写入顺序

消息事件进入 `persist_event()` 后，写入顺序是：

1. `_ensure_schema()`
2. `_should_accept_observed_event()`
3. `_append_event_log()`
4. `_upsert_conversation()`
5. `_upsert_platform_conversation()`
6. `_upsert_message()`
7. `_upsert_platform_message()`
8. `commit()`
9. `RpaEventStore.append()` 广播事件给客户端

涉及表：

#### rpa_events

插入第二条事件日志：

- `event_type = message_observed`
- `platform = wechat`
- `account_id = wechat`
- `conversation_key = wechat:wechat:邬鸿涛`
- `payload_json` 中包含 `platform_msg_id`、`direction`、`sender_role`、`content_type`、`content`、`metadata` 等
- `raw_event_json` 保存完整事件

#### conversations

`_upsert_conversation()` 会先按：

```sql
platform = 'wechat'
platform_conversation_id = 'wechat:wechat:邬鸿涛'
```

找到刚才的 `server_conv_id`，然后更新：

- `account_id = wechat`
- `customer_name = 邬鸿涛`
- `last_message = 你好`
- `last_time = message occurred_at`
- `status = active`
- `deleted_at = NULL`
- `updated_at = message occurred_at`

服务端 `unread_count` 当前不会在这个消息事件里递增，仍保持 `0`。

#### wechat_conversations

再次 upsert 同一条微信扩展会话：

- `conversation_id = server_conv_id`
- `wechat_account_id = wechat`
- `wechat_conversation_key = wechat:wechat:邬鸿涛`
- `display_name = 邬鸿涛`
- `last_observed_at = message occurred_at`
- `raw_payload_json = message payload`

注意：这里的 `raw_payload_json` 会被本次消息 payload 覆盖，不再是纯会话 payload。

#### messages

因为消息不存在，插入服务端统一消息记录：

- `id`：服务端消息自增主键，记为 `server_msg_id`
- `conversation_id = server_conv_id`
- `platform_message_id = platform_msg_id`
- `client_message_id = ''`
- `direction = in`
- `sender = customer`
- `sender_name = 邬鸿涛`
- `content_type = text`
- `content = 你好`
- `status = observed`
- `error_reason = ''`
- `message_time = message occurred_at`
- `created_at = message occurred_at`
- `updated_at = message occurred_at`

去重逻辑优先查 `platform_message_id`，其次查 `client_message_id`。同一个 `platform_msg_id` 再来时会更新这条记录，而不是新增。

#### wechat_messages

最后插入微信消息扩展记录：

- `message_id = server_msg_id`
- `conversation_id = server_conv_id`
- `wechat_account_id = wechat`
- `wechat_conversation_key = wechat:wechat:邬鸿涛`
- `wechat_display_name = 邬鸿涛`
- `platform_message_id = platform_msg_id`
- `direction = inbound`
- `sender_role = customer`
- `source_type = ui_observed`
- `confidence = payload.confidence`
- `verification_status = unverified`
- `original_timestamp = payload.original_timestamp`，为空时用 `occurred_at`
- `content_image_path = ''`
- `evidence_ref = ''`
- `raw_control_name = 你好`
- `raw_control_type = metadata.class_name`
- `role_method = metadata.direction_method`
- `role_confidence = metadata.role_confidence`
- `bubble_rect = metadata.rect`
- `message_list_rect = metadata.chat_context` JSON
- `observation_method = metadata.observation_method`
- `raw_payload_json = message payload`

`message_id` 唯一，指向服务端 `messages.id`。

## 三、推送给客户端

服务端 `RpaEventStore.append()` 在 `persist_event()` 成功后：

1. 给事件补 `seq` 和 `cursor`
2. 把事件放入内存队列
3. 通过事件 WebSocket 广播：

```json
{
  "type": "rpa_event",
  "event": { "...": "..." }
}
```

C++ 客户端 `IpcService` 收到后触发 `platformEventReceived(event)`。`WechatRPAAdapter` 订阅这个信号，只处理 `platform = wechat` 的事件。

当前历史 replay/backfill 开关默认关闭，不影响实时事件推送；实时事件仍会直接进入客户端缓存写入流程。

## 四、客户端缓存库写入

客户端缓存库是本地 `client_cache.db`。它也有 `conversations`、`messages`、`wechat_conversations`、`wechat_messages` 四张相关表，但主键是客户端本地自增 ID，不和服务端 ID 对齐。

客户端现在也使用完整 `conversation_key = wechat:wechat:邬鸿涛` 作为 `platform_conversation_id`；显示名仍单独使用 `payload.display_name`，为空时再从 key 的最后一段推导为 `邬鸿涛`。

### 1. conversation_observed 客户端写入

客户端处理顺序：

1. `IpcService::handleEventSocketPayload()`
2. `WechatRPAAdapter::handleRpaEvent()`
3. `WechatRPAAdapter::emitConversationObserved()`
4. `MessageRouter::onConversationObserved()`
5. `ConversationDao::upsertObservedCacheConversation()`

涉及表：

#### conversations

因为本地没有该会话，插入客户端统一会话记录：

- `id`：客户端自增主键，记为 `client_conv_id`
- `platform = wechat`
- `platform_conversation_id = wechat:wechat:邬鸿涛`
- `account_id = wechat`
- `customer_name = 邬鸿涛`
- `last_message = ''`
- `last_time = datetime('now','localtime')`
- `unread_count = 0`
- `status = active`
- `cache_scope = local_cache`
- `cache_origin = platform_observed_cache`
- `updated_at = datetime('now','localtime')`
- `created_at = CURRENT_TIMESTAMP`

唯一键是 `(platform, platform_conversation_id)`，也就是本地按 `wechat + wechat:wechat:邬鸿涛` 去重。

#### wechat_conversations

当前 `MessageRouter::onConversationObserved()` 只对千牛立即写平台扩展会话；微信的 `wechat_conversations` 不在这个阶段写。微信扩展会话会在后续 `message_observed` 进入 `onIncomingMessage()` 时写入。

### 2. message_observed 客户端写入

客户端处理顺序：

1. `IpcService::handleEventSocketPayload()`
2. `WechatRPAAdapter::handleRpaEvent()`
3. `WechatRPAAdapter::platformMessageFromEvent()`
4. 发出 `incomingMessage(PlatformMessage)`
5. `MessageRouter::onIncomingMessage()`
6. `MessageDao::existsByPlatformMsgId()` 全局查重
7. `MessageRouter::ensureConversation()`
8. `MessageDao::createObservedCacheMessage()`
9. `WechatMessageDao::upsertConversation()`
10. `ConversationDao::updateLastMessage()`
11. `ConversationDao::incrementUnread()`
12. `WechatMessageDao::createMessageExtension()`
13. 发出 `messageReceived`、`conversationUpdated` 等 UI 更新信号

涉及表：

#### conversations

`ensureConversation()` 先按：

```sql
platform = 'wechat'
platform_conversation_id = 'wechat:wechat:邬鸿涛'
```

查本地会话。因为 `conversation_observed` 已经创建过，所以这里直接返回 `client_conv_id`。

随后消息写入完成后更新：

- `last_message = 你好`
- `last_time = message createdAt`
- `updated_at = datetime('now','localtime')`
- 因为方向是 `in`，再执行 `unread_count = unread_count + 1`

所以该场景处理完成后，本地 `conversations.unread_count = 1`。

#### messages

`createObservedCacheMessage()` 插入客户端统一消息记录：

- `id`：客户端自增主键，记为 `client_msg_id`
- `conversation_id = client_conv_id`
- `platform_message_id = platform_msg_id`
- `client_message_id = ''`
- `direction = in`
- `sender = customer`
- `sender_name = 邬鸿涛`
- `content_type = text`
- `content = 你好`
- `status = observed`
- `error_reason = ''`
- `message_time = message createdAt`
- `cache_scope = local_cache`
- `cache_origin = platform_observed_cache`
- `created_at = message createdAt`
- `updated_at = message createdAt`

客户端去重先执行 `existsByPlatformMsgId(platform_msg_id)`，当前是全局按 `messages.platform_message_id` 查，不限定会话。

#### wechat_conversations

在 `onIncomingMessage()` 中写入或更新微信扩展会话：

- `conversation_id = client_conv_id`
- `wechat_account_id = payload._event_account_id`，通常是 `wechat`
- `wechat_conversation_key = payload._event_conversation_key`，即完整 `wechat:wechat:邬鸿涛`
- `display_name = 邬鸿涛`
- `session_control_hash = wechat|wechat:wechat:邬鸿涛`
- `last_unread_badge = payload.unread_count`，没有则为 `0`
- `last_observed_at = datetime('now','localtime')`
- `last_health_status = payload.status`，本场景通常为空
- `raw_payload_json = message payload`

`conversation_id` 唯一，指向客户端 `conversations.id`。

#### wechat_messages

最后写入微信消息扩展记录：

- `message_id = client_msg_id`
- `conversation_id = client_conv_id`
- `wechat_account_id = payload._event_account_id`
- `wechat_conversation_key = payload._event_conversation_key`
- `wechat_display_name = 邬鸿涛`
- `platform_message_id = platform_msg_id`
- `direction = inbound`
- `sender_role = customer`
- `source_type = ui_observed`
- `confidence = payload.confidence`
- `verification_status = unverified`
- `original_timestamp = payload.original_timestamp`，本场景通常为空
- `content_image_path = payload.content_image_path`，文本消息为空
- `evidence_ref = payload.evidence_ref`，文本消息为空
- `raw_control_name = 你好`
- `raw_control_type = metadata.class_name`
- `role_method = metadata.direction_method`
- `role_confidence = metadata.role_confidence`
- `bubble_rect = metadata.rect`
- `message_list_rect = metadata.chat_context` JSON
- `observation_method = metadata.observation_method`
- `raw_payload_json = message payload`

`message_id` 唯一，指向客户端 `messages.id`。

## 五、最终表关联

服务端事实库：

```text
rpa_events
  - conversation_observed 事件日志
  - message_observed 事件日志

conversations.id = server_conv_id
  platform = wechat
  platform_conversation_id = wechat:wechat:邬鸿涛
  last_message = 你好

wechat_conversations.conversation_id -> conversations.id
  wechat_conversation_key = wechat:wechat:邬鸿涛

messages.id = server_msg_id
  conversation_id -> conversations.id
  platform_message_id = platform_msg_id
  content = 你好

wechat_messages.message_id -> messages.id
wechat_messages.conversation_id -> conversations.id
```

客户端缓存库：

```text
conversations.id = client_conv_id
  platform = wechat
  platform_conversation_id = wechat:wechat:邬鸿涛
  last_message = 你好
  unread_count = 1

wechat_conversations.conversation_id -> conversations.id
  wechat_conversation_key = wechat:wechat:邬鸿涛

messages.id = client_msg_id
  conversation_id -> conversations.id
  platform_message_id = platform_msg_id
  content = 你好

wechat_messages.message_id -> messages.id
wechat_messages.conversation_id -> conversations.id
```

## 六、当前实现里的注意点

1. 服务端和客户端 WeChat 本地缓存的 `platform_conversation_id` 都使用完整 key：`wechat:<account_id>:<display_name>`；UI 显示名使用 `customer_name/display_name`，例如 `邬鸿涛`。
2. 服务端持久化成功后才广播实时事件；如果 `_should_accept_observed_event()` 过滤掉事件，客户端不会收到该事件。
3. 服务端 `rpa_events` 是事实事件日志，客户端缓存库没有对应的 `rpa_events` 表。
4. 服务端 `conversation_observed` 阶段会写 `wechat_conversations`；客户端 `conversation_observed` 阶段当前不会写 `wechat_conversations`，要等 `message_observed` 到达后才写。
5. 服务端消息去重依赖 `platform_message_id` 或 `client_message_id`；客户端实时消息去重依赖 `platform_message_id`。
6. 当前 `platform_msg_id` 不包含 `rect`，避免窗口滚动导致同一消息 ID 改变；但连续相同内容消息的 ID 冲突风险是当前阶段接受的限制。
