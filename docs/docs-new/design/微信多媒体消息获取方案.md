# 微信多媒体消息获取方案

日期：2026-06-08

## 1. 背景

当前 `python/rpa/platforms/wechat` 已经能够通过 UIAutomation 定位微信会话消息列表，并读取可见文本消息。

后续需要支持识别并获取以下非文本消息：

- 表情、动画表情
- 图片
- 视频
- 文件

总体原则是：

> UIAutomation 负责判断“是哪条消息、是什么类型、位于哪里”；右键菜单、剪贴板和截图负责获取真实内容。

不建议仅依赖 OCR 或截图判断消息归属。消息归属、方向、顺序和操作目标应继续由 UIA 控件树提供。

## 2. 当前代码链路

当前读取链路为：

```text
WechatSidecarAdapter
  -> WechatVisibleMessageReader.read_visible_messages
  -> collect_chat_message_samples / scored-list fallback
  -> WechatUiaMessageSample
  -> WechatSidecarAdapter._message_event
  -> message_observed
  -> PythonServiceTruthStore
```

### 2.1 UIA 消息定位

`python/rpa/platforms/wechat/uia.py`：

- `find_chat_message_list_control` 通过 `chat_message_list` 或 `mmui::RecyclerListView` 定位消息列表。
- `collect_chat_layout_samples` 遍历消息列表控件。
- `WechatUiaMessageSample` 当前保存消息类型、控件名称、控件类型、AutomationId 和矩形坐标。

`python/rpa/platforms/wechat/reader.py`：

- `WechatVisibleMessageReader.read_visible_messages` 负责读取可见消息。
- `_collect_messages_by_scored_list` 是主 UIA 采集失败时的兜底采集路径。
- `_message_kind` 根据 UIA 控件名称判断消息类型。
- `judge_message_role` 根据气泡图像差异或位置判断消息方向。

### 2.2 事件和存储

`python/rpa/platforms/wechat/adapter.py`：

- `_message_event` 把消息样本转换为 `message_observed` 事件。
- 事件 payload 中包含 `content_type`、`content` 和 UIA metadata。

`python/service/truth_store.py`：

- `messages.content_type` 可以保存统一消息类型。
- `wechat_messages.content_image_path` 可以保存截图或图片路径。
- `wechat_messages.evidence_ref` 可以保存证据文件引用。
- `wechat_messages.raw_payload_json` 可以保留完整媒体提取结果和诊断信息。

因此，第一阶段不需要先修改数据库结构，可以优先利用现有字段。

## 3. 已完成的小步实现

本轮已经打通“UIA 消息类型识别 -> 事件 content_type”的链路。

### 3.1 已支持的类型

| UIA 占位文本 | kind | 事件 content_type |
|---|---|---|
| `[图片]`、`[Image]` | `image` | `image` |
| `ChatBubbleReferItemView + 图片` | `image` | `image` |
| `[动画表情]`、`[表情]`、`[Emoji]` | `emoji` | `emoji` |
| `ChatBubbleReferItemView + 动画表情/表情` | `emoji` | `emoji` |
| `[视频]`、`[Video]` | `video` | `video` |
| `ChatBubbleReferItemView + 视频` | `video` | `video` |
| `[文件]`、`[File]` | `file` | `file` |
| `ChatBubbleReferItemView + 文件` | `file` | `file` |
| 未知类型、引用消息等 | 原始 kind | `text` |

### 3.2 具体实现

- `uia.collect_chat_layout_samples` 支持识别图片、表情、视频、文件占位符。
- `reader._message_kind` 的 scored-list fallback 同步支持这些类型。
- 实机验证发现微信 4.x 会把图片和动画表情暴露为 `mmui::ChatBubbleReferItemView`，`Name` 为裸文本 `图片/动画表情`，已纳入识别。
- 裸文本 `图片/文件/视频/表情` 只有在 `ChatBubbleReferItemView` 上才按媒体处理；普通 `ChatTextItemView` 仍按文本处理，避免误判用户真的发送“图片”两个字。
- `adapter._content_type_from_kind` 使用白名单把 `kind` 映射为 `content_type`。
- `adapter._message_event` 不再固定输出 `content_type=text`。
- 未知 UIA 类型仍然回退为 `text`，避免污染统一消息模型。

### 3.3 阶段二：截图证据

状态：已完成。

本轮已为图片、表情、文件和视频消息增加气泡截图证据：

- 新增 `python/rpa/platforms/wechat/media_evidence.py`。
- `MediaEvidenceWriter` 处理 `image`、`emoji`、`file` 和 `video`，不处理 `text`。
- `WechatVisibleMessageReader` 会把当前微信窗口句柄附加到消息样本。
- `WechatSidecarAdapter._message_event` 在生成 `platform_msg_id` 后尝试保存截图。
- 截图保存成功时写入 `content_image_path` 和 `evidence_ref`。
- 截图失败时只在 metadata 中记录 `evidence_status/evidence_error`，不阻断消息事件。
- 截图保存前会做媒体内容裁剪，去掉整行消息背景、头像、右侧空白和多余上下边距。
- 裁剪只作用于 `MediaEvidenceWriter` 证据保存，不改变 `capture_bubble` 本身，避免影响角色方向判断。

默认保存目录：

```text
python/rpa/_media/wechat/{image|emoji|file|video}/{platform_msg_id}.png
```

该目录已被 `.gitignore` 忽略，属于本地运行产物。

配置项：

```text
media_capture_evidence = true
media_artifact_dir = python/rpa/_media/wechat
```

## 4. 目标架构

建议将后续实现拆成两层。

### 4.1 消息定位层

由 UIAutomation 负责：

- 定位消息列表。
- 找到具体消息控件。
- 判断消息类型。
- 判断入站、出站方向。
- 记录消息顺序和矩形位置。
- 提供后续右键操作需要的目标位置。

建议未来为消息样本补充：

```text
kind
placeholder
rect
target_point
control_signature
raw_control_name
raw_control_type
```

原始 UIA control 可以在单次读取过程中短期保留，但不能写入事件或数据库。跨阶段定位需要依赖控件签名和矩形坐标。

### 4.2 媒体提取层

建议新增独立的媒体提取模块，例如：

```text
python/rpa/platforms/wechat/media_extractor.py
```

统一输入：

```text
消息类型
消息控件或目标坐标
微信窗口句柄
会话信息
```

统一输出建议为：

```text
content_type
content
extraction_method
artifact_path
evidence_ref
file_name
file_size
clipboard_formats
status
error
```

媒体提取失败不能阻止消息事件产生。失败时仍应保存 UIA 占位消息，并在 metadata 中记录失败原因。

## 5. 各类型获取策略

### 5.1 文本

继续使用 UIA 控件的 `Name`、`Value`、`LegacyIAccessibleName` 和 `HelpText`。

```text
content_type = text
content = UIA 读取的文本
```

### 5.2 表情和动画表情

第一优先级使用消息气泡截图裁切。

现有 `python/rpa/platforms/wechat/screenshot.py::capture_bubble` 已能按照控件矩形截取消息行区域。由于微信 4.x 的媒体控件矩形往往是整行而不是气泡本身，证据保存阶段会再调用 `trim_media_evidence` 裁剪最大前景区域，去掉头像和空白背景。

推荐第一版行为：

```text
content_type = emoji
content = [表情] 或 [动画表情]
content_image_path = 当前帧截图
evidence_ref = 当前帧截图
extraction_method = bubble_screenshot
```

动画表情截图只能保留当前帧，不等同于真实动画文件。真实动画文件提取可作为后续增强。

### 5.3 图片

推荐策略顺序：

1. UIA 定位图片消息控件。
2. 根据消息方向选择右键点：入站点击行宽约 25%，出站点击行宽约 75%，避免点到整行空白。
3. 选择“复制”。实机微信 4.x 图片、文件、视频菜单统一使用“复制”把文件或临时文件放入剪贴板 `CF_HDROP`。
4. 检查剪贴板格式。
5. 如果存在 `CF_HDROP`，复制真实文件到统一 artifact 目录。
6. 如果只有位图或 DIB，保存为 PNG。
7. 如果复制失败，使用气泡截图裁切兜底。

推荐输出：

```text
content_type = image
content = [图片] 或文件名
content_image_path = 图片文件或截图路径
evidence_ref = 图片文件或截图路径
extraction_method = clipboard_file / clipboard_bitmap / bubble_screenshot
```

### 5.4 视频

推荐策略顺序：

1. 右键选择“复制”。
2. 如果剪贴板提供 `CF_HDROP`，复制真实视频文件。
3. 如果复制不可用，保存视频消息气泡截图裁切结果。

推荐输出：

```text
content_type = video
content = [视频] 或文件名
evidence_ref = 视频文件路径或封面截图
extraction_method = clipboard_file / bubble_screenshot
```

当前阶段不走“另存为”兜底，避免引入文件选择对话框的稳定性成本。

### 5.5 文件

推荐策略顺序：

1. 从 UIA 文本中解析文件名、大小和状态。
2. 右键选择“复制”。
3. 从剪贴板 `CF_HDROP` 获取本地文件路径。
4. 复制到统一 artifact 目录。
5. 复制失败时保存文件消息气泡截图裁切结果。

文件路径复制已用真实文件卡片完成实机验证，并已接入主读取链路。

推荐输出：

```text
content_type = file
content = 文件名
evidence_ref = 文件路径或卡片截图
extraction_method = clipboard_file / bubble_screenshot
```

## 6. 右键和剪贴板职责

右键操作负责获取真实内容，不负责识别消息归属。

需要保证：

- 右键前重新确认目标消息仍位于预期位置。
- 右键菜单出现后，通过菜单文本或 UIA 控件选择动作。
- 微信 4.x 菜单是 `mmui::XMenu`，菜单项是 `mmui::XMenuView / XMenuItem`，查找应限制在微信菜单根节点下，避免误命中浏览器或编辑器里的“复制”。
- 微信菜单项矩形包含整行，点击几何中心可能落在右侧空白处；当前实现点击菜单项左侧文字区域。
- 操作完成后关闭菜单，避免影响后续扫描。
- 读取剪贴板时支持文本、位图和文件列表格式。
- 操作前保存剪贴板内容，结束后尽量恢复。
- 剪贴板和右键操作需要串行化，避免与消息发送同时操作剪贴板。

现有 `rpa.core.input_sim.ClipboardGuard` 只保存文本剪贴板，无法完整恢复文件列表和位图。媒体提取阶段需要单独设计剪贴板保护策略。

## 7. 消息 ID 和去重风险

当前微信 `platform_msg_id` 主要由以下数据生成：

```text
平台 + 账号 + 会话 + 方向 + content + rect
```

图片、表情、视频等消息的 `content` 经常只是相同占位符，矩形位置也会随着滚动变化。因此后续必须关注：

- 连续发送两张图片时的区分。
- 窗口滚动后同一消息 rect 变化。
- 媒体提取完成前后 content 变化导致消息 ID 变化。
- 未下载和已下载状态切换导致控件文本变化。

短期可以保持现有 ID，媒体提取结果作为同一事件的附加字段。中期应设计更稳定的控件签名或可见消息序列指纹。

## 10. 聚合界面展示接入与验证

状态：已接入，并经实机测试可正常显示在聚合界面。

当前展示链路：

```text
Python WeChat message_observed
-> WechatRPAAdapter::platformMessageFromEvent
-> MessageRouter::onIncomingMessage
-> messages + wechat_messages 入库
-> MessageDao::findById / listCachedMessages
-> AggregateChatForm / ModernMessageItemDelegate
```

关键实现点：

- `src/models/unifiedmodels.h/.cpp` 已补充 `MessageContentType::Emoji` 和 `MessageContentType::Video`，避免表情和视频在统一模型中退化为未知类型。
- `src/services/platforms/wechatrp_adapter.cpp` 会把 payload 中的 `content_type` 写入 `PlatformMessage.contentType`，把 `evidence_ref` 写入 `PlatformMessage.contentImagePath`。
- `src/core/messagerouter.cpp` 会保留平台 payload，并在消息入库后写入 `wechat_messages` 扩展表。
- `src/data/messagedao.cpp` 查询消息时会从 `wechat_messages.content_image_path`、`wechat_messages.evidence_ref` 回填 `MessageRecord.contentImagePath`。其中 `content_image_path` 为空时会继续回退到 `evidence_ref`，用于文件和视频真实文件路径展示。
- `python/service/cache_snapshot.py` 已在服务快照中输出 `content_image_path/evidence_ref`，聚合界面重启或重建本地缓存后仍能恢复媒体展示。
- `src/ui/aggregatechatform.cpp` 的 `ModernMessageItemDelegate` 已按 `contentType` 分流绘制：
  - `image`：显示图片缩略图。
  - `emoji`：显示较小的表情截图缩略图。
  - `file`：显示文件卡片，标题优先从消息文本中解析文件名，详情显示大小/来源；双击可打开本地 artifact。
  - `video`：显示视频卡片，有截图预览时显示预览，否则显示播放占位；双击可打开本地视频 artifact。
  - `text`：保持普通文本气泡。

多条未读混排场景的当前行为：

```text
同一会话一次读到：
1. 文件
2. 图片
3. 文本
4. 表情
```

处理结果：

- reader 按 UIA 气泡纵向位置排序，保持可见消息顺序。
- adapter 为每条消息分别生成 `message_observed`，不会合并多条消息。
- 文件、图片会先尝试右键“复制”；表情走气泡截图；文本只保留文本内容。
- C++ 侧每条消息单独入库、单独触发 UI append。
- 聚合界面按顺序显示为：文件卡片、图片缩略图、文本气泡、表情缩略图。

已验证：

- 图片、表情、文件、视频可以在聚合界面按对应形态正常显示。
- 文件和视频真实文件路径写入 `evidence_ref` 后，聚合界面可以通过 `MessageRecord.contentImagePath` 使用该路径展示卡片和双击打开。
- 图片真实文件路径写入 `content_image_path/evidence_ref` 后，聚合界面显示真实图片缩略图。
- 表情截图路径写入 `content_image_path/evidence_ref` 后，聚合界面显示裁剪后的表情缩略图。
- 服务快照恢复链路保留媒体路径，避免重启后媒体消息退化为纯文本。

当前限制：

- 文件和视频复制失败时会回退为气泡截图证据；界面仍显示文件/视频卡片，但双击打开的是截图证据而不是真实文件。
- 动画表情当前保存的是当前帧截图，不是真实动画文件。
- 多媒体采集仍依赖右键菜单和系统剪贴板，需要继续补充剪贴板保护和串行化策略。
- `platform_msg_id` 仍包含气泡 rect，滚动或窗口位置变化后仍有重复入库风险，后续需要优化稳定 ID。

## 11. 聚合界面点击预览与打开策略

目标：在当前“气泡内展示缩略图/卡片”的基础上，补齐用户点击后的查看体验。第一版优先稳定、轻量，不把聚合界面做成完整文档编辑器或复杂媒体播放器。

状态：已完成第一版实现。

当前实现入口：

```text
AggregateChatForm::openMessageMedia
```

消息列表点击媒体消息后，会根据 `MessageRecord.contentType` 和 `MessageRecord.contentImagePath` 分流到图片预览、文件系统打开、视频预览或截图证据预览。

### 11.1 图片和表情

当前气泡内已经显示图片缩略图和表情缩略图。

- 预览对话框展示 `MessageRecord.contentImagePath` 指向的图片文件。
- 如果图片来源是真实图片文件，展示原图。
- 如果图片来源是截图兜底，展示截图证据。

### 11.2 文件

当前气泡内已经显示文件卡片，包含文件名、大小/来源和文件图标。

当前点击行为：

- 点击文件卡片时，优先使用系统默认程序打开本地 artifact 文件。
- Word、PPT、Excel、PDF、WPS 文档等普通文档不在聚合界面内嵌预览，而是交给 Windows 文件关联处理，例如由 WPS、Office、PDF 阅读器或浏览器打开。

不建议第一版实现内嵌 Word/PPT/PDF 预览。原因：

- Office/WPS 文档内嵌预览依赖复杂，稳定性和兼容性成本高。
- 文档编辑工具已经能更好处理格式、字体、批注、宏、权限和只读状态。
- 聚合界面第一阶段只需要完成“客服能快速打开查看原文件”的目标。

兜底规则：

- 如果路径存在且是复制出的真实文件，直接系统打开。
- 如果路径只指向气泡截图证据，则弹图片预览对话框展示证据，并提示这不是原始文件。
- 如果路径不存在或不可读，则提示文件不可用。

### 11.3 视频

当前气泡内已经显示视频卡片。有截图预览时显示预览图，否则显示播放占位。

当前点击行为：

- 支持播放/暂停、进度条和时长显示。
- 预览窗口默认尺寸与图片预览对话框一致：`860x620`，最小尺寸 `420x320`。
- 提供“系统打开”动作，仍可交给系统默认播放器打开。

已实现 `VideoPreviewDialog`：

- `CMakeLists.txt` 已引入 `Qt6::Multimedia` 和 `Qt6::MultimediaWidgets`。
- 视频预览使用 `QMediaPlayer`、`QAudioOutput` 和 `QVideoWidget`。
- 点击视频文件不再直接拉起可能全屏的系统默认播放器，而是先进入可控尺寸的内置预览窗口。
- 如果视频复制失败、只有气泡截图证据，则弹图片预览对话框展示截图证据，而不是尝试播放。

### 11.4 统一点击判断

建议把点击行为统一抽象为媒体打开入口，按 `content_type` 和路径状态分流：

```text
content_type=image/emoji:
  path 是可读图片 -> 图片预览对话框
  path 不存在或不可读 -> 提示不可预览

content_type=file:
  path 存在且是原始文件 -> 系统默认程序打开
  path 是截图兜底 -> 图片预览对话框展示证据
  path 不存在或不可读 -> 提示文件不存在

content_type=video:
  path 存在且是视频文件 -> 内置视频预览对话框
  path 是截图兜底 -> 图片预览对话框展示证据
  path 不存在或不可读 -> 提示视频文件不存在
```

路径是否为截图兜底可以按以下规则判断：

- `content_type=file/video` 且路径后缀是 `.png/.jpg/.jpeg/.webp/.bmp` 时，优先视为截图证据。
- 路径位于 `python/rpa/_media/wechat/{file|video}/` 但实际文件扩展名为文档或视频扩展名时，视为真实 artifact。
- 后续可在 metadata 中显式保存 `artifact_kind=original_file/screenshot_evidence`，减少依赖扩展名判断。
