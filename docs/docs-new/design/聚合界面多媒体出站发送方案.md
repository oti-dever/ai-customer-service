# 聚合界面多媒体出站发送方案

日期：2026-06-10

## 1. 背景与目标

当前客户端聚合界面只支持把文本消息发送到微信、千牛平台。实际客服使用中，微信和千牛平台原生输入框均支持粘贴图片、视频、文件并发送，因此聚合界面也需要支持类似微信输入框的粘贴体验：

- 输入框可粘贴图片、视频、文件。
- 粘贴后在输入区内展示附件预览：图片缩略图、视频卡片、文件卡片。
- 输入框可同时保留文本和附件。
- 点击发送时，如果同时存在文本和附件，必须拆成多条消息发送。
- 首版目标平台为微信和千牛。

本文只规划设计方案，不包含代码实现。

## 2. 当前代码实现结论

### 2.1 聚合输入区仍是纯文本

聚合输入框当前由 `AggregateChatForm` 创建，控件类型是 `QPlainTextEdit`：

- `src/ui/aggregatechatform.cpp`：创建 `m_inputEdit = new QPlainTextEdit(inputInner)`。
- `src/ui/aggregatechatform.cpp`：`onSendClicked()` 中通过 `m_inputEdit->toPlainText().trimmed()` 获取文本。
- `onSendClicked()` 只在文本非空时发送，随后清空输入框并调用 `ConversationManager::instance().sendMessage(m_currentConvId, text)`。

因此当前聚合输入区没有附件状态，也没有从剪贴板解析图片、视频、文件的逻辑。

### 2.2 出站链路接口是纯文本签名

当前出站调用链为：

```text
AggregateChatForm::onSendClicked
  -> ConversationManager::sendMessage(int conversationId, QString text, QString clientMessageId)
  -> MessageRouter::sendMessage(int conversationId, QString text, QString clientMessageId)
  -> IPlatformAdapter::sendMessage(QString platformConversationId, QString text, QString clientMessageId)
  -> WechatRPAAdapter / QianniuRPAAdapter
  -> Python service command: send_message(text)
```

关键接口：

- `src/core/conversationmanager.h`
- `src/core/messagerouter.h`
- `src/services/platforms/iplatformadapter.h`
- `src/services/platforms/wechatrp_adapter.cpp`
- `src/services/platforms/qianniurp_adapter.cpp`

这些接口都以 `QString text` 作为核心载荷，不能表达图片、视频、文件路径、文件名、MIME、大小等信息。

### 2.3 MessageRouter 固定创建文本 pending 消息

`MessageRouter::sendMessage()` 当前创建 pending 出站消息时固定设置：

```cpp
pendingMessage.contentType = Models::MessageContentType::Text;
pendingMessage.content = text;
pendingMessage.status = Models::MessageStatus::Pending;
```

随后调用 `MessageDao::createOutboundCacheMessage(pendingMessage)`，再把文本交给平台适配器发送。

这意味着即使 UI 能拿到文件路径，当前路由层也无法为附件创建正确的 `content_type=image/video/file` pending 消息。

### 2.4 消息展示侧已经具备多媒体基础

现有模型和展示侧已经不是纯文本：

- `src/models/unifiedmodels.h` 的 `MessageContentType` 已包含 `Image`、`File`、`Video`、`Emoji`。
- `src/core/types.h` 的 `MessageRecord` 已包含 `contentType` 和 `contentImagePath`。
- `src/ui/aggregatechatform.cpp` 中已有图片、视频、文件气泡绘制逻辑：
  - 图片缩略图。
  - 视频卡片与视频预览弹窗。
  - 文件卡片与本地文件打开。
- `MessageListModel::signature()` 已把 `contentImagePath` 纳入签名，说明 UI 刷新已经考虑媒体路径变化。

因此，多媒体出站不需要从零设计消息展示，重点是补齐“输入态 + 出站载荷 + RPA 发送 + pending 落库”。

### 2.5 数据库存储路径需要注意

主表 `messages` 保存统一消息字段：

- `content_type`
- `content`
- `status`
- `client_message_id`

而平台扩展表保存媒体路径：

- `wechat_messages.content_image_path`
- `wechat_messages.evidence_ref`
- `qianniu_messages.content_image_path`
- `qianniu_messages.evidence_ref`

`MessageDao::messageSelectProjection()` 查询时会从微信/千牛扩展表聚合出 `content_image_path`。

当前 `MessageDao::createOutboundCacheMessage()` 只插入主 `messages` 表，不会为 out 消息同步写入 `wechat_messages` 或 `qianniu_messages` 扩展表。因此首版若要在点击发送后立即显示附件气泡，需要补一层 DAO 能力，把出站附件路径也写入对应平台扩展表。

### 2.6 Python RPA 发送侧当前只支持文本

微信侧：

- `python/rpa/platforms/wechat/adapter.py::_send_message()` 读取 `params["text"]`。
- 先调用 `_prepare_reply_draft()` 写入文本草稿。
- 再调用 `WechatMessageSender.send_prepared_message()` 点击发送按钮。
- `WechatMessageSender.prepare_reply_draft()` 只支持文本写入或文本剪贴板粘贴。

千牛侧：

- `python/rpa/platforms/qianniu/adapter.py::_send_message()` 读取 `params["text"]`。
- 调用 `QianniuSender.send_text()`。
- `QianniuSender.send_text()` 只负责文本输入和回车发送。

微信侧已有 `python/rpa/platforms/wechat/media_clipboard.py`，能读取 `CF_HDROP` 文件路径和剪贴板图片，但用途主要是入站多媒体获取，不是出站发送。

## 3. 产品交互设计

### 3.1 输入框表现

聚合输入区建议改造成一个组合输入控件，视觉上保持一个输入框：

```text
+------------------------------------------------------+
| [图片缩略图] [视频卡片] [文件卡片]                    |
| 文本输入光标与多行文本                               |
|                                                      |
| 表情 / 工具 / 文件按钮                 [发送]         |
+------------------------------------------------------+
```

附件展示规则：

- 图片：显示缩略图，保留宽高比，可点击预览，可删除。
- 视频：显示文件名、大小、视频图标或首帧缩略图，可删除。
- 文件：显示文件名、大小、文件类型图标，可删除。
- 多个附件按粘贴顺序排列。
- 附件区和文本区属于同一个输入面板，整体高度随内容自适应，但需要设置最大高度，超出后内部滚动。

### 3.2 粘贴行为

输入控件需要拦截 paste：

- `QMimeData::hasImage()`：表示剪贴板中是位图，保存为本地临时 PNG，再加入附件列表。
- `QMimeData::hasUrls()`：表示剪贴板中有文件路径，逐个校验文件存在性和可读性，再按 MIME/扩展名归类为图片、视频、文件。
- `QMimeData::hasText()`：表示有文本，按普通文本插入当前光标位置。

如果一次粘贴同时包含文件和文本：

- 文件进入附件列表。
- 文本进入文本编辑器。

建议首版支持的常见扩展：

- 图片：`png`、`jpg`、`jpeg`、`bmp`、`gif`、`webp`。
- 视频：`mp4`、`mov`、`avi`、`mkv`、`wmv`。
- 文件：其他可读普通文件。

### 3.3 发送拆分规则

点击发送时，把当前输入状态拆成多个发送任务：

1. 若只有文本：发送一条文本消息。
2. 若只有附件：每个附件发送一条独立消息。
3. 若附件和文本都存在：先按附件粘贴顺序逐条发送附件，最后发送文本。

每条拆分消息必须有独立的：

- `client_message_id`
- pending 消息记录
- sent/failed 状态
- 失败原因

不建议把“多个附件 + 文本”塞成一条复杂消息，因为当前 UI、数据库、发送状态、RPA 回执都更适合“一条出站任务对应一条消息”。

## 4. 客户端架构设计

### 4.1 新增统一出站载荷

建议新增出站载荷结构，例如：

```cpp
enum class OutgoingPartType {
    Text,
    Image,
    Video,
    File,
};

struct OutgoingMessagePart {
    OutgoingPartType type = OutgoingPartType::Text;
    QString text;
    QString localPath;
    QString fileName;
    qint64 sizeBytes = 0;
    QString mimeType;
};

struct OutgoingMessagePayload {
    QVector<OutgoingMessagePart> parts;
};
```

文本消息可以继续使用旧重载：

```cpp
sendMessage(int conversationId, const QString& text, const QString& clientMessageId = QString());
```

并新增多媒体发送入口：

```cpp
sendMessage(int conversationId, const OutgoingMessagePart& part, const QString& clientMessageId = QString());
sendPayload(int conversationId, const OutgoingMessagePayload& payload);
```

这样能减少对 AI 自动回复、测试、模拟平台的冲击。

### 4.2 聚合输入控件

建议新增 `AggregateComposeInput`，不要继续把所有逻辑堆在 `AggregateChatForm`。

职责：

- 管理文本编辑器。
- 管理待发送附件列表。
- 处理粘贴事件。
- 提供 `payload()`。
- 提供 `clear()`、`isEmpty()`。
- 支持草稿序列化和恢复。
- 发出内容变化信号，用于驱动发送按钮状态、输入区高度重算、草稿保存。

`AggregateChatForm` 保持编排职责：

- 当前会话切换。
- 发送按钮处理。
- 调用 `ConversationManager`。
- 清理草稿。
- 展示状态提示。

### 4.3 草稿保存

当前 `conversation_drafts.content` 是纯文本。建议升级为兼容 JSON：

```json
{
  "version": 1,
  "text": "123",
  "attachments": [
    {
      "type": "image",
      "path": "D:/ai-customer-service/.codex_tmp/compose/a.png",
      "name": "a.png",
      "size": 12345,
      "mime": "image/png"
    }
  ]
}
```

兼容策略：

- 如果 `content` 可解析为 JSON 且包含 `version`，按新草稿恢复。
- 如果解析失败，按旧纯文本草稿恢复。

首版也可以先只支持文本草稿持久化，附件切换会话即丢失，但这会和用户对微信式输入框的预期不一致。

### 4.4 pending 消息落库

发送附件时，`MessageRouter` 应创建正确的 pending 消息：

- 图片：
  - `content_type = image`
  - `content = [图片]` 或文件名
  - 平台扩展表 `content_image_path/evidence_ref = localPath`
- 视频：
  - `content_type = video`
  - `content = 文件名 + 大小`
  - 平台扩展表 `content_image_path/evidence_ref = localPath`
- 文件：
  - `content_type = file`
  - `content = 文件名 + 大小`
  - 平台扩展表 `content_image_path/evidence_ref = localPath`
- 文本：
  - 保持现状。

建议新增 DAO 方法：

```cpp
int MessageDao::createOutboundCacheMessage(const Models::Message& message,
                                           const QString& platform,
                                           const QString& localMediaPath,
                                           const QJsonObject& rawPayload);
```

或新增更明确的方法：

```cpp
int MessageDao::createOutboundMediaCacheMessage(...);
```

目标是保证创建 pending 后，当前消息列表能立刻展示正确的图片、视频、文件气泡。

### 4.5 会话 last_message

附件发送后，会话列表最后一条消息建议显示：

- 图片：`[图片]`
- 视频：`[视频] 文件名`
- 文件：`[文件] 文件名`
- 文本：文本摘要

如果一次发送多个附件加文本，按实际拆分顺序更新，最终 last_message 通常会落到最后发送的文本；如果没有文本，则落到最后一个附件。

## 5. C++ 到 Python 命令协议设计

当前命令参数只有文本：

```json
{
  "conversation_key": "...",
  "display_name": "...",
  "text": "...",
  "client_message_id": "...",
  "confirm_token": "manual_confirmed_by_agent"
}
```

建议扩展为：

```json
{
  "conversation_key": "...",
  "display_name": "...",
  "client_message_id": "...",
  "confirm_token": "manual_confirmed_by_agent",
  "content_type": "image",
  "text": "",
  "file_path": "D:/path/a.png",
  "file_name": "a.png",
  "mime_type": "image/png",
  "size_bytes": 12345

```

兼容规则：

- `content_type` 缺失时，按 `text` 处理。
- `content_type=text` 时，继续走原有文本逻辑。
- `content_type=image/video/file` 时，必须有 `file_path`。

Python 回执事件建议带回：

```json
{
  "event_type": "message_sent",
  "conversation_key": "...",
  "client_message_id": "...",

    "status": "sent",
    "content_type": "image",

    "file_path": "D:/path/a.png",
    "file_name": "a.png",
    "send_method": "clipboard_file+send_button"
  }
}
```

失败事件：

```json
{
  "event_type": "send_failed",
  "conversation_key": "...",
  "client_message_id": "...",
  "payload": {
    "error_message": "file_not_found",
    "content_type": "file",
    "file_path": "D:/path/a.docx"
  }
}
```

## 6. Python RPA 发送设计

### 6.1 媒体发送核心策略

微信和千牛均支持把文件从系统剪贴板粘贴到输入框，因此 Python 侧可复用一套核心策略：

```text
校验文件存在
  -> 选中/校验目标会话
  -> 聚焦平台输入框
  -> 写 Windows 剪贴板 CF_HDROP
  -> Ctrl+V 粘贴
  -> 等待平台输入框出现附件卡片或发送按钮可用
  -> 点击发送按钮或回车
  -> 发送 message_sent / send_failed 事件
```

注意：不能依赖用户粘贴到聚合界面时的系统剪贴板仍然有效。聚合界面必须把附件路径或临时图片路径传给 Python，Python 发送时重新设置系统剪贴板。

### 6.2 新增剪贴板写入能力

微信侧已有 `media_clipboard.py` 支持读取 `CF_HDROP`。出站需要新增写入能力：

```python
set_clipboard_file_paths(paths: list[Path]) -> bool
```

Windows 实现需要写入 `CF_HDROP` 数据结构。首版可先支持单文件，后续再支持多文件同时粘贴。但由于发送规则已经拆成“一附件一消息”，单文件能力足够。

### 6.3 微信发送

建议在 `WechatMessageSender` 新增：

```python
send_media(
    file_path: str,
    content_type: str,
    display_name: str = "",
    allow_foreground_fallback: bool = True,
) -> SendResult
```

行为：

- 复用 `_ensure_target_session_selected()` 校验目标会话。
- 复用 `find_chat_input()` 和 `focus_chat_input()`。
- 设置 `CF_HDROP`。
- 发送 Ctrl+V。
- 等待短时间，检查发送按钮是否可用。
- 点击发送按钮。

微信当前文本发送存在 background 写入和 foreground fallback。媒体发送基本只能走 foreground，因为剪贴板粘贴和附件卡片渲染依赖真实焦点。需要在日志中明确 `foreground=true`。

### 6.4 千牛发送

建议在 `QianniuSender` 新增：

```python
send_media(
    file_path: str,
    content_type: str,
    dry_run: bool = False,
    chat_root: Any | None = None,
    input_field: Any | None = None,
) -> SendResult
```

行为：

- 复用 `_resolve_input_field()` 定位输入框。
- 聚焦输入框。
- 设置 `CF_HDROP`。
- Ctrl+V 粘贴。
- 等待附件卡片进入输入框。
- 回车或点击发送。

千牛当前 `_send_message()` 使用 `current_session_chat_root()`，目标会话校验弱于微信。由于多媒体发送失败代价更高，建议在本需求中同步补强千牛目标会话选择/校验，至少保证 `display_name/conversation_key` 和当前聊天标题一致。

#### 6.4.1 千牛目标会话校验与切换策略

千牛媒体发送采用“先轻量校验，不命中再 fresh scan 切换，点击后必须二次校验”的混合策略，不采用“永远直接发送”，也不采用“每条消息都无脑扫描并点击切换”。

推荐新增内部能力：

```python
ensure_target_session_selected(display_name: str, conversation_key: str) -> EnsureSessionResult
```

执行流程：

```text
读取当前聊天标题 / 当前聊天上下文
  -> 如果当前会话等于目标会话：直接返回 ok
  -> 如果当前会话不是目标会话：
      -> fresh scan 当前可见会话列表
      -> 按 display_name / conversation_key 精确匹配目标 SessionItem
      -> 点击目标 SessionItem
      -> 等待会话切换稳定
      -> 二次读取当前聊天标题 / 当前聊天上下文
      -> 如果二次校验命中目标：返回 ok
      -> 如果二次校验未命中：重新 fresh scan 后重试 1-2 次
      -> 仍失败：返回 target_session_not_verified，禁止发送
```

这样设计的原因：

- 如果当前已经在目标会话，直接发送最快，也避免额外点击打断输入区。
- 如果当前不是目标会话，必须重新扫描会话列表，不能依赖旧缓存的 `SessionItem`，因为新客户消息会导致千牛会话列表动态变化。
- 扫描和点击本身仍不够安全，因为列表可能在“扫描结果生成”和“点击坐标执行”之间变化，所以点击后必须二次校验当前聊天确实切到目标。
- 多媒体发送依赖真实焦点和系统剪贴板，误发成本高于文本，因此校验失败时宁可中断，也不能降级为“发到当前会话”。

切换方式首选当前已有的会话列表能力：

```text
QianniuSessionReader.read_visible_sessions(limit=100)
  -> 精确匹配目标 title
  -> QianniuSessionReader.select_session(item)
      -> 优先 rect_click
      -> 失败后 fallback 到 UIA SelectionItem/Invoke
```

如果目标不在当前可见会话列表，第二阶段可考虑使用千牛会话搜索框：

```text
find_chat_search_input
  -> 输入目标会话名
  -> 等待搜索结果
  -> 点击目标会话
  -> 二次校验当前聊天
```

首版可以先不实现搜索 fallback，但必须明确：可见列表找不到目标时禁止发送，返回 `target_session_not_found` 或 `target_session_not_verified`。

#### 6.4.2 多条拆分消息下的校验规则

对于“附件 + 文本”拆分出的多条消息，千牛发送应按会话维度串行，并在每条发送前校验目标：

- 第一条消息发送前：执行完整 `ensure_target_session_selected()`。
- 后续每条消息发送前：先做轻量校验，确认当前仍是目标会话。
- 如果轻量校验失败：重新 fresh scan 并切换。
- 任意一条切换或二次校验失败：停止后续发送，避免误发。

这能兼顾安全和效率：第一条负责把平台切到正确会话，后续消息不重复无意义点击，但仍防止观察线程、新消息跳动或人工操作导致焦点漂移。

## 7. 发送队列与状态处理

### 7.1 为什么需要队列

“附件 + 文本”拆成多条消息后，不能并发发送到同一平台输入框，否则会出现：

- 文件还未粘贴完成，文本已经覆盖输入框。
- 多个附件顺序错乱。
- 回执无法对应 pending 消息。

因此聚合发送应按会话维度串行执行。

### 7.2 建议队列语义

首版可以在 `AggregateChatForm` 或 `MessageRouter` 内部简单串行：

```text
发送 part 1
  -> 创建 pending
  -> 调用 adapter
发送 part 2
  -> 创建 pending
  -> 调用 adapter
...
```

但更稳妥的是在 `MessageRouter` 或平台适配器层维护队列，因为当前 `WechatRPAAdapter` 和 `QianniuRPAAdapter` 已有 `m_commandInFlight`，说明平台命令本身需要串行。

建议原则：

- 每个 part 一条命令。
- 每条命令独立超时。
- 某个附件失败后，后续 part 是否继续发送需要产品定案。

推荐首版策略：

- 附件失败时停止后续发送，并保留未发送部分在输入框或提示用户。
- 文本失败时只标记文本消息失败，不回滚已发送附件。

## 8. 临时文件与清理策略

### 8.1 剪贴板位图

当用户直接复制图片内容，而不是复制图片文件路径时，Qt 只能拿到图片数据。聚合客户端需要把图片保存为本地临时文件，例如：

```text
D:/ai-customer-service/.codex_tmp/compose/{uuid}.png
```

或产品运行缓存目录：

```text
<app-data>/compose-media/{yyyyMMdd}/{uuid}.png
```

然后按普通图片附件发送。

### 8.2 原始文件路径

当用户粘贴的是文件路径时，首版可以直接引用原路径，不复制文件。发送前校验文件仍存在。

如果要增强可靠性，可以在粘贴时复制到应用缓存目录，避免用户发送前移动/删除原文件。但复制大视频会增加耗时和磁盘占用，建议首版只复制剪贴板位图，不复制已有文件路径。

### 8.3 清理

仅清理由剪贴板位图生成的临时文件，不清理用户原始文件。

清理时机：

- 发送成功后一段时间清理。
- 应用启动时清理超过 N 天的 compose 缓存。
- 草稿仍引用的临时文件不能清理。

## 9. 验收场景

### 9.1 UI 输入

- 在聚合输入框粘贴图片文件，出现图片缩略图。
- 在聚合输入框粘贴剪贴板位图，保存为临时 PNG 并出现缩略图。
- 粘贴视频文件，出现视频卡片。
- 粘贴 docx/pdf/xlsx 等文件，出现文件卡片。
- 附件可删除。
- 附件和文本可同时存在。
- 切换会话后草稿可恢复，或首版明确提示附件草稿不保留。

### 9.2 微信发送

- 只发文本，旧链路不回归。
- 只发图片，微信聊天窗口收到图片。
- 只发视频，微信聊天窗口收到视频。
- 只发文件，微信聊天窗口收到文件。
- 图片 + 文本，微信中表现为两条消息。
- 文件 + 视频 + 文本，微信中按顺序发送多条消息。

### 9.3 千牛发送

- 同微信发送场景。
- 发送前目标会话必须正确。
- 当前会话不匹配时不能误发。
- 当前已经是目标会话时，轻量校验通过后直接发送，不额外点击会话列表。
- 当前不是目标会话时，必须 fresh scan 会话列表，命中目标后切换，并在点击后二次校验当前聊天标题。
- 会话列表因新客户消息发生变化时，如果点击后校验不是目标会话，必须停止或重扫重试，不能继续发送。
- 多条拆分消息发送时，第一条执行完整校验/切换，后续每条发送前至少做轻量校验；校验失败则停止后续发送。

### 9.4 状态与展示

- 点击发送后，聚合消息区立即出现 pending 附件气泡。
- 发送成功后状态变为 sent。
- 发送失败后状态变为 failed，并展示失败原因。
- 会话列表 last_message 能显示 `[图片]`、`[视频] 文件名`、`[文件] 文件名`。

## 10. 主要风险

1. Windows 剪贴板 `CF_HDROP` 写入实现需要谨慎，必须正确构造 DROPFILES 结构，否则微信/千牛无法识别为文件粘贴。
2. 视频和大文件粘贴后平台可能需要上传时间，不能沿用文本发送的短确认超时。
3. 千牛当前发送侧目标会话校验较弱，多媒体发送前需要补强，避免误发。
4. 出站附件 pending 需要写入平台扩展表，否则聚合气泡无法立即显示本地文件。
5. 用户原始文件可能在发送前被移动或删除，需要发送前再次校验并给出失败原因。
6. 剪贴板位图保存出的临时文件需要生命周期管理，避免缓存无限增长。

## 11. 推荐实施顺序

1. 新增客户端出站载荷结构，保留旧文本重载。
2. 扩展 `MessageRouter` 和 DAO，支持创建图片、视频、文件 pending 消息，并写入平台扩展表。
3. 新增聚合输入组合控件，支持粘贴、预览、删除、payload 输出。
4. 扩展 C++ 到 Python 的 `send_message` 命令参数。
5. Python 侧实现单文件 `CF_HDROP` 写入和微信媒体发送。
6. Python 侧实现千牛媒体发送，并补强目标会话校验。
7. 加入发送队列与拆分发送策略。
8. 补充回归测试和人工验收脚本。

## 12. 设计取舍

首版不建议把多媒体做成“上传到中间服务再由平台 API 发送”，因为当前项目主链路是 RPA 操作真实平台输入框，且微信/千牛输入框已支持粘贴发送。

首版也不建议把多个附件合并成一条聚合消息，因为现有数据库、气泡展示、发送状态、回执模型都天然以“一条消息一个状态”为单位。拆分成多条消息更贴近平台实际表现，也更容易定位失败。

### 13.2 pending 消息与平台扩展表

`MessageRouter` 已支持媒体 pending 消息：

- 图片：`content_type=image`，`content=[图片]`。
- 视频：`content_type=video`，`content=[视频] 文件名`。
- 文件：`content_type=file`，`content=[文件] 文件名`。
- 文本：保持原 `content_type=text`。

媒体发送前会校验本地文件存在；不存在则直接 `messageSendFailed`，不调用平台 adapter。

媒体 pending 入库后，会同步写入平台扩展表：

- 微信：`wechat_messages.content_image_path/evidence_ref`
- 千牛：`qianniu_messages.content_image_path/evidence_ref`

这样聚合消息列表可以立刻通过现有 `MessageDao::messageSelectProjection()` 读到本地媒体路径，并展示图片缩略图、视频/文件卡片。
