# 仓库重构分析与优化方案

## 1. 结论先行

当前仓库适合继续重构，不适合推倒重来。

原因很直接：仓库里已经有可运行的 Qt/C++ 主程序、SQLite 持久化、AI 访问封装、平台适配器骨架、以及可工作的 Python RPA/自动化脚本。问题不是“没有资产”，而是资产分散、边界混杂、历史实现和 MVP 目标并不完全一致。

对当前 `dev/20260524-wg` 分支更合理的做法是：

1. 保留可跑通的桌面主壳和数据库能力
2. 把统一模型、Core、Platform、IPC、Storage 的边界收紧
3. 把 UI 中的 RPA 逻辑逐步抽出
4. 把 `python/rpa` 收敛成独立辅助服务，不再侵入主 UI
5. 先把模拟链路和单平台 PoC 跑通，再谈多平台演进

## 2. 仓库现状概览

### 2.1 C++ / Qt 主工程入口

当前主入口是 `src/main.cpp`，启动流程已经包含：

- `QApplication`
- 数据库初始化
- 登录窗口
- 主窗口
- 平台初始化

这说明桌面主壳已经存在，不需要重做。

### 2.2 `src/core`

现有 `src/core` 已经承担了部分核心职责：

- `ConversationManager`
- `MessageRouter`
- `AuthManager`
- `PlatformBootstrap`
- `types.h`

其中 `MessageRouter` 已经在做消息分发、去重、入库和状态回写，说明核心调度逻辑已经萌芽，但还没有完全按统一领域模型收敛。

### 2.3 `src/data`

`src/data` 里是现有 SQLite 数据访问层：

- `database.cpp/h`
- `conversationdao`
- `messagedao`
- `messagesendeventdao`
- `aiassistantdao`
- `userdao`

这部分和 MVP 的本地存储目标高度相关，值得保留，但需要从“业务表驱动”逐步过渡到“统一模型驱动”。

### 2.4 `src/services`

现有 `src/services` 分四块：

- `services/platforms`：微信 / 千牛 / 拼多多适配器
- `services/ai`：AI Provider、请求组装、OpenAI 兼容调用、流式会话
- `services/app`：应用层编排
- `services/wechat`：微信工作台相关服务

这里是当前最需要重构边界的地方。它已经不是纯 UI 辅助层，而是混合了应用服务、平台接入、AI 调用和部分业务编排。

### 2.5 测试资产

仓库里已经有 `tests/`，覆盖了部分 C++ 服务、数据访问、AI 抽象、消息路由，以及千牛/微信解析类 Python 测试。这是重构时的重要保护网，应保留并在模型、路由、存储迁移时同步扩展。

### 2.6 `src/ui`

UI 里现有内容很多，包括：

- 主窗口、登录、帮助、配置页
- 聚合会话窗口
- RPA 管理、RPA 控制台、RPA 进程控制
- 微信工作台窗口
- AI 客服后台窗口

这些界面能说明产品已经有相当功能量，但也意味着 UI 层现在承载了过多流程控制和平台操作。

### 2.7 `src/utils`

`src/utils` 主要是工具层：

- 日志
- 加密
- Windows 窗口辅助
- 光标控制
- 图片/资源辅助
- 样式和配置

这层总体可保留，但要防止继续长出业务逻辑。

### 2.8 Python `python/rpa`

`python/rpa` 已经不是空壳，而是一套完整的读写链路：

- `readers/`
- `writers/`
- `common/`
- `tools/`
- `config/`
- `db_helper.py`

它更像一个独立的自动化/采集子系统，而不是 C++ 主程序的附属脚本。当前最大问题是它和主程序共享数据库、共享业务状态、共享部分流程意图，边界还不够硬。

### 2.9 `docs/`

现在 `docs/docs-new` 已经补齐了 MVP 设计文档，`docs/docs-old` 保留了旧方案、旧 RPA 方案和历史设计。这个结构是合理的：

- `docs-new` 作为当前收敛后的主线
- `docs-old` 作为历史参考

### 2.10 资源目录

`resources/` 很完整，包含大量图标、logo、svg 和 UI 资源。它们是可复用资产，基本不该动，最多做命名归类，不必在重构早期重做。

### 2.11 数据库文件

`database/app.db` 已存在，且代码里有迁移逻辑。这说明仓库已经有真实运行数据和历史状态，不适合粗暴重置。数据库应视作现有兼容资产，重构时以迁移和兼容为主。

## 3. 架构差距分析

### 3.1 数据模型是否统一

不统一。

当前 `src/core/types.h` 里的 `PlatformMessage`、`ConversationInfo`、`MessageRecord` 仍是偏实现导向的结构，和 `docs/docs-new/design/统一数据模型与状态机.md` 定义的 `Conversation`、`Message`、`PlatformAccount`、`ConversationEvent`、`SendMessageCommand`、`AISuggestion` 还没有完全对齐。

主要差距：

- 缺少统一的 `sourceType`、`confidence`、`verificationStatus`
- 发送状态语义还偏旧
- 平台字段仍然直接进入核心结构
- 事件模型不完整

### 3.2 Core / UI / Data 是否解耦

不够解耦。

从现状看，UI、Core、Data 之间存在直接调用和共享类型的情况。`MessageRouter` 和 `ConversationManager` 已经在做部分编排，但还没有形成清晰的单向依赖。

问题表现：

- UI 里有较多业务流程控制
- Core 直接落库的情况较多
- Data 层仍以旧表结构为中心

### 3.3 平台适配器是否边界清晰

不够清晰。

`src/services/platforms` 里已经有 `IPlatformAdapter` 和多个具体平台适配器，但这些适配器目前更像“直接操作业务链路的实现”，不是纯边界适配器。它们仍与数据库、RPA、消息状态流转耦合较深。

### 3.4 Python 脚本是否被过度耦合

是。

`python/rpa` 现在承担了采集、发送、调试、校准、读写、数据库交互等多种角色，而且主程序 UI 里还直接提供 RPA 管理和控制台。这个结构能跑，但不适合作为稳定的 MVP 边界。

### 3.5 AI 服务是否独立

部分独立，部分耦合。

`src/services/ai` 已经形成了较完整的 AI 调用层，这是保留项。但它仍与业务层、UI 和历史聊天工作台逻辑混杂，需要收敛成明确的 `ipc` 或 `ai` 服务边界。

### 3.6 RPA / 自动化是否被正确隔离

没有完全隔离。

当前 RPA 逻辑同时存在于：

- `src/ui/rpaprocesscontroller.cpp`
- `src/ui/rpamanagedialog.cpp`
- `src/ui/rpa_console_window.cpp`
- `src/services/wechat/wechatworkbenchservice.cpp`
- `python/rpa/*`
- `database/app.db`

这说明 RPA 既是运行时能力，也是 UI 控制项，还在共享数据库。对于 MVP 来说，这种耦合太强。

### 3.7 是否存在重复逻辑或历史遗留分支

存在，而且比较明显。

典型表现：

- 微信 / 千牛 / 拼多多有各自 RPA 路线
- `poc/` 里还有历史 PoC 版本
- `docs-old` 里保留了一整批旧方案
- `src/ui` 和 `src/services` 中有多套“工作台”概念并存

这不是问题本身，但需要明确哪些是历史实现，哪些是当前主线。

## 5. 推荐的新工程边界

### 5.1 `models`

建议把 `src/core/types.h` 里的基础类型升级为 `models` 边界，内容包括：

- `Conversation`
- `Message`
- `PlatformAccount`
- `ConversationEvent`
- `SendMessageCommand`
- `AISuggestion`
- 枚举和来源可信度字段

现有迁移方式：

- 先保留旧 `types.h`
- 再逐步拆出 `src/models/*`
- 核心代码优先切到新模型

### 5.2 `core`

建议保留并收敛：

- 会话管理
- 消息分发
- 去重
- 状态流转
- AI 请求编排
- 平台状态协调

现有映射：

- `ConversationManager` 保留，但职责收窄
- `MessageRouter` 改成事件归一与分发中心
- `PlatformBootstrap` 改成平台装配器，而不是业务启动器

### 5.3 `platform`

建议把 `src/services/platforms` 迁移为真正的平台适配层。

适配器只负责：

- 识别平台状态
- 采集可见内容
- 上报事件
- 准备草稿或人工确认发送所需信息

不负责：

- 直接操作 UI
- 直接落库
- 直接处理 AI 业务逻辑

### 5.4 `ipc`

当前仓库里没有明显独立 `ipc` 层，建议把它从现有 `services/ai` 和 `wechatworkbenchservice` 中抽出来。

职责：

- C++ 与 Python 通信
- AI 请求与响应
- 超时、错误和请求 ID

### 5.5 `storage`

保留 `src/data`，但把它定位成 `storage` 边界。

职责：

- SQLite 持久化
- 草稿和最近会话
- 账号状态
- 适配器状态

### 5.6 `ui`

保留当前 Qt UI，但要把 RPA 操作和平台接入动作从 UI 中抽出去。

UI 只做：

- 展示
- 交互
- 提交操作意图

### 5.7 Python 服务边界

建议把 `python/rpa` 分成两类：

1. `python/service` 类能力
   - AI 辅助
   - 策略判断
   - 请求组装
   - 日志

2. `python/rpa` 类能力
   - 采集
   - 发送前准备
   - 可见内容识别
   - 平台校准

两者不应该再被 UI 直接揉在一起。

### 5.8 RPA / 自动化边界

RPA 应定位为平台适配器或独立辅助进程，不是主 UI 的控制逻辑。

原则：

- 主程序只发意图
- RPA 进程负责执行
- 结果通过统一事件回传
- 失败默认降级

## 10. 风险与取舍

### 10.1 最容易踩坑的地方

1. 把 UI 当业务层
2. 把 Python 当主业务中心
3. 保留两套模型并行太久
4. 真实平台 PoC 过早进入主流程
5. 继续把 RPA 写进主窗口逻辑

### 10.2 取舍建议

- 先保主流程，再清历史包袱
- 先统一模型，再改 UI
- 先模拟，再真实
- 先人工确认，再考虑自动化增强
- 先单平台，再多平台
