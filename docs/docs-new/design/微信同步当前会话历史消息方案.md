# 微信同步当前会话历史消息方案

## 1. 背景

当前微信平台监听链路已经可以读取当前可见微信会话中的消息，并通过 Python sidecar 事件写入聚合会话列表。现有能力偏实时监听和可见窗口采样：

- C++ 聚合界面负责展示会话、消息、按钮入口和用户操作。
- `WechatRPAAdapter` 通过命令 WebSocket 向 Python 服务发送平台命令，并通过事件桥接收 `conversation_observed`、`message_observed`、`message_sent` 等事件。
- Python `WechatSidecarAdapter` 已提供 `fetch_visible_messages`，但它只读取当前聊天区可见或当前控件树可采样的消息。
- `WechatVisibleMessageReader.read_visible_messages()` 的 `tail_only` 只是从采样结果尾部截断，并不执行聊天区向上滚动。
- 微信多媒体消息识别和提取已经接入 `_message_event()`，会根据控件类型构建 `content_type`、`content_image_path`、`evidence_ref` 和相关 metadata。

因此，如果需要用户主动补采当前微信会话最近一段历史，不能只在现有 `fetch_visible_messages` 上加一个更大的 `limit`，而应新增一条明确的“历史同步”能力。

## 2. 产品定义

功能名称建议为：

> 同步当前会话最近历史

命名意图：

- “当前会话”明确作用域是聚合界面当前选中的微信会话。
- “最近历史”明确不是读取微信数据库全量历史，而是从当前微信聊天区通过 UI 滚动补采最近一段历史。
- “同步”强调结果会进入本地业务库或服务事实库，随后在聚合界面展示。

建议 UI 入口：

- 在聚合聊天头部 `m_chatHeaderBar` 右侧添加图标按钮。
- 图标使用 `resources/aggregate_reception_icons/sync_hitstory_messages_icon.svg`。
- 后续实现时需要把该 SVG 加入 `resources/app_resources.qrc`，才能通过 `:/aggregate_reception_icons/sync_hitstory_messages_icon.svg` 加载。
- 按钮只在当前会话平台为 `wechat` 时显示或启用。
- Python 服务不可用、当前无会话、正在执行历史同步时禁用。
- 不要求用户勾选“微信”并点击“开始监听”。历史同步是手动触发的一次性平台命令，只依赖 Python 服务已启动并可连接，以及本机微信窗口可操作。
- 首版暂不考虑“微信监听正在运行时同步历史”的并发场景。因为历史同步会切换并滚动微信聊天区，若 observer 同时扫描未读，容易互相干扰；后续再补任务锁、暂停 observer 或提示先停止监听。

点击后弹出对话框：

- 标题：`同步当前会话最近历史`
- 数量输入：默认 `20`，建议范围 `1-50`
- 可选提示：同步期间会切换并滚动微信当前聊天区，完成后尽量恢复到底部
- 确认按钮：开始同步
- 取消按钮：关闭；同步中则尝试取消后关闭

同步完成反馈：

- 新增消息数
- 跳过重复消息数
- 多媒体提取成功数和失败数
- 是否已恢复到底部

历史同步应新增独立 handler，避免污染实时监听路径。

## 4. 历史同步语义

历史同步不是实时新消息。它应满足：

- 不增加未读数。
- 不触发自动回复。
- 不覆盖会话最后消息。
- 不弹出普通“新消息”提示。
- 不改变当前会话的处理状态。
- 可以补充聚合消息列表中的历史气泡。
- 可以进入微信扩展表，保留观测证据和媒体提取结果。

建议事件层新增语义：

```text
history_message_observed
history_sync_progress
history_sync_completed
history_sync_failed
history_sync_cancelled
```

如果短期不新增事件类型，也至少要在普通 `message_observed` payload 中增加明确 metadata：

```json
{
  "source_type": "ui_observed",
  "metadata": {
    "history_sync": true,
    "history_sync_request_id": "...",
    "suppress_unread": true,
    "suppress_auto_reply": true,
    "preserve_conversation_last_message": true,
    "observation_method": "uia_history_scroll"
  }
}
```

更推荐新增专门事件类型，原因是当前 `MessageRouter` 对普通 `message_observed` 有实时消息副作用：

- `ConversationDao::updateLastMessage(...)`
- 入站消息 `incrementUnread(...)`
- UI 当前会话收到入站后触发自动回复判定

历史导入如果复用普通事件，需要 C++ 多处按 metadata 分支，风险更高。

## 5. Python 采集流程

建议新增方法：

```python
WechatVisibleMessageReader.read_recent_history_messages(
    display_name: str,
    limit: int,
    max_scrolls: int,
    settle_ms: int,
) -> HistoryMessageBatch
```

基本流程：

1. 找到微信主窗口。
2. 确认或切换到目标会话。
3. 找到聊天消息列表控件。
4. 从当前底部或当前可见位置采样一次。
5. 向上滚动聊天区。
6. 等待控件树稳定。
7. 再次采样。
8. 将新样本按批次去重并合并到结果集。
9. 达到用户请求数量、到达顶部、连续多次没有新增样本、用户取消或超时后停止。
10. 尽量滚回底部。
11. 构建历史消息事件并写入 store。

停止条件建议：

- 已收集去重后消息数 >= `limit`
- `scroll_count >= max_scrolls`
- 连续 `2-3` 次滚动后没有新增消息
- 聊天区已到顶部或样本首条持续不变
- 总耗时超过上限，例如 `15-30s`
- 收到取消信号

默认参数建议：

| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| `limit` | `20` | 用户默认同步数量 |
| `max_limit` | `50` | UI 和 Python 双侧都限制 |
| `max_scrolls` | `8-12` | 防止长时间操作微信窗口 |
| `settle_ms` | `300-800` | 等待 UIA 控件树更新 |
| `timeout_ms` | `30000` | 单次同步最大时长 |

## 6. 去重策略

历史同步不建议把 `rect` 作为核心去重字段。

原因：

- 同一条微信消息随聊天区滚动后屏幕坐标会变化。
- `visible_sequence_index` 也会随每次采样窗口变化。
- 把 `rect` 或可见序号写进核心 ID，会把同一条历史消息识别成多条。

建议分两层去重。

### 6.1 批次内去重

在一次历史同步任务中，先维护内存级去重 key：

```text
conversation_key
direction
sender_role
content_type
normalized_content
stable_media_hint
nearby_context
```

说明：

- `normalized_content` 去除多余空白和换行差异。
- `stable_media_hint` 可使用媒体类型、控件 name、文件名、可复制路径或 evidence 摘要。
- `nearby_context` 用前后各一条消息的短摘要帮助区分重复短文本，例如“好的”“收到”。
- 不使用 `rect`。
- 不使用 `visible_sequence_index` 作为稳定 ID 的核心组成部分。

### 6.2 入库去重

当前 `platform_msg_id` 由 Python `_message_id_parts()` 和 `_sha1()` 生成。实时可见采样场景中，这个策略基本可用；历史滚动场景中建议新增历史专用 ID 生成策略。

建议格式：

```text
wechat_history_<sha1(account_id|conversation_key|direction|content_type|normalized_content|nearby_context)>
```

短文本重复风险处理：

- 如果同一批次出现多个完全相同 key，不应全部合并掉。
- 可以追加同类重复计数，例如 `duplicate_ordinal`。
- `duplicate_ordinal` 应由合并后的逻辑顺序生成，而不是屏幕坐标生成。

媒体消息重复风险处理：

- 优先使用可稳定提取的文件名、文件大小、复制结果路径、证据截图 hash。
- 如果只有 `[图片]`、`[文件]` 这类占位内容，可结合前后文本上下文和批次内顺序区分。

## 7. 多媒体消息处理

同步历史时必须考虑多媒体消息，因为当前微信链路已经支持多媒体消息获取。

现有 `_message_event()` 已经会：

- 根据 `sample.kind` 得到 `content_type`
- 调用 `WechatMediaExtractor.extract(...)`
- 写入 `content_image_path`
- 写入 `evidence_ref`
- 写入媒体提取相关 metadata

历史同步不应另写一套媒体 payload 构造逻辑。建议：

- 复用 `_message_event()` 的 payload 构造能力。
- 或抽出公共 builder，让实时消息和历史消息共用同一套媒体提取、证据写入和 metadata 结构。

首版多媒体策略：

| 类型 | 首版处理 |
| --- | --- |
| 文本 | 完整同步 |
| 图片 | 同步 `content_type=image` 和占位内容；能提取则写 `content_image_path/evidence_ref` |
| 表情 | 同步 `content_type=emoji`；能留证据则写 evidence |
| 文件 | 同步 `content_type=file` 和文件名/占位；能复制则写路径，失败不阻断 |
| 视频 | 同步 `content_type=video` 和占位；能留证据则写 evidence |

媒体提取失败不应导致整个同步失败。建议 metadata 标记：

```json
{
  "media_extract_status": "success|failed|skipped",
  "media_extract_error": "...",
  "history_sync": true
}
```

UI 上可以允许显示“媒体占位已同步，原文件未提取”。

## 8. C++ 入库与 UI 副作用控制

### 8.1 不增加未读

历史消息不能进入普通入站未读累加路径。实现方式：

- 新增 `MessageRouter::handleHistoryMessageObserved(...)`
- 或在普通入站处理前识别 `metadata.history_sync == true`

如果使用 metadata 分支，应跳过：

```cpp
if (msg.direction == "in")
    convDao.incrementUnread(convId);
```

### 8.2 不覆盖最后消息

历史消息通常早于当前最后消息，不能无条件执行：

```cpp
convDao.updateLastMessage(convId, msg.content, now);
```

建议：

- 历史消息导入默认不更新 last message。
- 如果历史同步发现当前会话没有任何消息，可允许使用本批次最新一条填充 last message。
- 如果 payload 未来能带真实平台时间，可只在消息时间晚于当前 lastMessageAt 时更新。

### 8.3 不触发自动回复

当前 `AggregateChatForm::onNewMessage()` 对当前会话入站消息会调用 `tryAggregateAutoReply(..., "T2")`。

历史消息导入应：

- 不走 `onNewMessage()` 的实时提示路径；或
- 在 `onNewMessage()` 中识别历史 metadata/sourceType 后跳过自动回复。

更推荐让历史导入完成后触发一次会话消息刷新，而不是逐条伪装实时新消息刷新 UI。

### 8.4 消息排序

当前 `MessageDao::listByConversation()` 按 `created_at ASC` 展示。历史同步如果使用导入时间作为 `created_at`，可能导致历史消息排到最新位置。

需要明确历史消息时间字段：

- 如果微信 UI 能读到原始时间，写入 `original_timestamp` 并尽量解析成 `platformDisplayedAt/createdAt`。
- 如果不能读到时间，至少在本批次中按采集逻辑顺序分配稳定时间，保证历史消息在 UI 中相对顺序正确。
- 不应让向上滚动采集到的旧消息排在当前最新消息之后。

这部分可能需要 DAO 或模型层支持“观测时间”和“平台显示时间”的展示排序策略。

## 10. 可实施步骤

### P0：文档和协议确认

目标：先把语义定清楚，避免后续实现时把历史导入混入实时消息。

任务：

1. 确认功能名称为“同步当前会话最近历史”。
2. 确认 UI 数量范围为 `1-50`，默认 `20`。
3. 确认历史同步不增加未读、不触发自动回复、不覆盖会话最后消息。
4. 确认历史同步支持文本和多媒体占位，媒体提取失败不阻断整批同步。
5. 确认是否新增专门事件类型 `history_message_observed`，推荐新增。
6. 确认历史同步不依赖“微信开始监听”，只要求 Python 服务可连接。
7. 确认首版暂不支持监听运行中同步历史；若检测到监听运行，可先提示用户停止监听，或后续补任务锁。

验收：

- 方案文档完成。
- 协议字段和副作用规则已明确。

### P1：Python 历史同步能力

目标：Python 能在指定微信会话内向上滚动并采集最近 N 条历史。

任务：

1. 在 `WechatSidecarAdapter.command()` 注册 `sync_history_messages`。
2. 在 reader 层新增滚动历史采样方法。
3. 复用现有可见消息采样、角色判断和媒体提取逻辑。
4. 新增历史同步批次内去重，不使用 `rect` 作为核心 key。
5. 为历史消息生成历史专用 `platform_msg_id`。
6. 输出进度、完成、失败事件。
7. 同步结束后尽量滚回聊天底部。

验收：

- 对当前微信会话同步 20 条文本历史，结果顺序正确。
- 滚动重叠区域不会重复生成多条相同消息。
- 图片、文件、视频、表情至少以正确 `content_type` 和占位进入 payload。
- 媒体提取失败时同步仍完成，metadata 记录失败原因。

### P2：C++ 命令入口和 UI

目标：聚合界面当前微信会话可发起同步。

任务：

1. 将 `sync_hitstory_messages_icon.svg` 加入 `resources/app_resources.qrc`。
2. 在 `AggregateChatForm` 头部右侧新增 `QToolButton`。
3. 当前会话不是微信时隐藏或禁用按钮。
4. 点击按钮弹出数量输入对话框。
5. 确认后通过 `WechatRPAAdapter` 或 `IpcService` 发送 `sync_history_messages`。
6. 同步中禁用按钮，展示状态文案。
7. 完成后刷新当前会话消息列表。

验收：

- 微信会话显示按钮，非微信会话不显示或禁用。
- 输入 `20` 后能发出命令。
- 超过 `50` 的输入被 UI 阻止。
- Python 服务不可用时给出明确提示。

### P3：历史消息入库语义

目标：历史消息进入库和 UI，但不产生实时消息副作用。

任务：

1. C++ 识别 `history_message_observed` 或 `metadata.history_sync == true`。
2. 导入消息时跳过未读累加。
3. 导入消息时默认不覆盖会话 last message。
4. 导入消息时不触发自动回复。
5. 确保 `wechat_messages` 扩展表仍写入原始 payload 和媒体字段。
6. 根据平台显示时间或批次顺序保证 UI 排序正确。

验收：

- 同步历史后，当前会话未读数不增加。
- 当前会话最后消息不被旧消息覆盖。
- AI 自动回复模式开启时，同步历史不会自动发起回复。
- 消息列表能显示新导入的历史消息。

### P4：取消、并发和恢复

目标：提升真实使用稳定性。该阶段不属于首版必要范围。

任务：

1. 支持取消正在进行的同步任务。
2. 同一时间限制每个微信账号只执行一个历史同步任务。
3. 与发送消息、未读扫描、实时 observer 互斥或排队。
4. 失败时记录诊断 artifact。
5. 同步后尽量恢复到底部；恢复失败时提示用户。

验收：

- 同步中再次点击按钮不会启动第二个任务。
- 同步中发送消息有明确阻止或排队策略。
- 取消后 Python 停止滚动，C++ 状态恢复。
- 失败日志包含 request_id、display_name、scroll_count 和错误原因。

### P5：真实微信验收

目标：在真实微信客户端中验证可靠性。

验收场景：

1. 当前会话只有文本，要求同步 20 条。
2. 当前会话包含连续重复短文本，例如多条“好的”。
3. 当前会话包含图片、表情、文件、视频。
4. 当前会话历史不足 20 条。
5. 当前会话已在中间位置，不在底部。
6. 微信窗口被遮挡或最小化。
7. 同步过程中用户手动切换微信会话。
8. 自动回复模式开启时同步历史。
9. Python 服务断开时点击按钮。
10. 同步完成后重启客户端，历史消息不重复。

## 11. 风险和取舍

### 11.1 UIA 滚动稳定性

微信聊天区是 `mmui::RecyclerListView`，控件复用和懒加载会导致滚动后控件树变化。历史同步要接受“最近 N 条尽力同步”的产品边界，不能承诺和微信数据库完全一致。

### 11.2 消息时间不完整

微信 UI 不一定每条消息都有可读时间。没有真实平台时间时，只能保证批次内相对顺序。后续如需严格历史排序，需要继续增强时间识别。

### 11.3 重复短文本

短文本重复是历史同步去重最大风险。完全相同内容既可能是同一条重复采样，也可能是真实多条消息。首版可通过 nearby context 和 duplicate ordinal 降低误判，但无法做到 100% 精确。

### 11.4 多媒体提取成本

媒体复制、截图证据和上下文菜单操作可能明显增加耗时，并更依赖前台焦点。首版应允许媒体占位先同步，提取失败不阻断整个任务。

### 11.5 与实时 observer 并发

历史同步会滚动聊天区，实时 observer 同时扫描未读可能互相干扰。首版暂不考虑监听运行中同步历史的场景，也不要求用户先开启监听。

后续如果要支持“监听运行中仍可同步历史”，建议 Python 侧增加平台自动化任务锁，同账号同窗口串行执行；或者在历史同步开始前暂停 observer，完成后再恢复。

## 12. 推荐首版范围

首版建议只做：

- 当前微信会话。
- 用户手动触发。
- Python 服务启动并可连接即可，不要求勾选微信并点击“开始监听”。
- 数量 `1-50`。
- 文本完整同步。
- 多媒体识别和占位同步，尽力提取证据。
- 历史消息不增加未读、不触发自动回复、不覆盖 last message。
- 同步完成后刷新当前会话消息列表。
- 暂不支持监听运行中同步历史；该场景留到任务锁/observer 暂停恢复阶段。

暂不做：

- 全会话批量历史同步。
- 后台无焦点完全静默滚动承诺。
- 微信数据库级全量读取。
- 严格平台时间排序。
- 跨设备历史补齐。
- 监听运行中同步历史的并发保障。

## 13. 当前进度

截至 2026-06-16，本方案已确认以下设计决策：

1. 功能名称采用“同步当前会话最近历史”。
2. 同步入口放在聚合聊天头部当前会话标题右侧，仅微信会话可用。
3. 同步数量默认 `20`，最大 `50`。
4. 历史同步是用户主动触发的一次性命令，不依赖“微信开始监听”。
5. 首版只要求 Python 服务已启动并可连接，微信客户端已打开、已登录且 UIA 可操作。
6. 首版暂不考虑监听运行中同步历史；后续通过任务锁、暂停 observer 或 UI 提示补齐。
7. 去重核心不使用 `rect`，避免滚动后同一消息因坐标变化被识别成重复新消息。
8. 多媒体消息同步需要复用现有媒体识别和提取链路，提取失败时降级为占位同步。
9. 历史导入不增加未读、不触发自动回复、不覆盖会话最后消息。

## 14. 结论

“同步当前会话最近历史”适合作为微信聚合接待的手动补采能力。它应被设计为独立的历史导入流程，而不是扩大现有 `fetch_visible_messages` 的读取范围。

关键实现原则：

- Python 负责切会话、滚动、采样、媒体提取和批次去重。
- C++ 负责入口、命令、进度展示和导入后的 UI 刷新。
- 历史消息必须有独立语义，不能冒充实时新消息。
- 去重不要依赖 `rect` 和可见序号。
- 多媒体复用现有提取链路，失败可降级为占位同步。

按上述边界实施，可以在不破坏当前实时监听和自动回复链路的前提下，为客服提供一个可控、可解释的历史补采入口。
