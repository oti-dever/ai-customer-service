# C++客户端与Python服务端解耦架构方案

## 1. 目标

以微信平台举例，其他平台同理。

把当前微信链路拆成两个可独立运行、可独立调试的部分：

- `C++` 只负责桌面客户端、聚合界面、统一会话模型、历史数据展示、客户端缓存和用户操作入口。
- `Python` 只负责微信自动化、消息观察、会话切换、消息发送和本地持久化。

核心原则是：

- 客户端和服务端可以分别启动、分别调试。
- 客户端连接 Python 服务，不等于自动监听所有平台。
- 只有用户选择平台并发出 `connect` 后，Python 才启动对应平台 observer。
- 只有平台进入监听状态后，Python 才向 C++ 推送该平台事件。
- 未监听的平台不扫描、不切换窗口、不推送事件。
- 客户端关闭后，Python 不再推送；Python 停止后，也不再产生新的自动化数据。
- 客户端不保存全量会话作为长期真相，只保存缓存和游标；数据库是服务端内部的唯一真相源。

## 2. 角色边界

### 2.1 C++ 客户端

- 负责 UI、聚合界面、会话列表、消息列表、发送按钮和状态显示。
- 负责保存客户端侧缓存和连接状态，例如当前选中会话、当前连接地址、消息游标和历史同步进度。
- 负责通过 Python 服务端同步历史消息，并把事件流增量追加到界面。
- 负责发起命令请求，例如发送消息、连接服务端、健康检查、历史同步。
- 负责展示可用平台列表，让用户选择要监听的平台。
- 负责按用户选择对单个平台或多个平台发送 `connect` / `disconnect`。

### 2.2 Python 服务端

- 负责微信窗口发现、会话扫描、未读检测、消息读取、会话切换和消息发送。
- 负责把采集结果写入本地数据库。
- 负责对客户端提供历史同步接口，客户端不直接访问数据库。
- 负责在具体平台进入监听状态后，把该平台增量事件推送给 C++。
- 负责区分运行模式，控制是否允许发送。
- 负责维护平台注册、平台健康状态和平台 observer 生命周期。
- 未收到平台 `connect` 前，只保持服务与 bridge 可用，不主动扫描平台窗口。

## 3. 运行模式

### 3.1 正式模式

适合真实业务运行。

- 允许读取微信未读会话、读取消息、切换会话。
- 允许把消息和会话写入数据库。
- 不允许发送消息。
- 不允许把数据推送给客户端，除非客户端已建立连接并显式订阅。

### 3.2 调试模式

适合开发者验证自动化链路。

- 允许读微信。
- 允许执行开发者主动命令。
- 允许发送消息。
- 仍然遵守连接状态：未连接时只写数据库，不推送客户端。

## 4. 连接模型

建议把系统理解成“客户端 + 服务端”的双进程结构，二者通过双向 WebSocket 通信，消息格式统一使用 JSON。服务端可以和客户端运行在同一台机器，也可以后续扩展为远程服务。

### 4.1 连接建立

客户端启动后，用户填写服务端 IP 和端口，建立连接。

连接建立后：

- Python 服务端保持 bridge、HTTP、WebSocket 和平台注册能力可用。
- 客户端可以查询可用平台、平台健康状态和历史同步能力。
- 客户端不默认对微信、千牛等平台发 `connect`。
- Python 不因为客户端刚连接就自动扫描平台窗口。
- 客户端可以通过 Python 服务端按游标同步已存在的历史和增量。
- 客户端可以把界面状态和服务端历史对齐。
- 客户端若断开重连，只需重新请求服务端同步并恢复缓存，不需要依赖 Python 重新补全所有会话。

### 4.2 平台监听选择

平台监听是服务连接之后的显式动作。

推荐流程：

1. 客户端启动后只连接 Python 服务。
2. Python 服务只保持 bridge、HTTP、WebSocket 和平台注册能力可用，不主动监听平台。
3. UI 通过平台状态接口查询可用平台列表，例如微信、千牛。
4. 用户勾选一个或多个平台，点击“开始监听”。
5. C++ 对选中的平台发送 `connect`。
6. Python 对对应平台启动 observer，并通过事件桥推送消息。
7. 用户可单独停止某个平台监听，C++ 对该平台发送 `disconnect`。

这几个状态必须分开理解：

- Python 服务在线：只表示服务进程可连接。
- 平台已注册：只表示服务端知道这个平台 adapter。
- 平台已监听：表示该平台已经收到 `connect`，observer 正在运行。

当前接口建议：

- `GET /api/platforms`：返回平台注册列表、展示名、是否 registered、是否 listening、observer 是否运行。
- `GET /api/platform/health?platform=wechat|qianniu`：返回单个平台健康状态，可用于更细的诊断。
- `POST /api/platform/command` + `command=connect|disconnect`：显式启动或停止单个平台监听。

### 4.3 连接断开

连接断开后：

- Python 停止向客户端推送。
- 已监听的平台是否继续 observer 由后续产品策略决定；默认建议客户端断开后停止推送，但不把“客户端连接”与“服务进程存在”混在一起。
- 客户端只保留本地缓存和已有数据，不再期待实时事件。

### 4.4 历史同步

客户端如果需要历史消息，不应该依赖实时推送补齐，而应该：

- 通过 Python 服务端同步会话和消息。
- 再用事件流补齐连接之后的新变化。

这意味着数据库是服务端内部的唯一真相源，事件流只是连接后的增量分发。客户端不直接读写数据库，数据库结构和存储类型属于服务端实现细节。

## 5. 数据流

### 5.1 Python -> 数据库

Python 检测到：

- 未读会话
- 可见消息
- 发送结果
- 健康状态

先写入数据库，形成可审计、可恢复的数据底座。

### 5.2 Python -> C++ 事件推送

只有平台进入监听状态后，Python 才把该平台事件推送给 C++。
通信载体建议统一为 WebSocket，消息体统一用 JSON。

典型事件包括：

- `conversation_observed`
- `message_observed`
- `message_sent`
- `send_failed`
- `account_health_changed`

其中 `message_observed` 由 Python 做平台侧最终判定，至少应输出：

- `sender_role`: `customer` 表示客户消息，`agent` 表示我方客服消息，`system` 表示系统消息。
- `direction`: `inbound` / `outbound` / `system`，用于客户端内部未读、发送状态、AI 上下文等统一逻辑。
- C++ 不再按平台样式猜测消息左右，只对上述协议字段做校验和极少量旧值兼容。

### 5.3 C++ -> Python 命令

客户端主动发请求时，Python 执行命令并返回结果。

典型命令包括：

- `connect`
- `disconnect`
- `fetch_conversations`
- `fetch_visible_messages`
- `sync_history`
- `send_message`
- `health_check`

## 6. 建议的状态机

状态机不建议把服务生命周期、运行模式和客户端连接混在一个枚举里。三者应该保持独立，组合后表达当前运行状态。

### 6.1 Python 服务状态

- `stopped`
- `starting`
- `running`
- `error`

### 6.2 Python 运行模式

- `formal`
- `debug`

### 6.3 客户端连接状态

- `no_client`
- `client_connecting`
- `client_connected`
- `client_stale`

### 6.4 平台监听状态

每个平台独立维护监听状态：

- `registered`：平台 adapter 已注册，但未监听。
- `connecting`：客户端已发 `connect`，等待平台 observer 启动。
- `listening`：平台 observer 正在运行。
- `stopping`：客户端已发 `disconnect`，等待停止完成。
- `stopped`：平台未监听。
- `error`：平台 observer 或健康检查异常。

### 6.5 数据处理状态

- 未监听：不主动扫描平台窗口，不推送平台事件。
- 已监听：写库 + 推送该平台事件。
- 重连：客户端先通过服务端同步历史游标，再继续订阅增量。
- 客户端是否连接、平台是否监听、Python 服务是否运行，是三类独立状态。

## 7. 这套方案的价值

- C++ 和 Python 可以分别启动，排查边界更清楚。
- 微信上下文和自动化逻辑都收敛在 Python，客户端不再依赖平台细节。
- 历史消息有服务端数据库兜底，实时消息有事件流兜底。
- 断线、重启、关闭都更容易恢复。
- 正式模式和调试模式的权限边界清晰，便于控制风险。
- 客户端不需要长期持有全部会话对象，只保留当前展示和恢复所需缓存。
- 用户可以只启动一个平台监听，便于联调、降低窗口焦点干扰，也更符合真实业务授权习惯。

## 8. 风险点

- 需要明确“历史同步”和“事件推送”的责任边界，避免重复显示或重复处理。
- 需要为事件增加游标和幂等键，避免重连后重复显示。
- 需要定义连接断开后的客户端提示和恢复策略。
- 需要把正式模式的“只读”约束落实到发送链路上。
- 需要保证客户端缓存可由服务端历史同步随时重建，不能把缓存当成唯一状态源。
- 需要避免“服务在线”被误解成“所有平台已监听”。
- 需要定义多平台同时监听时的 UI 状态、错误提示和单平台停止策略。

## 9. 结论

建议把当前微信链路继续收敛成：

- `C++ = 客户端 + 聚合展示 + 本地缓存 + 命令入口`
- `Python = 微信自动化服务 + 数据采集 + 事件源 + 本地写库`

未建立连接时，Python 仍能独立运行并落库；建立连接后，才对客户端实时推送。

进一步收口后，客户端连接 Python 服务只代表控制通道可用；具体平台是否监听，必须由用户在 UI 中显式选择并确认。

## 10. 待确认决策点

1. 客户端重启后，历史同步是“全量重建当前缓存”还是“按游标增量恢复”？
   - 这里问的是客户端重新打开时，缓存怎么恢复。
   
     * “全量重建当前缓存”就是客户端启动后重新向 Python 服务端请求会话列表和最近消息，把内存缓存重新搭起来。优点是简单、稳，不怕之前客户端状态丢了；缺点是数据很多时启动会慢一点。
   
     * “按游标增量恢复”就是客户端记住上次同步到哪里，重启后通过服务端只拉取之后新增的数据。优点是快；缺点是更复杂，一旦游标错了，就可能漏消息或状态不一致。
   
   - 先选“全量重建当前缓存”，后面数据量大了再做增量优化。
   
2. WebSocket 协议要不要固定成 `event / command / response` 三类信封？
   - 这里问的是双方传 JSON 时，要不要统一外壳格式。
   
     比如：
   
     ```json
     {
       "type": "event",
       "id": "evt_001",
       "payload": {}
     }
     ```
   
     如果不固定格式，早期也能跑，但后面消息类型多了，排查会乱。固定成三类后，含义很清楚：
   
     - event：Python 主动告诉客户端“我看到新消息了”
     - command：客户端主动告诉 Python“请发送这条消息”
     - response：Python 回答“刚才那个命令成功/失败”
   
   - 先固定省事。
   
3. 正式模式里 `send_message` 是“完全禁用”还是“只禁客户端发起，保留本机开发命令入口”？
   - 这里问的是正式运行时，发送动作到底允许到什么程度。
   
     * “完全禁用”就是正式模式下无论谁调用 send_message 都拒绝。优点是安全；缺点是线上排查时不方便临时验证发送链路。
   
     * “只禁客户端发起，保留本机开发命令入口”就是客户端 UI 不能发，但开发者在本机调试命令里还能发。优点是调试方便；缺点是权限边界更复杂，容易误用。
   
   - 采用正式模式完全禁用发送，调试模式才允许发送。这样边界最清楚。
