# 外部工具辅助平台接入方案

本文合并并收口原 `不局限纯技术从零开发.md` 与 `保留自研C++聊天界面.md` 的观点，用于指导 M5 单平台真实 PoC 及后续多平台接入。

核心结论：

- 保留自研 C++ 聚合工作台作为客服主界面。
- 不把平台接入理解为“从零自研所有 UI 自动化能力”。
- Python sidecar 作为外部工具适配与 AI 编排中心。
- 外部工具只负责可见内容采集、环境隔离、草稿回填和健康检测。
- AI 只生成建议或草稿，最终发送必须由客服人工确认。
- 当前只保留两类外部工具路线：RPA 和指纹浏览器。
- 当前不考虑商业客服 SaaS 旁路、硬件 HID、投屏群控路线。

相关主文档见《[多平台会话聚合_架构分析文档.md](多平台会话聚合_架构分析文档.md)》。

---

## 1. 为什么引入外部工具

微信 PC、千牛 PC、拼多多商家后台等目标平台在当前阶段无法获得稳定可用的官方客服 API / SDK。若完全自研平台适配层，团队需要长期维护：

- Web 页面 DOM 选择器
- Windows 控件树和无障碍接口
- OCR 与截图识别
- 多账号浏览器环境
- 登录态和弹窗异常识别
- 输入框定位与草稿回填

这些工作维护成本高、受平台改版影响大，也容易把 C++ 客户端拖入大量与业务无关的脏活。

因此，项目采用“自研核心 + 外部工具辅助”的路线：

```text
C++ 自研工作台
  负责：会话列表、消息展示、AI 建议、人工确认、状态与审计

Python sidecar / adapter hub
  负责：外部工具协议、AI 编排、适配器调度、日志与降级

外部工具
  负责：真实平台消息采集、页面/窗口定位、草稿回填
```

这样做不是放弃自研，而是把自研能力集中在真正形成产品差异的部分：统一会话模型、客服工作流、AI 建议、本地缓存、状态机和审计。

---

## 2. 保留自研 C++ 工作台的边界

C++ 客户端仍然是最终客服工作台，而不是外部工具的壳。

它应继续负责：

- 聚合会话列表和消息区渲染
- 会话状态、消息状态和失败降级展示
- AI 建议入口和草稿编辑
- 人工确认发送入口
- 本地 SQLite 缓存和最近会话恢复
- 操作提示、风险提示和审计可视化

C++ 客户端不应直接承担：

- 具体平台 DOM 选择器维护
- 千牛 / 微信控件树查找细节
- 指纹浏览器 Profile 生命周期细节
- RPA 脚本执行细节
- 平台页面异常分支的脚本编排

这些能力应下沉到 Python sidecar 和外部工具适配器。

---

## 3. 当前保留的外部工具路线

### 3.1 Web 平台：AdsPower / 紫鸟 / 指纹浏览器

适用平台：

- 拼多多商家后台
- 抖店、视频号小店等 Web 工作台
- 其他以浏览器页面承载客服流程的平台

推荐职责：

- 管理店铺账号对应的独立浏览器 Profile
- 隔离 Cookie、本地存储和登录态
- 通过 Local API 启动、停止或唤起指定浏览器环境
- 暴露 CDP 调试端口给 Python 适配器
- 配合 Playwright / Puppeteer 做可见 DOM 采集
- 将 AI 建议回填到页面输入框，等待人工确认

推荐链路：

```text
指纹浏览器 Profile
  -> CDP / Playwright
  -> Python WebPlatformAdapter
  -> 统一 Message / Conversation
  -> C++ 聚合工作台
```

工程约束：

- 不绕过登录、验证码、权限和平台访问控制。
- 不将网络抓包或未授权接口作为默认主链路。
- DOM 采集只读取客服正常可见内容。
- 页面选择器必须有版本标记和健康检测。
- 回填动作默认只填入草稿，不自动点击发送。

### 3.2 桌面客户端：影刀 / UiBot RPA

适用平台：

- 千牛 PC 客户端
- 微信 PC 客户端
- 其他以 Windows 桌面客户端承载客服流程的平台

推荐职责：

- 识别客户端窗口、登录态和当前会话
- 读取可见会话列表和消息内容
- 监听或轮询新消息提示
- 定位输入框并回填草稿
- 执行低频、人工触发的辅助流程
- 将结果通过本地 HTTP / gRPC / SQLite inbox 投递给 Python 或 C++

推荐链路：

```text
影刀 / UiBot 脚本
  -> 本地 HTTP / gRPC / SQLite inbox
  -> Python DesktopRpaAdapter
  -> 统一 Message / Conversation
  -> C++ 聚合工作台
```

工程约束：

- RPA 只做“搬运工”和“回填助手”，不做无人值守发送。
- RPA 异常时必须停止，不允许盲目重试。
- 输入框回填前必须校验窗口标题、账号、会话对象和输入框状态。
- 涉及金额、退款、投诉、隐私等内容必须人工确认。
- RPA 脚本变更不应要求 C++ 客户端重新发版。

### 3.3 后期设想：两机隔离 RPA 工作机

桌面客户端 RPA 路线存在一个典型风险：焦点劫持。影刀 / UiBot 如果通过传统模拟点击、窗口激活、快捷键或系统消息完成输入框定位和回填，可能会把 Windows 前台焦点从客服正在使用的 C++ 工作台切走，造成客服输入中断、鼠标错位，甚至误填到错误窗口。

当前 M5 不把两机部署作为必选方案，只作为后期可评估的部署形态。若后续客户现场对焦点隔离、坐席体验或 RPA 稳定性要求较高，可以考虑：

```text
客服主机 A
  C++ 聚合工作台
  Python sidecar / adapter hub
  客服查看会话、编辑草稿、确认动作

RPA 工作机 B
  千牛 / 微信客户端
  影刀 / UiBot
  RPA Worker Agent
  负责采集、回填、截图证据、状态回传
```

推荐通信链路：

```text
C++ 工作台
  -> Python adapter hub
  -> LAN HTTP / WebSocket / gRPC
  -> RPA Worker Agent
  -> 影刀 / UiBot / 平台客户端
```

这种形态的价值：

- RPA 点击、窗口激活、输入框定位都发生在工作机 B，不抢客服主机 A 的焦点。
- 客服可以在 A 上稳定使用自研 C++ 工作台。
- B 可以专门维护千牛 / 微信登录态和 RPA 运行环境。
- 后续可以演进为一个客服主机管理多个 RPA Worker。

但它会引入新的工程成本：

- 需要定义远程 Worker 协议、认证、心跳和重连。
- 需要处理局域网发现、防火墙、版本兼容和日志收集。
- 需要明确人工确认动作发生在哪里。
- 需要避免“客服在 A 点确认，B 无校验自动发送”变成新的误发路径。

因此当前建议是：

- M5 仍优先验证单机 PoC：采集可见消息、生成 AI 草稿、回填或复制到剪贴板。
- 两机隔离只保留为后期部署设想，不进入当前 MVP 验收。
- 即使未来引入两机方案，RPA Worker 也应优先只做草稿回填和状态回传。
- 若未来需要由 B 代执行发送，必须增加会话截图、窗口标题、账号、会话对象、待发送文本的二次校验，并在 C++ 工作台明确展示“人工确认后由 RPA 代执行”。

## 4. 不进入当前路线的方案

### 4.1 商业客服 SaaS 旁路

暂不考虑快麦、乐言、福客等商业客服 SaaS 作为旁路主链路。

原因：

- 会削弱自研 C++ 工作台作为主界面的产品价值。
- 商业接口、Webhook、插件能力不确定。
- 数据留存、审计和 AI 编排能力可能受第三方平台限制。
- 当前 MVP 更需要验证自研统一模型和真实平台最小接入链路。

后续如果出现明确商务合作或客户指定平台，可以作为独立调研项，而不是当前 M5 主线。

### 4.2 硬件 HID / 投屏群控

暂不考虑硬件键鼠盒子、手机投屏和群控路线。

原因：

- 运维复杂度高，设备管理成本重。
- 审计、故障恢复和现场部署复杂。
- 容易把软件 PoC 变成硬件运维 PoC。
- 当前阶段还没有足够证据证明必须使用物理输入或投屏方案。

这些方案只作为远期特殊场景备选，不进入 MVP 和 M5 PoC。

## 5. Python sidecar 的适配器职责

引入外部工具后，Python sidecar 应从“AI 服务”逐步扩展为“AI + 外部工具适配 hub”。

建议抽象以下能力：

```text
ExternalToolAdapter
  - health()
  - start()
  - stop()
  - poll_events()
  - fill_draft(conversation_id, text)
  - open_conversation(conversation_id)
```

入站事件建议统一为：

```json
{
  "platform": "pdd_web",
  "account_id": "shop_001",
  "external_conversation_id": "conv_xxx",
  "customer_name": "买家昵称",
  "content": "消息正文",
  "content_type": "text",
  "source_type": "rpa_dom_observed",
  "confidence": 85,
  "observed_at": "2026-05-25T16:00:00+08:00",
  "evidence_ref": "dom_snapshot_hash"
}
```

草稿回填建议统一为：

```json
{
  "platform": "pdd_web",
  "account_id": "shop_001",
  "external_conversation_id": "conv_xxx",
  "text": "建议回复内容",
  "require_manual_confirm": true
}
```

注意：`require_manual_confirm` 在 MVP 阶段必须始终为 `true`。

## 6. 统一模型中的来源标记

外部工具路线必须继续服务统一模型，而不是绕过统一模型。

建议扩展或约定以下 `sourceType`：

| sourceType | 含义 | 默认可信度 | 说明 |
|---|---|---:|---|
| `rpa_dom_observed` | 指纹浏览器 + CDP / Playwright 读取的可见 DOM 内容 | 80-85 | 可归档但必须标记来源 |
| `rpa_ui_observed` | 影刀 / UiBot 通过桌面 UI 或无障碍读取的内容 | 70-80 | 适合千牛、微信 PC |
| `rpa_ocr_observed` | RPA 脚本通过 OCR 获取的内容 | 45-70 | 不建议单独作为权威来源 |
| `rpa_notification_observed` | RPA 或系统通知捕获的新消息提示 | 30-60 | 仅作为提醒或同步触发 |
| `manual_confirmed` | 客服人工确认过的内容或发送动作 | 90-100 | 可触发人工发送记录 |

发送相关状态必须区分：

- 草稿已回填：表示外部工具把内容填入平台输入框。
- 人工已确认：表示客服确认发送。
- 发送结果已观察：表示页面或客户端出现可见成功/失败状态。
- 结果不确定：必须标记为 `unknown` 或 `pending_manual`，不能重复盲发。

## 7. M5 PoC 推荐切入点

M5 建议从一条最小链路开始，不同时推进多个平台。

优先建议：

1. **拼多多 Web + 指纹浏览器 / 普通受控浏览器 + CDP**
   - 更容易观测 DOM 和调试。
   - 更适合验证 Python adapter hub。
   - 先采集可见消息，再做草稿回填。

2. **千牛 PC + 影刀 / UiBot RPA**
   - 更贴近电商客服真实流程。
   - 适合验证桌面客户端 RPA 搬运链路。
   - 需要重点控制窗口识别和误填风险。

M5 最小验收：

- 外部工具能采集至少一条真实平台可见消息。
- Python adapter 能转换为统一 `Message` / `Conversation`。
- C++ 聚合工作台能展示该会话和消息。
- AI 只生成建议或草稿。
- 外部工具最多只做草稿回填或复制到剪贴板。
- 最终发送必须由人工确认。
- 外部工具失败时，C++ 显示降级状态，不崩溃、不误发。

---

## 8. 结论

当前最稳妥的路线不是“全自研平台适配器”，也不是“把客服工作台交给第三方 SaaS”，而是：

```text
C++ 自研工作台
  + Python sidecar / adapter hub
  + 指纹浏览器 / RPA 外部工具
  + 统一模型、状态机和人工确认边界
```

外部工具的价值是降低平台适配层维护成本；项目自身的价值仍然是统一客服工作台、AI 辅助、状态闭环、数据留存和审计。
