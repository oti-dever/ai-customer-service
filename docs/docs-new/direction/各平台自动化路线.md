# 各平台自动化路线分析

> 核心前提：在拿不到官方 API / SDK 的前提下，各平台如何实现客服自动化接入。
>
> 本文档服务于 M5 单平台真实 PoC 阶段的选型决策。

## 已确认前提

| 平台 | 官方能力状态 | 说明 |
|---|---|---|
| 个人微信 PC 客户端 | 不可用 | 无开放 API，第三方均为非官方手段；企业微信有 API 但需企业资质 |
| 千牛 PC 客户端 | 不可用 | 经淘宝开放平台官方确认：聊天 IM 消息 API 权限不对 ISV 开放（隐私管控）；智能客服类目已停止接入；TMC 仅覆盖交易事件不含聊天内容 |
| 拼多多商家后台网页版 | 不可用 | 开放平台要求企业认证 + 招商小二审批，客服消息类接口非个人可申请 |

---

## 一、个人微信 PC 客户端

### 1.1 官方 API / SDK 路线

**结论：不可行。**

微信未向第三方开放个人号消息 API。企业微信有官方 API（会话存档、客户联系等），但需要企业资质和微信认证，不适用于当前项目定位。

### 1.2 Windows UI Automation 路线（wxauto / pywinauto）

**技术现状（2026 年）：**

- wxauto（6900+ stars）是最知名的 UIAutomation 微信自动化库，**已于 2025 年 10 月停止维护**。
- 微信 4.0.5 将 UI 框架从 Qt Widgets 迁移至 Qt Quick/QML，默认情况下控件不再暴露独立 HWND，UIAutomation 工具（如 Inspect.exe）只能看到空壳窗口。
- **但微信并非完全屏蔽 UIA，而是采用"按需暴露"策略**：当系统检测到合法无障碍客户端（如 Windows 讲述人、屏幕阅读器）接入时，微信会加载完整的 UIA Provider，将控件树重新暴露出来。

**"按需暴露"机制详解：**

- 微信 4.1.x 通过系统 API 检测是否有引用 `UIAutomationClient.dll` 和 `UIAutomationTypes.dll` 的 UIA 客户端附着到窗口
- 触发方式一（简单但体验差）：开启 Windows 讲述人（Win + Ctrl + Enter），UI 树即恢复完整暴露
- 触发方式二（推荐）：自建轻量级 UIA 客户端程序，伪装为无障碍客户端附着到微信窗口，触发完整控件暴露
- 触发后可使用 Inspect.exe 正常查看控件树，pywinauto/FlaUI 等工具可正常定位元素
- 已有多个商业 RPA 产品和开源项目（有客 RPA、wxautox4 等）验证此方案可行于微信 4.1.7 / 4.1.8

**UIA 暴露后的能力边界：**

- ✅ 可获取：文本消息内容、联系人名称、窗口标题、按钮/输入框等标准控件
- ❌ 不可直接获取：图片消息、文件消息、语音消息等富媒体内容（仅显示占位符或空节点）
- ⚠️ 富媒体变通方案：通过 UIA 定位消息气泡 → 模拟右键菜单 → 选择"复制" → 读取剪贴板内容；或结合 OCR 识别消息区域图像
- ⚠️ 消息发送者信息：微信 4.x 群聊场景下发送者昵称的 UIA 暴露不完整，可能需要额外处理

**关键约束：**

- 需要先触发无障碍模式（自建 UIA 客户端或开启讲述人），否则控件树为空
- 微信窗口必须保持前台可见（不可最小化）
- 消息读取依赖轮询，无事件回调机制
- 微信对 UIA 客户端的识别存在私有逻辑，不同版本行为可能变化，无法 100% 保证永久兼容
- 禁止微信自动更新仍是基本要求

**适配项目现状：**

- wxautox4：针对微信 4.0.5 适配，已转为商业付费（需激活码），Beta 状态
- wx4py：支持微信 4.1.x，模拟前台操作，功能受限（如群聊无法获取发送者信息）
- 有客 RPA 等商业方案：通过自建 UIA 代理触发暴露，验证可行

**风控风险：** 中等。纯 UI 模拟不侵入进程，但微信 Windows 端反自动化投入最大，高频操作仍可能触发重新登录提示。开启讲述人本身不会触发风控，但自动化操作的频率和模式仍需拟人化处理。

### 1.3 图像识别 + OCR 路线

**技术方案：** 使用 OpenCV/PyAutoGUI 截图 + PaddleOCR/WeChat Native OCR 识别聊天窗口文字，通过坐标点击实现交互。

**优点：** 不依赖控件树，理论上可适配任何版本微信。

**缺点：**
- 极度依赖窗口尺寸和 DPI 设置
- OCR 识别准确率受字体、表情、图片消息影响
- 开发和调试成本极高
- 无法精确区分消息边界和时间戳
- 实时性差，截图 + 识别延迟明显

**适用场景：** 仅作为 UIA 失效时的兜底验证手段，不适合作为主链路。

### 1.4 浏览器自动化路线

**结论：不适用。** 微信 PC 客户端是原生桌面应用，不是 Web 应用，浏览器自动化方案无法直接适用。微信网页版已于 2019 年关闭注册入口，存量账号极少。

### 1.5 第三方 RPA 工具路线（影刀 / UiBot）

**影刀 RPA 在微信场景的能力：**
- 影刀 6.0 支持"AI 搭建流程"，可通过自然语言描述生成自动化流程
- 底层依赖 UIAutomation + 图像识别的混合定位
- 商业 RPA 产品（如有客 RPA）已通过自建 UIA 客户端触发微信无障碍模式实现自动化
- 影刀/UiBot 如果同样实现了无障碍触发机制，理论上可操作微信 4.x

**评估：** 商业 RPA 工具可能已内置无障碍模式触发能力（如有客 RPA 已验证），但引入商业工具带来授权成本和集成灵活性限制。自研项目建议自行实现 UIA 客户端触发机制，成本可控且可完全掌控。

### 1.6 本地客户端 Hook / 逆向路线（仅风险评估）

**WeChatFerry（DLL 注入）：**
- 通过注入 DLL 拦截微信进程内部函数，提供消息回调和后台运行能力
- 必须锁定微信 3.9.x 版本（目前适配到 3.9.12.51）
- 功能最完整：事件回调、后台运行、发送全类型消息

**SQLite 数据库直读：**
- 微信本地消息存储使用 AES 加密的 SQLite
- 运行时从内存获取解密密钥后可直接查库
- 只读能力，不能发送消息

**风险评估：**
- **法律合规：** 注入进程涉及《计算机信息系统安全保护条例》，有明确法律风险
- **账号安全：** 微信 Windows 端反注入检测在持续加强，封号报告多
- **运维成本：** 微信强制更新后函数偏移变化，维护成本极高
- **企业环境：** 可能触发终端安全软件告警

**结论：不推荐作为主链路，不纳入 M5 PoC 方案。**

### 1.7 微信平台对比总结

| 维度 | UIA（触发无障碍模式） | 图像识别 + OCR | RPA 工具 | Hook (WeChatFerry) |
|---|---|---|---|---|
| 可行性 | **中**（文本消息可行，富媒体受限） | 低 | 中 | 高（锁旧版） |
| 开发成本 | 中（需自建 UIA 客户端触发暴露） | 极高 | 中 | 低（直接用 SDK） |
| 稳定性 | 中（暴露机制可能随版本变化） | 低 | 低-中 | 中（锁版本可用） |
| 账号风控风险 | 低-中（UIA 本身不触发风控） | 低 | 中 | 高 |
| 运维复杂度 | 中-高（需维护触发机制 + 版本） | 极高 | 高 | 高（版本锁定） |
| 与 python/ 复用 | 中 | 低 | 低 | 高（Python SDK） |
| M5 PoC 适合度 | ⚠️ 技术可行但不优先 | ❌ 不推荐 | ❌ 不推荐 | ❌ 不推荐 |

### 1.8 微信平台结论

**不推荐作为 M5 PoC 平台，但技术可行性高于此前预期。**

通过触发无障碍模式（讲述人或自建 UIA 客户端），微信 4.x 的 UI 树可以恢复暴露，文本消息的读取和发送通过 UIA 技术上可行。但存在以下制约因素：

- 无障碍暴露属于微信私有策略，无法保证跨版本稳定性
- 富媒体消息需要额外手段（OCR / 剪贴板），增加复杂度
- 微信作为高敏感社交平台，风控压力大于电商平台
- 微信客服场景的业务优先级在当前项目中低于电商

微信平台适合在 M6+ 多平台扩展阶段，以"UIA（触发无障碍）+ OCR 补充"的混合方案接入，优先处理文本消息场景。

## 二、千牛 PC 客户端

### 2.1 官方 API / SDK 路线（旺旺插件 JSSDK / TOP API）

**技术概述：** 千牛是基于插件体系的能力开放平台，ISV 可通过 JSSDK 开发运行在千牛容器内的应用，也可通过 TOP API 调用服务端接口。

**JSSDK 能力边界（插件端）：**
- 可监听 `wangwang.active_contact_changed` 事件（联系人切换），获取当前聊天对象标识
- 可监听交易、退款等业务事件（`bench.trade_info`、`bench.refund_info`）
- 可通过 `my.qn.openChat` 唤起对话窗口并预填待发送文本（但这不是自动发送，仅打开窗口）
- 可通过 `insertText2Inputbox` 向当前聊天输入框插入文本
- 可通过 `getChatContactList` 获取聊天窗口接入列表
- **⚠️ 隐私合规改造：** JSSDK 中大量 `nick` 参数已标记"即将下线"，新开发必须使用 `securityUID`（二方加密 UID）/ `openUID`（三方）+ `bizDomain` 替代

**TOP API 层面的 IM 接口（存在但不可申请）：**

淘宝开放平台文档中存在以下 IM 相关 TOP API：
- `alibaba.interaction.im.conversation.list.query`（IM 会话列表查询）
- `alibaba.interaction.im.message.list.query`（IM 历史聊天消息查询，含 content 字段）
- `alibaba.interaction.im.assistant.message.send`（销售助手发送消息）
- `alibaba.xspace.messagelog.getinformation`（瓴羊客服云按 SID 获取消息记录）

**但经淘宝开放平台官方智能助理确认（2026-05）：ISV 通常无法直接申请到聊天 IM 消息的 API 权限。** 具体原因：
1. **隐私与安全限制：** 客户聊天消息属于高度敏感的隐私数据，平台对此类数据的访问有严格管控，不会对外开放给第三方应用
2. **权限组开放策略：** 上述 IM 接口归属特定业务域（OKKI&TM、瓴羊客服云等），不在公开可申请的权限范围内
3. **智能客服类目已停止接入：** 智能客服相关的应用类目已经停止开放接入，暂时未提供相关能力
4. **TMC 不含聊天内容：** 开放平台提供的消息推送服务（TMC）可用于接收交易、商品、退款等系统事件消息，但不包括买家与卖家之间的聊天内容

**千牛系统消息能力（可用但不是聊天消息）：**
- `taobao.jindoucloud.message.send` 等系统消息 API 仅用于 ISV 向商家发送业务告警/提醒类通知
- 系统消息接入仅限"商家经营主链路环节"（交易、物流、退款等），需联系官方运营人工审批
- 这些是单向通知能力，不是双向聊天消息读写

**ISV 资质门槛：**
- 需企业支付宝实名认证，注册资金 50 万元以上
- 千牛插件新服务商需入驻聚石塔，已上线应用评分 4.5+、付费用户 3000+
- 内部自用工具可走私有插件（不上架），但仍需企业认证
- 消息 API 严禁营销骚扰，违规将被处罚

**结论：官方路线对客服聊天消息场景不可行。** 即使获得 ISV 资质，也无法通过官方 API 读取买家聊天消息原文或代替客服发送聊天消息。JSSDK 提供的能力限于联系人切换感知、打开聊天窗口、插入输入框文本等辅助操作，不构成完整的消息采集与自动回复闭环。此路线不纳入任何阶段方案。

### 2.2 Windows UI Automation 路线（uiautomation / pywinauto / Clicknium）

**技术现状（社区有案例，但存在控件不可达问题）：**

- 已有多个千牛自动化实现（博客园 automation9417 案例、GitHub dify-on-qianniu 等）
- 通过监控千牛客户端"消息提醒"弹窗识别新消息
- 使用 `WindowControl(Name='消息提醒')` 定位弹窗元素

**⚠️ 千牛 UIA 的实际限制：**

千牛客户端底层使用 DirectUI / 自绘控件技术，**并非所有界面元素都有标准控件树结构**。具体表现：

- **"消息提醒"弹窗**：作为系统级通知窗口，通常可被 UIA 识别（这是多数案例的切入点）
- **聊天窗口内部**：消息列表、消息气泡内容等核心区域大量使用自绘渲染，Inspect.exe 看到的可能是空节点或仅有容器级元素
- **阿里云 RPA 自身也无法捕获千牛部分控件**（阿里云开发者社区有用户反馈此问题）
- 与微信 4.x 类似，千牛部分区域的 UIA 控件"看不到"

**社区实际做法（混合方案）：**

dify-on-qianniu 项目（~380 行 Python）的真实技术栈：
- 使用 Clicknium（商业 RPA 引擎）进行元素定位
- **对于可识别控件**（如消息提醒弹窗、发送按钮、回复输入框）：通过 Clicknium 录制 UIA 选择器定位
- **对于不可识别区域**（如聊天消息内容）：**截取聊天窗口图像 → AI 视觉模型 / OCR 提取文字**
- 即：UIA 负责窗口导航和按钮操作，OCR/截图负责消息内容获取
- 关键元素录制：new_message、current_user、chat_window、reply_text、button_send
- Clicknium 文档明确提到："某些特殊的桌面软件中某些控件是没有控件树结构的，此时使用图像识别将发挥重要作用"

**博客园 uiautomation 案例的实际情况：**
- 主要依赖"消息提醒"弹窗作为新消息触发信号（此窗口可被 UIA 识别）
- 进入对话后的消息内容获取方式各案例不同，部分使用剪贴板复制、部分使用 OCR
- 非纯粹的 UIA 控件遍历方案

**关键约束：**
- 千牛窗口需保持前台可见
- 千牛版本更新可能导致控件 ID / 结构变化，需跟进适配
- 消息捕获依赖弹窗轮询或接待列表轮询
- **聊天消息内容不一定能通过纯 UIA 获取**，大概率需要 OCR 或剪贴板辅助

**总结：千牛 UIA 路线 = "UIA 导航 + 图像/OCR 消息获取"混合方案**

- UIA 能做到：定位窗口、检测弹窗、点击按钮、操作输入框
- UIA 做不到（或不稳定）：直接读取聊天消息内容文本
- 消息内容获取需要补充手段：截图 OCR、右键复制到剪贴板、AI 视觉识别
- 与当前项目 `python/` 目录的 RPA 能力有复用空间，但需新增 OCR 模块

### 2.3 图像识别 + OCR 路线

**方案描述：** 截取千牛聊天窗口 → OCR 识别消息文本 → AI 生成回复 → 模拟键盘输入。

**实际案例：** dify-on-qianniu 项目采用此路线获取聊天内容（截图 → Dify 视觉模型识别文字），作为 UIA 无法精确读取消息内容时的补充手段。

**适用场景：** 当 UIA 可以定位窗口和按钮但无法精确读取消息文本时，OCR 作为消息采集的补充。

**风险：** OCR 准确率、表情包/图片消息处理、版本适配。

### 2.4 浏览器自动化路线

**适用性分析：** 千牛有网页版工作台（"千牛工作台 Web 版"），但：
- 功能完整度低于 PC 客户端
- 客服核心功能（接待列表、消息接收）在网页版的可用性未经验证
- 如果网页版客服功能完整，则可复用拼多多场景的浏览器自动化方案

**结论：** 暂不作为主路线，如后续验证网页版客服可用，可作为备选。

### 2.5 第三方 RPA 工具路线

**影刀 RPA + AI 方案：**

已有成熟的"影刀 + AI"千牛客服方案（2025-2026 社区案例）：
- 影刀监控千牛客户端消息提醒
- 抓取聊天记录和订单信息
- 调用大模型（通义千问/DeepSeek）生成回复
- 影刀模拟发送回复

**优缺点：**
- 优点：可视化编排，调试便利，影刀 6.0 支持 AI 工作流
- 缺点：引入商业工具依赖（授权费），与当前项目自有 Python RPA 体系独立，后续难以深度集成

**与自研方案的关系：** 影刀方案可作为快速验证工具，但不适合作为产品化主链路。建议参考其控件定位思路，用自研 `python/` 代码复现核心逻辑。

### 2.6 DLL Hook 路线（仅风险评估）

- 存在旧方案通过替换千牛目录 DLL 或易语言实现的 Hook
- 千牛底层为阿里自研客户端，函数地址随版本快速变化
- 无持续维护的社区生态
- **不推荐，不纳入任何阶段方案。**

### 2.7 千牛平台对比总结

| 维度 | ISV 插件 JSSDK | UIA + OCR 混合方案 | 纯图像识别 + OCR | 影刀 RPA | DLL Hook |
|---|---|---|---|---|---|
| 可行性 | ❌ 不可行（聊天消息权限不开放） | **中-高**（导航可行，内容需 OCR） | 中 | 高 | 低 |
| 开发成本 | — | **中**（UIA + OCR 两套） | 中 | 低（可视化） | 高 |
| 稳定性 | — | **中**（依赖弹窗 + OCR 准确率） | 低-中 | 中 | 极低 |
| 账号风控风险 | — | **低** | 低 | 低 | 高 |
| 运维复杂度 | — | **中** | 高 | 中 | 极高 |
| 与 python/ 复用 | — | **高** | 中 | 低 | 低 |
| M5 PoC 适合度 | ❌ 不可行 | ✅ **推荐（混合方案）** | ⚠️ 辅助 | ⚠️ 备选 | ❌ 不推荐 |

### 2.8 千牛平台结论

**推荐方案：UIA 导航 + OCR/剪贴板消息获取的混合路线。**

千牛 PC 客户端使用 DirectUI 自绘技术，UIA 控件树**并不完整**——弹窗通知、按钮等系统级元素通常可见，但聊天消息内容区域大概率为自绘不可达。社区已有可运行案例验证"UIA 导航 + 截图 OCR 取消息"的混合方案可行，与当前项目 Python RPA 能力有复用空间，风控风险低。需要注意消息内容获取的 OCR 准确率问题。

## 三、拼多多商家后台网页版

### 3.1 官方 API / SDK 路线

**结论：不可行（当前条件下）。**

- 拼多多开放平台客服消息接口仅向企业认证开发者开放
- 需招商小二审批认证
- 个人开发者仅能申请"多多进宝"推广类接口
- 高级接口需企业认证 + 保证金

### 3.2 浏览器扩展路线（Chrome Extension）— 推荐方案

**技术原理：**

Chrome Extension 运行在拼多多商家后台页面内部，可直接访问 DOM，通过 MutationObserver 监听新消息到达，通过 Native Messaging 将事件推送至外部 Python 进程。

**架构：**

```
拼多多页面 (Extension Content Script 注入)
    ↓ MutationObserver 监听聊天区域 DOM 变化
    ↓ 提取新消息文本、发送者、时间
    ↓ chrome.runtime.sendMessage → Background Script
    ↓ Native Messaging Host (Python 进程)
    ↓ gRPC / HTTP → C++ 主壳
```

**核心优势：**

- Extension 运行在页面进程内，天然复用真实登录态和浏览器指纹
- MutationObserver 是事件驱动，实时性优于外部轮询
- 不存在 `navigator.webdriver` 等自动化特征检测问题
- Chrome 官方 Native Messaging 是正规进程间通信机制
- 可安装到指纹浏览器实例中，同时满足环境隔离需求

**关键风险：**

- 拼多多页面结构变更后需更新 DOM 选择器
- Extension 安装需手动侧载（非 Chrome Web Store 发布）
- 发送消息时直接操作输入框 DOM 可能触发框架层事件校验

**与指纹浏览器的配合：** 指纹浏览器（AdsPower/BitBrowser）暴露 Chromium 实例，Extension 可直接安装运行，互不冲突。这解决了多账号环境隔离问题。

### 3.3 Playwright + CDP 接管路线 — 备选方案

**技术原理：** 指纹浏览器启动带登录态的 Chrome 实例，暴露 CDP 调试端口。Playwright 通过 `connect_over_cdp` 接管已运行的浏览器，执行 DOM 操作。

**实现方式：**

```python
# 指纹浏览器 Local API 启动环境，获取 Debug Port
# Playwright 接管
browser = await playwright.chromium.connect_over_cdp(f"http://127.0.0.1:{debug_port}")
page = browser.contexts[0].pages[0]
# 监听新消息、操作输入框
```

**优缺点对比 Extension：**

| 对比项 | Extension | Playwright CDP |
|---|---|---|
| 消息实时性 | 高（MutationObserver 事件驱动） | 中（需轮询或 CDP evaluate） |
| 自动化检测风险 | 最低（页面内执行） | 低（CDP 底层操作，isTrusted=true） |
| 开发调试体验 | 中（需打包加载） | 高（Python 直接编写） |
| 部署复杂度 | 低（一次安装） | 中（需维护 CDP 连接） |
| 页面改版适配 | 需更新选择器 | 需更新选择器 |

**PoC 阶段建议：** 如果团队对 Extension 开发不熟悉，可先用 Playwright CDP 快速验证链路，后续再迁移至 Extension 以获得更好实时性。两者可共用同一个指纹浏览器实例，切换成本低。

### 3.4 图像识别 + OCR 路线

**适用性：** 拼多多是标准 Web 应用，DOM 结构化数据远优于图像识别。OCR 在此场景下完全没有必要，属于降级兜底中的兜底。

**结论：不推荐。**

### 3.5 第三方 RPA 工具路线

**影刀 RPA + AI 拼多多方案：**

已有影刀社区案例（2025 年 12 月）：
- 影刀抓取拼多多聊天记录和订单信息
- AI 分析需求生成回复话术
- 影刀模拟发送回复

**评估：**
- 影刀底层对 Web 页面的操作仍然是 DOM 定位或图像定位
- 引入商业工具依赖，与自有代码体系分离
- 影刀在电商场景成熟度高，可作为快速验证参考
- 但不建议作为产品化主链路（授权成本、定制灵活性受限）

### 3.6 网络层拦截（WebSocket 监听）— 高风险补充

**技术原理：** 拼多多客服消息通过 WebSocket 实时推送，可通过 Playwright `page.on('websocket')` 或 Extension `chrome.debugger` API 监听 WebSocket 帧，直接获取结构化消息数据。

**优势：** 数据最精确、最完整（含订单号等 DOM 可能缺失的字段）。

**高风险因素：**
- 拼多多 anti-content 参数涉及设备指纹、动态加密盐、时间戳同步
- 加密逻辑定期更新混淆规则
- 被检测到的概率高于纯 DOM 操作

**结论：不纳入主链路。仅在需要获取 DOM 无法提供的结构化字段时作为补充校验。**

### 3.7 进程 Hook 路线（仅风险评估）

- 存在通过 Hook 拼多多客户端进程实现自动回复的方案
- 拼多多有完善的前端反作弊体系
- 进程级 Hook 被检测风险极高
- **不推荐，不纳入任何阶段。**

### 3.8 拼多多平台对比总结

| 维度 | Chrome Extension | Playwright CDP | 影刀 RPA | WebSocket 拦截 | 进程 Hook |
|---|---|---|---|---|---|
| 可行性 | **高** | **高** | 高 | 中 | 低 |
| 开发成本 | 中 | **低** | 低 | 高 | 极高 |
| 稳定性 | **中-高** | 中 | 中 | 低 | 极低 |
| 账号风控风险 | **最低** | 低 | 低 | 中-高 | 极高 |
| 运维复杂度 | 低 | **低** | 中 | 高 | 极高 |
| 与 python/ 复用 | 中（需 JS 开发） | **高** | 低 | 中 | 低 |
| M5 PoC 适合度 | ✅ 中期推荐 | ✅ **PoC 首选** | ⚠️ 参考 | ❌ 不推荐 | ❌ 不推荐 |

### 3.9 拼多多平台结论

**推荐方案：Playwright CDP 接管（PoC 阶段）→ Chrome Extension（产品化阶段）。**

PoC 阶段用 Playwright CDP 快速验证消息采集和发送能力，开发效率最高。验证通过后迁移至 Extension 方案获得更好的实时性和稳定性。

## 四、通用路线补充（其他电商/客服平台）

### 4.1 抖店 / 抖音电商

## 五、M5 PoC 推荐路线

### 5.1 推荐平台：拼多多商家后台网页版

**选择理由：**

1. **技术可行性最高：** Web 应用天然适合自动化，DOM 结构可直接操作，无需对抗原生控件变更
2. **风控风险可控：** 通过指纹浏览器隔离环境 + CDP/Extension 复用真实登录态，检测风险最低
3. **开发效率最高：** Playwright Python API 可直接与现有 `python/` 代码复用，无需额外技术栈
4. **社区验证充分：** 影刀、DrissionPage 等工具在拼多多场景均有成功案例
5. **与项目架构契合：** Web 消息采集 → Python 服务处理 → gRPC 推送 C++ 主壳，链路清晰

### 5.2 推荐自动化路线：Playwright CDP 接管 + 指纹浏览器

**具体方案：**

1. 使用 AdsPower（或其他指纹浏览器）创建独立浏览器环境
2. 手动登录拼多多商家后台，保持登录态
3. Python 通过指纹浏览器 Local API 获取 Debug Port
4. Playwright `connect_over_cdp` 接管浏览器
5. 监听客服对话页面 DOM 变化，捕获新消息
6. 推送至 AI 服务获取回复建议
7. 将建议展示在 C++ 主壳界面
8. 人工确认后通过 Playwright 操作输入框发送

### 5.3 PoC 最小闭环能力

| 能力 | M5 必须实现 | 说明 |
|---|---|---|
| 会话识别 | ✅ | 从 DOM 中提取当前对话的买家标识 |
| 消息读取 | ✅ | 监听聊天区域 DOM 变化，提取新消息文本 |
| AI 建议生成 | ✅ | 调用已有 Python AI 服务生成回复建议 |
| 建议展示 | ✅ | 在 C++ 主壳 UI 中展示 AI 建议 |
| 人工确认发送 | ✅ | 用户点击确认后，通过 CDP 操作页面发送 |
| 状态回传 | ✅ | 发送成功/失败状态回传至主壳更新 UI |

### 5.4 M5 暂不实现的能力（避免过度工程化）

| 能力 | 暂不做 | 原因 |
|---|---|---|
| 多账号并发 | ❌ | PoC 仅验证单账号链路 |
| 自动发送（无人工确认） | ❌ | 产品边界要求人工确认 |
| 完整消息历史同步 | ❌ | 仅处理实时新消息 |
| 复杂消息类型（图片/视频/商品卡片） | ❌ | 先支持文本消息 |
| 多平台聚合 | ❌ | M5 仅单平台 |
| 指纹浏览器自动管理 | ❌ | 手动创建环境即可 |
| WebSocket 消息拦截 | ❌ | DOM 方案足够验证 |
| 完善的异常恢复 | ❌ | 基础重连即可 |

## 六、备选 PoC 路线：千牛 PC + UIA/OCR 混合

如果因业务原因（如团队实际使用千牛而非拼多多），需要优先对接千牛：

**方案：** Python uiautomation 库（窗口导航/按钮操作）+ OCR/剪贴板（消息内容获取）

**最小闭环：**
1. 监控千牛"消息提醒"弹窗（`WindowControl(Name='消息提醒')`，此控件通常可被 UIA 识别）
2. 通过 UIA 自动点击进入对话
3. 读取聊天窗口消息内容——**优先尝试 UIA 控件树取文本；若不可达则截图 + OCR，或模拟 Ctrl+A/Ctrl+C 取剪贴板**
4. AI 生成回复建议 → 推送至主壳
5. 人工确认后通过 UIA 定位输入框 → 填入文本 → 点击发送按钮

**相比拼多多方案的劣势：**
- 需要千牛窗口前台可见，占用桌面空间
- 千牛使用 DirectUI 自绘，聊天内容区域大概率需要 OCR 补充（而非纯 UIA 遍历）
- 千牛版本更新可能破坏弹窗/按钮的控件定位
- UIA 轮询实时性低于 DOM MutationObserver
- 多账号场景需要多个千牛实例（每个占一个 Windows 会话）
- OCR 引入额外的准确率不确定性（表情、图片消息处理困难）

## 七、实施路线图

### 7.1 短期 PoC 方案（M5，2-3 周）

**目标：** 验证"消息采集 → AI 建议 → 人工确认发送"单链路可行性。

**技术选型：**

2. 编写 Python 脚本：通过 Local API 获取 Debug Port → Playwright 接管

### 7.3 长期可扩展方案（M6+）

**目标：** 多平台聚合，插件化扩展。

**架构目标：**
- 平台适配器插件化（每个平台一个独立适配器）
- 支持热加载新平台适配器
- 统一消息模型跨平台聚合
- 多账号调度和负载分配
- 完善的健康检查和自动恢复

## 八、需要保留的抽象边界

无论选择哪条路线，以下抽象层必须保持独立：

```
┌─────────────────────────────────────────────────────┐
│  C++ 主壳 (UI 展示层)                                │
│  - 会话列表、消息区、AI 建议展示、确认发送按钮         │
├─────────────────────────────────────────────────────┤
│  消息路由层 (MessageRouter)                           │
│  - 统一消息格式分发、状态回传                         │
├─────────────────────────────────────────────────────┤
│  平台适配层 (PlatformAdapter 接口)                    │
│  - 定义：connect / listen / sendMessage / getStatus  │
│  - 每个平台一个实现                                   │
├─────────────────────────────────────────────────────┤
│  RPA 操作层 (具体自动化实现)                          │
│  - Playwright CDP / Chrome Extension / UIA          │
│  - 环境管理：指纹浏览器启动/停止                     │
├─────────────────────────────────────────────────────┤
│  消息解析层 (MessageParser)                           │
│  - DOM 选择器解析 / UIA 控件树解析 / OCR 解析        │
│  - 输出统一 Message 结构                             │
├─────────────────────────────────────────────────────┤
│  输入发送层 (InputSender)                             │
│  - 文本填入 + 发送触发                               │
│  - 支持草稿回填模式（人工确认后发送）                 │
└─────────────────────────────────────────────────────┘
```

**关键设计原则：**
- UI 层不知道底层用的是 Playwright 还是 UIA
- PlatformAdapter 接口统一，新增平台只需实现新的 Adapter
- MessageParser 和 InputSender 可独立替换（DOM → OCR → UIA）
- 环境管理（指纹浏览器/窗口管理）与业务逻辑分离

## 十、结论与决策摘要

### 推荐方案（M5 PoC）

| 决策项 | 选择 | 原因 |
|---|---|---|
| PoC 平台 | **拼多多商家后台网页版** | 技术可行性最高、风控风险最低、开发效率最高 |
| 自动化路线 | **Playwright CDP + 指纹浏览器** | Python 直接开发、复用现有代码、PoC 验证最快 |
| 产品化升级 | Chrome Extension + Native Messaging | 实时性和稳定性更优 |

### 备选方案

| 场景 | 方案 |
|---|---|
| 业务需求优先千牛 | 千牛 PC + Python uiautomation |
| Extension 开发能力不足 | 长期停留在 Playwright CDP 方案 |
| 后续需要微信支持 | 评估 wx4py (微信 4.x) 或 WeChatFerry (锁 3.9.x) |

### 不建议方案

| 方案 | 原因 |
|---|---|
| 微信作为 M5 PoC 平台 | 4.x 架构变更导致所有合规路线不稳定 |
| 任何平台的 DLL Hook / 进程注入 | 法律风险、运维成本、账号安全 |
| WebSocket 拦截作为主链路 | 加密参数频繁变化，维护成本极高 |
| 纯图像识别 + OCR 作为主链路 | 准确率和实时性均不满足需求 |
| 影刀/UiBot 作为产品化主链路 | 商业授权依赖、集成灵活性不足 |
