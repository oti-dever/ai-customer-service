# 产品形态

更新时间：2026-06-15

## 当前判断

现阶段产品形态优先采用“单机客服工作台”：

- 每个客服电脑上部署完整项目包，包括 C++ 客户端、Python 服务、RPA 运行环境和本机统一数据库 `database/app_data.db`。
- 每个客服主要管理自己电脑上的微信、千牛等平台窗口。
- 平台会话、消息、聚合界面发出的消息、AI 辅助过程数据和界面状态数据都优先保存在本机。
- 数据天然按客服电脑隔离，一台机器的问题只影响当前客服。

这个方向比“客户端和 Python 服务分离部署”更符合当前项目的技术现实。项目当前依赖前台 RPA 模拟人工操作，不是纯后台协议接入。Python 服务需要操作本机真实窗口、焦点、剪贴板、控件树、截图和发送框，因此监听、识别和发送动作天然绑定到客服当前电脑的桌面环境。

## 推荐默认形态：每个客服电脑独立运行

默认部署模型：

```text
客服电脑
├─ C++ 客户端
├─ Python 服务
├─ 微信/千牛等平台窗口
├─ RPA 自动化能力
└─ database/app_data.db
```

默认职责边界：

- Python 服务负责监听平台窗口、识别新会话/新消息、执行发送、写入业务事实表。
- C++ 客户端负责聚合界面展示、用户操作入口、AI 辅助、UI 状态和本地配置。
- `app_data.db` 是单机事实库，默认由 Python 写业务会话/消息，C++ 直接读取展示。
- 清空聊天记录、删除会话、发送消息等业务修改动作继续通过 Python 服务执行。
- C++ 可以写用户、草稿、上次选择、AI 请求记录、客户画像等客户端私有数据。

这个模型的优点：

- 部署和排障简单，每台客服电脑自包含。
- 数据归属清楚，客服自己的数据留在自己的电脑上。
- RPA 操作路径最短，不需要跨机器调度窗口状态。
- 故障半径小，单台电脑异常不会影响其他客服。
- 与当前 `single_host_service_db` 和 `app_data.db` 统一库方案一致。

## 暂不主推的形态一：客户端和 Python 服务分离部署

设想形态：

```text
电脑 A：C++ 客户端
电脑 B：Python 服务 + 微信/千牛窗口
外部数据库：MySQL 或其他中心库
```

这个形态看起来可以让所有消息流向客户端所在电脑，但实际会引入更复杂的数据和运行管理问题：

- Python 服务所在电脑才有真实平台窗口，客户端所在电脑无法直接感知窗口焦点、控件树、截图、登录态和发送框状态。
- 如果数据集中到 MySQL，需要设计客服、设备、平台、账号、窗口、会话、消息、发送任务、RPA 状态等多维归属。
- 多客服账号、多微信窗口、多千牛窗口在同一台 Python 服务电脑上切换，会增加焦点争抢和响应延迟。
- 发送仍然是前台 RPA 模拟人工操作，集中化数据库不会消除 RPA 本身的速度瓶颈。
- 故障链路变长，网络、数据库、远程 Python 服务、目标平台窗口任一环节异常都会影响客户端体验。

因此该形态暂不作为近期默认目标。

## 暂不主推的形态二：一个客户端连接多个 Python 服务

设想形态：

```text
电脑 A：C++ 客户端
电脑 B/C/D：多个 Python 服务 + 多个平台窗口
外部数据库：MySQL 或其他中心库
```

这个形态进一步扩大了复杂度：

- 多个 Python 服务同时写同一业务库，需要处理写入并发、消息顺序、去重、会话归属和发送状态一致性。
- 同一客户可能出现在多个平台账号或多个客服窗口中，需要额外的合并和隔离策略。
- 客户端需要知道每条会话归属哪个服务、哪个机器、哪个账号、哪个窗口。
- 清空、删除、发送等 mutation 需要路由到正确的 Python 服务，否则容易出现“数据库已改但真实平台窗口未操作”或相反的问题。
- RPA 不是后台任务队列，多个窗口之间频繁切换会降低稳定性和响应速度。

该形态更像后期企业级集中调度系统，不适合作为当前产品主线。

## 后续可保留的扩展方向

虽然当前默认不做中心化主链路，但可以保留轻量扩展能力：

- 数据导出：每台客服电脑定期导出会话量、消息量、AI 使用量、响应指标等统计数据。
- 管理端统计：后期可以从各单机节点汇总只读指标，而不是让所有节点实时共写中心主库。
- 备份同步：支持把本机 `app_data.db` 或导出数据备份到指定目录/对象存储。
- 远程服务模式：作为架构预留，只在后期明确需要多客服集中调度、账号托管或后台协议接入时再重新设计。

## 当前落地原则

后续实现优先围绕默认同机模式推进：

- 默认运行模式继续保持 `single_host_service_db`。
- 默认统一库继续使用 `database/app_data.db`。
- Python 服务是业务事实写入方。
- C++ 客户端是业务事实读取方，同时写客户端私有状态。
- 旧 `client_cache_db` 模式只作为兼容或调试能力，不作为主链路继续扩展。
- 远程服务和中心库方案只做概念预留，不影响当前默认链路。
