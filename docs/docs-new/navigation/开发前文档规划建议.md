当前项目已经形成了从总体架构到 MVP 落地的完整文档包，核心文档包括：

- [`桌面客服工作台系统_架构分析文档.md`](../direction/桌面客服工作台系统_架构分析文档.md)
- [`多平台会话聚合_架构分析文档.md`](../direction/多平台会话聚合_架构分析文档.md)
- [`MVP范围说明.md`](../scope/MVP范围说明.md)
- [`用户场景与核心流程.md`](../scope/用户场景与核心流程.md)
- [`统一数据模型与状态机.md`](../design/统一数据模型与状态机.md)
- [`平台接入PoC方案.md`](../design/平台接入PoC方案.md)
- [`工程骨架与模块边界.md`](../design/工程骨架与模块边界.md)
- [`开发任务拆分与里程碑.md`](../execution/开发任务拆分与里程碑.md)
- [`测试验收清单.md`](../execution/测试验收清单.md)

这意味着“开发前还缺什么”已经基本补齐。当前不再需要继续补大而全的架构文档，而是应该进入实现阶段，并在实现过程中按需维护少量补充说明。

现在仍然有价值的不是再写更多宏观文档，而是：

- 保持这些文档与代码实现一致
- 根据实际开发进展更新任务拆分和验收项
- 在真正接入第一个平台后，补充 PoC 结果和已知限制

换句话说，文档阶段已经从“规划不足”切换到“文档够用，进入开发”。

如果后续开发中需要更细粒度的说明，可以再补以下轻量文档，但不是当前阶段的硬需求：

- UI 信息架构草案
- 本地存储设计
- 日志与审计设计
- AI 辅助回复边界
- 安全与合规注意事项

这几份文档只要 1-2 页或少量说明即可，不需要再展开成大文档。

## 5. 暂时不建议再写太深的文档

以下文档可以后置，不建议 MVP 前投入太多：

| 文档 | 后置原因 |
|---|---|
| 完整插件市场设计 | 当前先验证单平台接入和统一模型，不需要插件市场 |
| 完整多租户 SaaS 架构 | MVP 是桌面工作台和本地链路优先 |
| 大规模分布式运维方案 | 现阶段没有规模化部署数据 |
| 全平台自动回复风控体系 | 自动发送暂不进入 MVP |
| 完整数据仓库 / BI 方案 | 先保证消息采集和会话处理可靠 |
| 完整商业化权限体系 | 先保留账号和角色字段即可 |
| 全量自动化测试平台 | 先写 MVP 验收清单和关键手工测试 |

## 8. 文档与开发任务的关系

- [`MVP范围说明.md`](../scope/MVP范围说明.md) 用于确定开发边界，防止范围膨胀
- [`用户场景与核心流程.md`](../scope/用户场景与核心流程.md) 用于指导 UI 和交互
- [`统一数据模型与状态机.md`](../design/统一数据模型与状态机.md) 用于指导 C++ models、core、storage 和 Python 数据协议
- [`平台接入PoC方案.md`](../design/平台接入PoC方案.md) 用于指导第一条真实平台接入链路
- [`工程骨架与模块边界.md`](../design/工程骨架与模块边界.md) 用于指导项目目录和模块职责
- [`开发任务拆分与里程碑.md`](../execution/开发任务拆分与里程碑.md) 用于生成实际开发任务
- [`测试验收清单.md`](../execution/测试验收清单.md) 用于判断每个阶段是否可交付

当前已经可以开始开发，建议开发过程中只维护两类文档：

1. 任务拆分与里程碑
2. 测试验收清单

如果实际 PoC 有新发现，再补充一份简短的 PoC 结果说明即可，不必再扩写成新的大文档。
