# MVP 范围说明

## 1. MVP 目标

MVP 的目标不是一次性完成微信、千牛、拼多多多平台聚合，也不是实现完整自动客服系统。MVP 的核心目标是：

1. 跑通 Windows 桌面客服工作台的基础体验
2. 建立统一会话与消息模型
3. 建立本地会话管理、消息状态和草稿能力
4. 打通 C++ / Qt 客户端与 Python 辅助服务的基础通信
5. 实现 AI 辅助回复建议，但不自动发送
6. 通过一个真实平台 PoC 验证非官方技术接入的可行性

MVP 必须先证明“统一工作台 + 单平台接入 + 人工确认发送”这条主链路可行，再考虑多平台扩展、自动化增强和复杂风控。

## 2. MVP 阶段基本原则

- 先单平台，不同时推进微信、千牛、拼多多
- 先模拟数据跑通完整客户端链路，再接真实平台
- 先统一模型和状态机，再做平台细节
- 先人工确认发送，不做无人值守自动发送
- 平台接入通过适配器隔离，UI 不直接依赖具体平台
- 非官方技术接入必须可关闭、可降级、可审计
- Hook / 注入 / 未授权流量抓包不进入 MVP

## 3. 平台范围

### 3.1 MVP 默认平台策略

MVP 分两步：

1. **模拟平台**
   - 用模拟消息源跑通会话列表、消息区、发送状态、AI 建议、本地存储和异常状态

2. **单平台真实 PoC**
   - 从一个平台开始验证真实接入
   - 优先选择 Web 化平台或可控测试页面
   - 不建议第一阶段直接做个人微信 Hook 或复杂桌面注入

### 3.2 候选平台优先级

| 平台 | MVP 优先级 | 推荐接入方式 | 说明 |
|---|---|---|---|
| 模拟平台 | P0 | 本地模拟适配器 | 必须先完成，用于验证统一模型和 UI |
| 拼多多商家后台网页版 | P1 | 受控浏览器 / DOM 监听 / 人工确认发送 | 更适合验证 Web 自动化链路 |
| 千牛 PC 客户端 | P2 | UI 自动化 / 无障碍 / OCR / 人工确认 | 桌面可见内容采集难度较高 |
| 个人微信 PC 客户端 | P3 | UI 自动化 / 通知 / OCR / 人工确认 | 风险最高，不建议作为第一个真实 PoC |

### 3.3 MVP 只承诺一个真实平台 PoC

MVP 完成时，只要求一个真实平台 PoC 能证明：

- 可以识别账号或登录状态
- 可以观察到会话列表或新消息提醒
- 可以获取当前会话的可见消息
- 可以生成 AI 回复建议
- 可以填入草稿并由人工确认发送
- 可以在失败时安全降级

不要求同时支持多个真实平台。

## 4. 功能范围

### 4.1 必做功能

| 功能 | MVP 要求 |
|---|---|
| 桌面主窗口 | 提供基础客服工作台界面 |
| 会话列表 | 展示模拟平台会话，支持选中、未读、最后消息时间 |
| 消息区 | 展示当前会话消息，支持文本消息 |
| 输入区 | 支持输入草稿、清空、发送模拟消息 |
| 消息状态 | 支持 `draft`、`pending`、`sent`、`failed`、`unknown` 等基础状态 |
| 平台账号状态 | 展示模拟账号或 PoC 平台账号状态 |
| 统一模型 | 实现 Conversation、Message、PlatformAccount 基础字段 |
| 本地缓存 | 保存最近会话、消息、草稿和基础状态 |
| Python 通信 | C++ 客户端可调用 Python 服务获取 AI 建议 |
| AI 建议 | 根据当前会话上下文生成回复建议 |
| 人工确认发送 | AI 建议只能填入输入框，不自动发送 |
| 平台适配器接口 | 定义模拟适配器和真实 PoC 适配器统一接口 |
| 日志 | 记录消息采集、发送确认、AI 建议、异常降级 |

### 4.2 真实平台 PoC 功能

真实平台 PoC 只要求验证最小链路：

- 识别平台窗口或浏览器页面
- 判断登录状态是否可用
- 识别当前会话或会话列表中的新消息
- 读取当前可见消息文本
- 将识别结果转换为统一 Message
- 支持 AI 生成回复建议
- 将建议文本填入目标输入框或复制到剪贴板
- 发送前必须人工确认

### 4.3 AI 功能范围

MVP 只做 AI 辅助回复建议：

- 输入：当前会话最近若干条消息、平台类型、可选客户上下文
- 输出：1-3 条候选回复
- 行为：插入输入框或复制，不自动发送
- 日志：记录请求时间、输入摘要、输出摘要、是否被采用

MVP 不做复杂知识库、自动质检、自动工单、自动售后决策。

## 5. 明确不做事项

MVP 明确不做以下内容：

| 不做事项 | 原因 |
|---|---|
| 同时接入微信、千牛、拼多多 | 范围过大，无法验证单链路质量 |
| 个人微信 Hook / DLL 注入 | 风险高，不适合作为 MVP 默认路径 |
| 未授权流量抓包 / 中间人代理 | 合规和账号风险高 |
| 无人值守自动发送 | 误发风险高 |
| 批量自动回复 | 风控和业务风险高 |
| 完整插件热更新 | MVP 只需要接口边界，不需要复杂插件平台 |
| 完整权限系统 | MVP 可保留账号和角色字段 |
| 多租户 SaaS 架构 | 当前重点是桌面端基础链路 |
| 完整 BI / 数据仓库 | 先保证消息链路可用 |
| 复杂风控策略平台 | MVP 只做硬性禁用和人工确认 |
| 高保真 UI 设计 | 先保证工作台主流程可用 |

## 6. 核心用户场景

### 6.1 模拟平台主流程

1. 客服打开 Windows 桌面客户端
2. 客户端加载本地会话列表
3. 模拟平台产生新消息
4. 会话列表出现未读提示
5. 客服打开会话
6. 消息区显示最近消息
7. 客服点击 AI 建议
8. Python 服务返回候选回复
9. 客服选择一条建议填入输入框
10. 客服手动点击发送
11. 消息状态从 `pending` 变为 `sent` 或 `failed`

### 6.2 真实平台 PoC 流程

1. 客服正常登录目标平台
2. 工作台识别目标平台窗口或浏览器页面
3. 适配器采集当前可见会话和消息
4. 系统生成统一 Conversation / Message
5. 客服在统一工作台查看消息和 AI 建议
6. 客服选择建议文本
7. 系统将文本复制或填入原平台输入框
8. 客服在原平台确认发送
9. 系统记录发送确认日志

## 7. 数据与状态范围

### 7.1 MVP 核心实体

- `Conversation`
- `Message`
- `PlatformAccount`
- `ConversationEvent`
- `SendMessageCommand`
- `AISuggestion`

### 7.2 MVP 必须支持的字段

| 实体 | MVP 字段 |
|---|---|
| Conversation | id、platformType、title、lastMessage、lastMessageAt、unreadCount、status |
| Message | id、conversationId、direction、contentType、content、status、createdAt、sourceType、confidence |
| PlatformAccount | id、platformType、displayName、status、lastActiveAt |
| AISuggestion | id、conversationId、content、createdAt、accepted |

### 7.3 数据来源可信度

MVP 必须记录数据来源：

- `mock`
- `dom_observed`
- `ui_observed`
- `ocr_extracted`
- `notification_observed`
- `manual_confirmed`

MVP 不把 OCR 或通知内容作为高置信度权威消息，只作为辅助信息。

## 8. 技术范围

### 8.1 C++ / Qt 客户端

MVP 需要实现：

- 主窗口
- 会话列表
- 消息列表
- 输入区
- AI 建议展示区
- 平台状态提示
- 本地数据管理
- 平台适配器接口
- 与 Python 服务通信

### 8.2 Python 辅助服务

MVP 需要实现：

- 基础 HTTP 或本地 IPC 服务
- AI 回复建议接口
- 简单规则保护，例如禁止自动发送
- 日志输出

### 8.3 平台适配器

MVP 至少实现：

- `MockPlatformAdapter`
- 一个真实平台 PoC Adapter

适配器能力最小集合：

- `start`
- `stop`
- `healthCheck`
- `fetchVisibleConversations`
- `fetchVisibleMessages`
- `prepareReplyDraft`

MVP 不要求实现完整插件热加载。

## 9. 验收标准

### 9.1 模拟平台验收

MVP 必须满足：

- 客户端可启动并显示主窗口
- 能展示模拟会话列表
- 能打开会话并展示消息
- 能输入和发送模拟消息
- 消息状态可变化
- 能请求 Python 服务返回 AI 建议
- AI 建议不会自动发送
- 关闭重启后最近会话和草稿仍可恢复

### 9.2 真实平台 PoC 验收

至少满足：

- 能识别目标平台是否处于可用状态
- 能采集一条可见消息并转成统一 Message
- 能将消息展示到统一工作台
- 能基于该消息生成 AI 建议
- 能将建议填入原平台输入框或复制到剪贴板
- 发送动作由人工确认
- 采集失败时系统能降级，不崩溃、不误发

### 9.3 安全验收

MVP 必须满足：

- 不保存明文密码
- 不记录明文 Cookie / Token
- 不做未授权抓包解密
- 不做 Hook / 注入作为默认能力
- 不做自动批量发送
- 日志中不保存完整敏感截图

## 10. 成功标准

MVP 成功的标志不是接入了多少平台，而是：

- 统一会话工作台主流程跑通
- 单个平台 PoC 能证明技术接入路线可行
- AI 建议能进入人工确认流程
- 系统在失败时能安全降级
- 代码结构为后续多平台适配留下边界

## 11. 后续演进方向

MVP 完成后再考虑：

1. 增加第二个平台适配器
2. 完善本地 SQLite 表结构和全文搜索
3. 增加平台接入健康监控
4. 增加更严格的日志与审计
5. 评估自动化填充的稳定性
6. 评估是否需要受控浏览器框架
7. 评估是否进入更高风险的实验链路
