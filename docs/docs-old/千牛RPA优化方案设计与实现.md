# 千牛 RPA 优化方案设计与实现（聊天区截图优先 · 免聊天区 OCR）

本文档描述 **千牛 Reader 聊天区** 的优化：**对聊天区仅截图、不做 OCR**，将 **PNG 路径** 交给聚合界面展示；**会话列表、会话标题、Writer 会话校验等仍保留既有 OCR/像素逻辑**。与《AI辅助回复方案设计与实现.md》衔接时，聚合侧「生成本条回复」可改为 **「聊天区截图 + 店铺知识 system + 可选用户补充」** 的 **多模态请求**，而不再依赖聊天区 OCR 文本。

**关联文档**：《AI辅助回复方案设计与实现.md》、《AI助手集成方案设计与实现.md》（**豆包多模态请求体、OpenAiCompatClient `content` 数组** 已与内置助手对齐）、《RPA-DB-协议.md》（若 inbox 字段扩展需同步）、《各平台消息链路.md》。

**与当前工程现状对齐**

| 能力 | 状态 |
|------|------|
| 主窗口「机器人管理」对话页 **豆包（火山方舟）** + **图片理解** | **已接入**：选图/图+文发送、流式回复、落库与请求组装见《AI助手集成方案设计与实现》**§5.5～§5.6、§10.8**。 |
| **千牛聊天区仅截图、入库 PNG 路径、聚合气泡显图** | **已落地（§8 阶段 A～C）**：`chat_content_mode=image_only`、PNG 落盘与 `content_image_path`、聚合 `AggregateChatForm` 气泡上图。 |
| 聚合「AI 辅助」对 **图片入站** 调多模态 | **已落地（§8 阶段 D）**：「生成本条回复」在入站含本地截图且文件存在时组装 **`image_url`（data URL）+ 文本**；与 **豆包/DeepSeek 线路选择**、方舟 **`thinking` 关闭**、篇幅约束等见《AI辅助回复方案设计与实现》**§5.5～§5.6、§8**。 |

**状态**：方案已定稿；**具体步骤见 §8**；实现时按阶段推进并更新本文「修订记录」。

## 1. 背景与动机

- 千牛 RPA 单轮扫描中，**聊天区 OCR（`cur_ocr`）** 常为耗时大户；**会话列表 / 标题 / Writer** 等仍依赖 OCR 或像素扫描，不宜一刀切去掉。
- **业务目标**是「知道对方发了什么」：对客服而言，**看清一条消息**即可；不一定先转成字符串再在聚合里展示。
- **因此**：聊天区只截 **一条消息带** PNG，**入库路径**，聚合用 Qt 显示图片；**省掉聊天区 Paddle 推理**，降低延迟与算力占用。
- **实测反馈**：在 **关闭聊天区 OCR（`image_only`）** 后，单轮扫描相对原 **聊天区 OCR** 路径可缩短 **数秒级** 耗时（与硬件、ROI、模型负载相关）。

---

## 2. 设计原则

| 原则 | 说明 |
|------|------|
| 范围可控 | **仅**削减「聊天区」文本 OCR；列表、标题、红点、Writer 等路径不变（除非另有专项）。 |
| 人在回路 | 客服以看图为主；AI 辅助为增强能力，**不**因本方案承诺全自动回复。 |
| 可演进 | 首版以 **路径 + 展示 + 多模态 AI 试验** 为主；去重、清理规则可随线上数据收紧。 |

---

## 3. 数据与存储

| 项 | 定案 |
|----|------|
| 落盘格式 | **PNG 文件路径** 写入业务可消费的结构（与 `rpa_inbox_messages` / 聚合模型对齐；可为新列或附件表）。 |
| 容量策略 | **按会话（`platform_conversation_id`）仅保留最近 10 张**聊天区截图；**超过 10 张即删除最旧文件**（与 DB 记录一致删除）。 |
| 目录建议 | 独立子目录（如 `python/rpa/_media/qianniu_chat/` 下按会话或按日期分桶），避免单目录文件过多。 |

---

## 4. 增量与去重

| 项 | 定案 |
|----|------|
| 主路径 | 与 **未读/红点 → 必要时切换会话** 一致：**需要读当前会话聊天**时才截聊天区；无红点可走 `list_first` 短路，**不截聊天区**。 |
| 去重策略 | **时间窗口 + 像素 diff**（如 MD5/感知 hash/帧差阈值）；聊天区 ROI 已压到「约一条消息」，重复概率低，首版不必过重。 |
| 风险接受 | 极端情况再加强；与「每会话 10 张上限」共同控制堆积。 |

---

## 5. 检索与运营（非目标）

- **会话内关键词搜索、导出报表**：当前产品 **未做**，本方案 **不新增** 义务。
- 需核对原文时，客服 **回千牛客户端** 查看；聚合侧以 **看图 + 发信** 为主。

## 6. AI 辅助回复与多模态

### 6.1 总表

| 项 | 定案 |
|----|------|
| 频率与成本 | 接受「**一条新消息 ≈ 一张图**」进入 AI；内容多为短文本气泡，体量可控。 |
| 模型分工 | **豆包（火山方舟）** 作为 **看图 / 多模态问答** 的主选：官方能力含 **图片理解**，支持 **仅据图生成回复** 或 **图片 + 用户问题联合回答**。**DeepSeek** 可作纯文本或过渡；若某环境 **不支持图像输入**，**降级**（提示切换模型、回退 OCR 或仅文本）在实现阶段定。 |
| 验证路径 | **机器人管理 → 内置 AI 助手**：对话页 **选图/贴图 + 可选文字**，将 **图片 + 系统提示词（及用户问题）** 发往方舟接口，测 **质量与耗时**。 |
| 提示词 | **一条带图 + system（+ 可选 user 文本问题）**；后续叠 **回复策略**，与《AI辅助回复方案设计与实现.md》同步。 |

### 6.2 内置 AI 助手与千牛 RPA 的衔接（现状）

- **已具备（机器人管理）**：在预设 **豆包（火山方舟）** 下，对话页可通过 **工具栏选图** 发送 **仅图 / 图+文**；`OpenAiCompatClient` 已支持 **`messages[].content` 为数组**（`image_url` + `text`），详见《AI助手集成方案设计与实现》**§5.5、§5.6**。
- **已衔接（聚合 + 千牛 RPA）**：当 `messages` 中 **最后一条客户入站** 带 **`content_image_path`**（千牛聊天区 PNG）且文件可读时，聚合 **「生成本条回复」** 使用 **`imageFileToDataUrl`**（与内置助手同源逻辑抽至 `src/utils/imagedataurl.cpp`）组装 **data URL**，与 **店铺知识 system**、**user 文本段** 一并经 **`OpenAiCompatClient`** 流式请求；**线路选择**为聚合内 **「模型：豆包 / DeepSeek」**（读 `QSettings` 的 `ai/presets/doubao_ark`、`deepseek_deepseek-chat` 等，与机器人管理预设对齐）。**纯文本线路**（如 DeepSeek）在 **存在截图入站** 时会 **弹窗提示切换到豆包（方舟）**。
- **体验与协议细节**：对火山方舟请求根域名自动附加 **`thinking: { type: "disabled" }`** 以关闭豆包 Seed 等默认思考链、降低延迟；聚合侧另设 **`max_tokens`** 与 system 中的 **短答** 约束，见《AI辅助回复方案设计与实现》。
- **可选后续**：将 **多模态组装** 再抽一层与 `RobotAssistantWidget` 完全复用（当前为 **共享 data URL 工具 + 同一 `OpenAiCompatClient`**）。

### 6.3 火山方舟 / 豆包侧要点（与设计对齐）

以下摘自方舟公开文档（如《豆包大模型 1.8》《图片理解》《多模态理解》），用于实现时对齐协议，**具体 model / 接入点 ID 以控制台为准**（常见为 **`doubao-seed-…`** 或 **`ep-…`** 接入点）。

| 要点 | 说明 |
|------|------|
| 能力 | 支持 **图片理解**；多模态文档中明确 **图像描述、视觉问答** 等 **文本输出** 场景，与本方案「看图回复 / 图文问答」一致。 |
| 接口形态 | **Chat Completions**（`/api/v3/chat/completions`）与 **Responses**（`/api/v3/responses`）均可把 **图片作为输入**；示例基座 **`https://ark.<region>.volces.com/api/v3`**，鉴权 **`Authorization: Bearer <API Key>`**。 |
| 请求体 | 与「仅 `content: "字符串"`」不同：用户消息需 **`content` 为多段结构**（如 OpenAI 兼容的 **`image_url` + `text`**，或 Responses 的 **`input_image` + `input_text`**）。**图片** 可用 **公网 URL**、**Base64（`data:{mime};base64,...`）**、**Files API 返回的 `file_id`**，或 SDK 支持的 **`file://` 本地路径**（依文档与 SDK）。 |
| 多轮同图 | **Chat API 无状态**：若多轮对话都针对 **同一张图**，文档要求 **每次请求都带上该图片信息**；若用 Responses 且配合平台侧上下文能力，可按官方推荐方式减少重复传图（以最新文档为准）。 |
| 与 DeepSeek 并行 | 内置助手中 **DeepSeek** 与 **豆包** 分轨：前者适合已落地的 **纯文本**；**发图、截图理解、图文联合问答** 以 **豆包 + 多模态 body** 为准。 |

## 7. 边界与假设

- 聊天区 ROI **一条消息** + **系统提示词（及后续策略）**，默认可支撑模型理解「对方刚发了什么」。
- **列表/标题/Writer** 仍负责 **会话身份与发送安全**；与本方案正交。

## 8. 可实现的具体步骤

下列步骤按 **依赖顺序** 排列；可 **分阶段发版**（先落库+展示，再 AI；或先开关灰度）。

### 阶段 A：配置与 Reader 截图链路（Python）

1. **配置项**（如 `qianniu_config.json`）：增加 **`chat_content_mode`**（或等价命名）：`text` | `image_only`；默认 `text` 保持现状，`image_only` 时聊天区不走 OCR。
2. **`qianniu_reader.py` / `_process_current_chat`**：在 `image_only` 分支中，对 **`chat_region` 已有截图缓冲区**（`cb_b` 等）**写 PNG 到约定目录**（文件名含时间戳或单调 id，避免覆盖）。
3. **跳过** 对聊天区内容的 `layout_parser` / 行级 OCR；**保留** 列表 OCR、标题 OCR、红点逻辑、多会话切换逻辑不变。
4. **去重**：写入前对 **本次 PNG 字节或缩小采样** 做 hash，与 **本会话上一张**（或时间窗口内上一张）比较；一致则 **不写库、不落盘**（或仅更新时间戳，按产品定）。
5. **入库**：调用现有 `write_inbox_message` / `db_helper` 扩展：除 `platform`、`platform_conversation_id`、客户名等外，增加 **图片绝对路径** 或 **相对项目根路径**；若当前表仅 `content` 文本，则 **迁移 SQLite**：新增列（如 `content_image_path`）或约定 `content` 特殊前缀 + 路径（以《RPA-DB-协议》为准，优先 **显式列**）。
6. **日志**：`reader_segment_timing` 下将原 `cur_ocr` 改为 **`cur_png_write`** 等毫秒段，便于对比优化效果。

### 阶段 B：按会话保留最近 10 张（Python 或 Qt）

1. **策略**：每次成功写入 **新 PNG** 且绑定 `platform_conversation_id` 后，查询该会话已存在的 **图片路径列表**（按时间排序），**超过 10 条则删除最旧文件**及对应 DB 行（若一行一图）或仅删文件（若以单表多版本记录，按设计裁剪）。
2. **触发时机**：推荐 **每次新图入库后同步裁剪**；启动 Reader 时 **可选全量扫描目录** 纠偏（防崩溃残留）。
3. **异常**：删文件失败（占用中）时打日志，下次重试或人工清理。

### 阶段 C：聚合与 DB（C++/Qt）

1. **Schema**：与 `MessageDao` / `rpa_inbox_messages` 读取路径对齐，**SELECT** 带出图片路径列；**迁移脚本** 给现有库加列（默认值 NULL，旧数据仍走文本）。
2. **`QianniuRPAAdapter`**（或统一 inbox 消费处）：轮询/映射时，若路径非空，**构造消息模型为「图片消息」**（或文本占位 `[图片]` + 隐藏路径供 UI）。
3. **聚合聊天 UI**（如 `AggregateChatForm` 或消息 Delegate）：**QLabel/QPixmap 或异步加载** 显示本地 PNG；限制 **最大显示边长** 与 **异步解码** 避免卡 UI。
4. **发送区**：纯展示链路不要求从图片反解文字；若仍有「引用本条回复」类功能，需约定 **引用的是路径还是 OCR 缓存**（首版可仅路径）。

### 阶段 D：AI 多模态（聚合 AI 辅助；内置助手已具备发图能力）

**状态：已按「生成本条回复」单路径落地**（与内置助手会话表仍独立）。

1. **协议**：`OpenAiCompatClient` **流式 SSE** 与 **`content` 数组**（`image_url` + `text`）与内置助手一致；聚合侧 **`messages` 来源**为当前会话 **`MessageDao::latestInboundSnapshot`**（含 `content` + `content_image_path`）。
2. **内置助手**：**选图 + 豆包多模态** 已落地；验收时仍可用助手页 **同一张 PNG** 对比 **DeepSeek 纯文本 vs 豆包看图**。
3. **聚合「AI 辅助」**（已实现要点）：  
   - **入站仅有文本**：`user` 为单段字符串（与首版一致）。  
   - **入站含可读 PNG 路径**：`user.content` 为 **数组**（**先图后文**）；图来自 **本地文件 → data URL**（**≤约 5MB** 与内置助手一致）。  
   - **模型线路**：界面 **「模型：豆包（火山方舟）/ DeepSeek」** 选择 **`aggregateAi/sessionModelKey`**，API 读 **`ai/presets/<slug>`**（`baseUrl` / `model` / `apiKey`），与机器人管理 **同一套预设**；**截图入站 + 当前选 DeepSeek** → **弹窗引导切豆包并完成方舟配置**（实现上通过 Base URL / 模型名 **是否像方舟视觉端点** 判断）。  
   - **方舟请求**：根 JSON 对 `volces.com` / `ark.cn-beijing` 自动 **`thinking: disabled`**；并带 **`max_tokens`** 与 **短答 system**，见《AI辅助回复方案设计与实现》**§5.5～§5.6**。
4. **UI**：输入区为 **底部半透明浮层** 叠在消息列表之上，聊天记录可滚至其下 **透出**；**模型选择**在输入框 **上方**（详见该文档 **§5.1**）。

### 阶段 E：验收与回滚

1. **验收**：同一会话连续收消息，**延迟对比**（`cur_ocr` 消失 vs 写 PNG）；**磁盘** 每会话 ≤10 张；**聚合** 图文一致；**Writer** 发消息不受聊天区模式影响（仍依赖标题/列表等 OCR）。
2. **回滚**：`chat_content_mode=text` 即恢复聊天区 OCR；配置与迁移列 **向后兼容**。

## 9. 风险摘要

| 风险 | 缓解 |
|------|------|
| 多模态 API 行为不一致 | 内置助手先行验证；聚合侧降级与提示。 |
| 磁盘与权限 | 目录权限、磁盘满、路径长度限制。 |
| 重复入库 | 像素/hash + 每会话 10 张上限。 |
