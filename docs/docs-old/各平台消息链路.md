## 一、整体链路（读 + 写）

```
[平台客户端/网页] 
    ↓ 读：RPA/API 抓消息
[rpa_inbox_messages 或直接 messages]
    ↓ Qt 适配器消费
[MessageRouter → conversations/messages]
    ↓
[聚合界面展示]
    ↑
[用户输入回复 → sendMessage]
    ↑ 写：RPA/API 发消息
[平台客户端/网页]
```

## 二、各平台实现策略

### 1. 微信 PC

**读（平台 → 聚合界面）**

- 当前主路径仍是 **Python RPA + PaddleOCR + 布局分析**。
- 纯 UIAutomation 作为 Reader 主方案目前仍未证实可替代 OCR；但 `poc/pywechat-poc2.0` 已表明，在特定环境下，UIA 对**会话切换、输入框、发送按钮**具备较高价值，适合作为混合增强能力，详见《[微信RPA融合评估与落地建议](./微信RPA融合评估与落地建议.md)》。
- 流程：`PrintWindow` 截微信窗口 → 裁剪消息区 → PaddleOCR 识别 → 按 x 坐标区分左/右/居中（对方/自己/系统）→ 增量去重 → 写入 `rpa_inbox_messages`。
- Qt 端：`WechatRPAAdapter` 轮询 inbox 消费，这部分已经实现。
- 若不走聚合主链路，当前仓库还新增了 **微信独立工作台** 路线：Qt 直接通过 Python UIA 服务读取当前可见会话与消息，不写入微信消息 DB，详见《[微信独立工作台方案设计与实现](./微信独立工作台方案设计与实现.md)》。

**写（聚合界面 → 平台）**

- 当前已落地主路径：**坐标点击 + 剪贴板/逐字输入**。
- 后续更优方向：保留当前写链路作为默认路径，再引入 UIA 作为 fallback，用于**群聊切会话、输入框定位、发送按钮点击**。
- 流程：点击输入框（按配置坐标）→ Ctrl+V 粘贴 → 点击发送按钮或 Enter。
- 实现位置：Python `writer.py` 读取 `messages(sync_status=10)`，执行模拟输入，更新 `sync_status` 为 11/12。

**待做**

- 实现 `wechat_reader_paddleocr.py`（按微信 RPA 文档 v3 方案）。
- 实现 `writer.py` 的微信发送逻辑。
- 配置 `chat_region`、输入框、发送按钮坐标（可用 Qt 校准工具）。

### 2. 千牛 PC

**读**

- 接待中心是 CEF 内嵌，UIAutomation 同样不可用。
- 方案与微信类似：**截图 + OCR + 消息解析**。
- 差异：千牛有会话列表 + 消息区，布局和解析规则不同。

实现路线有两种：

| 路线              | 读                             | 写                       | 优点                               | 缺点                                                 |
| ----------------- | ------------------------------ | ------------------------ | ---------------------------------- | ---------------------------------------------------- |
| **A：Python RPA** | Python 写 `rpa_inbox_messages` | Python Writer 发         | 与微信统一，OCR 一次实现多平台复用 | 需为千牛单独写 Reader/Writer                         |
| **B：C++ RPA**    | 千牛适配器内直接 OCR           | 千牛适配器内直接模拟输入 | 不依赖 Python 进程                 | OCR 需在 C++ 中集成（Windows OCR 或 PaddleOCR ONNX） |

更推荐 **路线 A**：  
- 微信、千牛都用 Python RPA，OCR、截图、模拟输入等通用逻辑可复用。  
- Qt 端只做 inbox 消费和 UI，架构简单。

**写**

- 需要：切换会话 → 点击输入框 → 粘贴 → 点击发送。
- 千牛文档里已有坐标比例、`InputSimulator`、`ClipboardGuard` 等设计，可直接在 Python 中实现。

**待做**

- 新增 `qianniu_reader.py`：窗口发现、区域截图、OCR、千牛专用 `MessageParser`。
- 新增 `qianniu_writer.py`：会话切换、输入框点击、粘贴、发送。
- 千牛 `RegionConfig` 校准（会话列表、消息区、输入框、发送按钮）。

### 3. 拼多多网页

**读**

- 在浏览器中运行，有两种思路：

| 方案                            | 实现方式                                          | 优点                           | 缺点                                          |
| ------------------------------- | ------------------------------------------------- | ------------------------------ | --------------------------------------------- |
| **A：RPA 截图 + OCR**           | 与微信/千牛相同： HWND 找浏览器窗口 → 裁剪消息区 → OCR → 解析 | **与现有 Python RPA 完全同构**；主程序**无需** Qt WebEngine；包体与运行时负担小 | 浏览器多标签、缩放/高分屏、动态布局下标定与解析成本高；依赖 OCR 准确率与增量去重 |
| **B：QWebEngineView + JS 注入** | 内嵌加载拼多多商家后台，用 `runJavaScript` 读 DOM | 直接拿结构化数据，稳定、无 OCR | 主程序需引入 WebEngine（**体积大**）；DOM 仍可能随改版变化 |

**选型建议（与千牛、微信对齐时）**  

- 若希望 **消息链路统一走「截图 + OCR + 模拟输入」**、**聚合端不嵌入 Chromium**，与千牛/微信同一套 `python/rpa/common`，优先 **方案 A**：拼多多仅多一个 `pdd_reader.py` / `pdd_writer.py`（窗口类名、区域、解析规则与千牛类似但单独维护）。  
- 若后续验证 **DOM 长期稳定** 且团队接受 **独立 Qt 套件或子进程** 带 WebEngine，再用 **方案 B** 降低 OCR 成本；两条路可以 **阶段 0 并行 POC**，再定主路线。

**方案 B 的实现思路（备选）**

1. 用 `QWebEngineView` 加载拼多多商家后台登录页。
2. 用户手动登录后，注入 JS 监听聊天相关 DOM（消息列表、会话列表等）。
3. 通过 `QWebEnginePage::runJavaScript` 的回调，把消息数据传回 C++。
4. 在 C++ 中构造 `PlatformMessage`，交给 `MessageRouter`。

**写**

- **方案 A**：与千牛类似——焦点到浏览器窗口与输入框 → 剪贴板粘贴 → Enter/点击发送（需校准坐标或相对比例）。
- **方案 B**：用 JS：`document.querySelector` 找到输入框，`element.value = text` 或 `element.dispatchEvent` 模拟输入，再触发发送按钮的 click。

**待做**

- **方案 A**：浏览器 HWND 发现、消息区/输入区 `RegionConfig`、拼多多专用解析规则、`pdd_reader` / `pdd_writer` 接入 `rpa_inbox_messages`。
- **方案 B**：分析拼多多商家后台聊天页面的 DOM；`PinduoduoWebAdapter`；登录态与 Cookie；可能的反爬。

#### 3.1 独立验证工程（`test/`）

- 主工程引入 Qt WebEngine 前，先在仓库内用**独立 Qt 工程**验证可行性。
- 当前工程路径：**`test/pdd-webengine`**（QWebEngineView 加载商家后台，手动登录后尝试 `runJavaScript` 读 DOM、模拟输入与发送）。
- **约定**：后续凡需「与主程序解耦」的验证（新平台、新控件、Web 注入 POC 等），优先在 **`ai-customer-service/test/<子目录>`** 下新建小工程，避免污染主工程 CMake 与依赖。
- 独立工程内的 `build/`、`.qtcreator` 等产物建议本地忽略或 `.gitignore`，**只提交源码与 CMake 清单**即可。

#### 3.2 分阶段落地（与主链路对齐后再进排期文档）

| 阶段 | 目标 | 说明 |
|------|------|------|
| **阶段 0** | DOM / 发送可行性 | 在 `test/pdd-webengine` 中确认：能否稳定选中消息列表、输入框、发送按钮；输出选择器草案与改版风险说明。 |
| **阶段 1** | 单会话读通 | 定时或 MutationObserver 拉取新消息 → 映射为统一消息模型 → 写入 `rpa_inbox_messages`（或与现有 inbox 消费路径一致）。 |
| **阶段 2** | 单会话写通 | 轮询 `messages(sync_status=10)` → JS 填输入并触发发送 → 回写 11/12（与现有 Writer 轮询 DB 方式一致，执行层为 Web 而非 Win32）。 |
| **阶段 3** | 工程化 | 登录态失效检测、页面跳转恢复、选择器失效告警；多会话切换与发送前会话校验（对齐千牛 Writer 思路）。 |

若主路线定为 **方案 A**，阶段 0 重点验证：**固定窗口尺寸或比例下**，消息区 OCR 与增量是否达标；若改为 **方案 B**，阶段 0 验证 DOM 与注入可行性。两条路也可先做 A 跑通闭环，再视需要用 B 降 OCR 成本。

**与「四周路线图」的关系**：拼多多阶段 0～3 的任务与验收，待千牛/微信基础链路稳定后，再合并进 **`docs/项目任务排期-四周路线图-*.md`**，届时再按周拆解执行；本文档仅保留方案与阶段说明。

## 四、共享基础设施（Python 侧）

为减少重复开发，建议在 `python/rpa/` 下抽象出通用模块：

```
python/rpa/
├── common/
│   ├── screenshot.py      # PrintWindow/BitBlt 封装
│   ├── ocr_engine.py      # PaddleOCR 封装
│   ├── layout_parser.py   # 左/右/居中布局分析（可配置阈值）
│   ├── incremental.py     # 增量检测、content_hash 去重
│   └── input_sim.py      # 点击、剪贴板、键盘模拟
├── readers/
│   ├── wechat_reader.py  # 微信专用：窗口发现、区域、解析规则
│   ├── qianniu_reader.py # 千牛专用
│   └── pdd_reader.py     # 拼多多（可选）：浏览器 HWND + 区域 OCR，与千牛同构
├── writers/
│   ├── wechat_writer.py
│   ├── qianniu_writer.py
│   └── pdd_writer.py     # 拼多多（可选）
├── main.py               # 启动 Reader + Writer 循环
└── config/
    ├── wechat_config.json
    └── qianniu_config.json
```

每个平台的 Reader 只负责：  
- 窗口发现  
- 区域坐标  
- 平台特定的消息解析规则  

OCR、截图、增量检测、Writer 的模拟输入逻辑都可以复用。

## 五、需要你拍板的点

1. **千牛实现路线**：Python RPA（与微信统一）还是 C++ RPA（不依赖 Python）？
2. **拼多多**：是否优先做？若做，优先 **截图 + OCR（与千牛同构、主程序无 WebEngine）** 还是 **QWebEngineView + JS**（需引入 WebEngine，见上文取舍）？
3. **Writer 触发方式**：  
   - 方案 A：Python 轮询 `messages WHERE sync_status=10`；  
   - 方案 B：Qt 通过进程间通信（如命名管道、本地 socket）主动通知 Python 有待发消息。  
   方案 A 实现简单，方案 B 延迟更低。
4. **多会话**：微信/千牛都有多会话，是先做「单会话 MVP」再扩展，还是一开始就支持多会话切换？

你可以先选：  
- 千牛用 Python 还是 C++；  
- 拼多多是否纳入当前阶段；  
- Writer 用轮询还是通知。  

确定后，可以按平台拆成具体任务列表，一步步实现。

---

我的建议：千牛实现路线与微信统一；拼多多暂时不做，关于是否用QWebEngineView，可以之后单独创建个新qt项目测试下是否可行再决定；Writer 触发方式先用方案A吧，先简单实现，不要求延迟；先做「单会话 MVP」再扩展吧；python/rpa目录下的结构按照你的来。

这些决策都很合理，可以按这个方向推进。

## 1. 千牛与微信统一用 Python RPA

- 一套 OCR、截图、模拟输入逻辑，两个平台共用
- Qt 端只做 inbox 消费，架构简单
- 后续加京麦、抖店等，也是同样的模式

---

## 2. 拼多多暂缓，QWebEngineView 单独验证

- 先把微信、千牛这条主链路跑通，再考虑拼多多
- 用独立 Qt 项目验证 QWebEngineView + JS 注入，风险可控
- 验证通过后再决定是否在主项目中引入 Qt WebEngine

---

## 3. Writer 用轮询（方案 A）

- 实现成本低，不涉及进程间通信
- 800ms–1s 轮询对人工回复场景足够
- 以后若需要更低延迟，再考虑 IPC 或 Qt 主动拉起 Python 子进程

---

## 4. 单会话 MVP

- 微信、千牛都先只支持「当前选中会话」
- 多会话切换、会话列表识别可以放到第二阶段
- 能先把「读 + 写」闭环跑通，再扩展

---

## 5. `python/rpa` 目录结构

建议采用如下结构，便于后续扩展：

```
python/rpa/
├── common/                 # 平台无关的通用模块
│   ├── __init__.py
│   ├── screenshot.py       # PrintWindow/BitBlt
│   ├── ocr_engine.py       # PaddleOCR 封装
│   ├── layout_parser.py    # 左/右/居中布局分析
│   ├── incremental.py      # 增量检测、去重
│   ├── input_sim.py        # 点击、剪贴板、键盘
│   └── db_helper.py        # SQLite 读写（inbox/messages）
├── readers/
│   ├── __init__.py
│   ├── wechat_reader.py
│   └── qianniu_reader.py
├── writers/
│   ├── __init__.py
│   ├── wechat_writer.py
│   └── qianniu_writer.py
├── config/
│   ├── wechat_config.json
│   └── qianniu_config.json
├── main.py                 # 入口：按配置启动 reader + writer
└── requirements.txt
```

`db_helper.py` 可以统一封装对 `rpa_inbox_messages` 和 `messages` 的访问，避免 Reader/Writer 各自直接写 SQL。

## 6. 实施顺序建议

1. **common/**：先实现 `screenshot`、`ocr_engine`、`layout_parser`、`incremental`、`db_helper`
2. **wechat_reader**：基于 common 实现，写入 `rpa_inbox_messages`
3. **wechat_writer**：轮询 `messages(sync_status=10)`，执行模拟发送
4. **千牛**：复用 common，实现 `qianniu_reader` + `qianniu_writer`
