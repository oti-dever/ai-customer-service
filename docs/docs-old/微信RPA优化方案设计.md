# 微信 RPA 优化方案设计

本文与 [微信RPA链路耗时分析与优化方向](./微信RPA链路耗时分析与优化方向.md)、[微信RPA实现方案](./微信RPA实现方案.md)、[微信RPA改造方案设计与实现](./微信RPA改造方案设计与实现.md)、[微信独立工作台方案设计与实现](./微信独立工作台方案设计与实现.md) 配套：在**不推翻现有主路径**的前提下，记录微信 PC 端自动化的**约束认知**、**可选技术路线**与**分阶段优化方向**，供评审与落地时裁剪。

---

## 1. 背景与当前主路径

- **主路径（仓库现状）**：以 **窗口句柄 + 屏幕区域截图 + PaddleOCR** 为主读取聊天内容；**会话切换**依赖红点扫描、会话列表 OCR、必要时搜索框路由；与 Qt 客户端通过 **SQLite（`rpa_inbox_messages` 等）** 协同。
- **目标**：在合规与安全前提下，降低**端到端延迟**、提高**稳定性**（少受 DPI/主题/版本 UI 漂移影响），并控制**维护成本**。

### 1.1 版本策略：优先微信 4.1 及以上

- **产品定位**：当前主流用户已普遍使用 **微信 PC 4.1+**（含 4.0 后期演进线），方案与测试资源**优先对齐该版本区间**。
- **兼容性策略**：**不主动承诺**对 3.9 时代的专用自动化栈（如旧版仅适配 3.x 的库）做长期并行维护；若商业上仍需覆盖旧版，单独评估成本与分支策略。
- **与实现方案文档的关系**：实现方案中已记录 **4.0.5+ / 4.1 起 UIA 遍历困难** 的实测结论；本设计在**不推翻 PaddleOCR 主路径**前提下，讨论 **4.1+ 下的混合增强**（见下节「可借鉴能力」）。

### 1.2 可借鉴的外部思路（多后端 / 坐标 / 快捷键 / 搜索框等）

开源项目（如 [pywechat](https://github.com/Hello-Mr-Crab/pywechat)）的常见做法**不等于**「依赖完整 UIA 树」，很多流程与 **Inspect 看不到内部控件** 并不矛盾。下列能力可与现有 **红点 + 会话列表 OCR + 搜索框回退** 对照，作为 **4.1+ 上的增强或降级路径**（需 POC 与版本锁定）：

| 思路 | 说明 | 与我方结合方式（设计级） |
|------|------|--------------------------|
| **多后端** | pywinauto 除 UIA 外还有 **win32** 等；UIA 浅树时仍可能操作 **HWND、客户区消息** | 与「主窗 HWND + 校准 ROI」一致；失败时明确降级到坐标/键鼠，而非假设能枚举子控件 |
| **相对坐标** | 不依赖控件名，按**窗口客户区**比例或像素点击 | 已有 `wechat_config.json` 中比例/像素字段；可统一**坐标换算与 DPI**，减少漂移 |
| **快捷键** | 全局/窗口级快捷键打开搜索、切换模块 | 与现有搜索路由协同；注意与 **Reader/Writer 窗口锁**、前后台焦点冲突 |
| **搜索框流程** | Ctrl+F / 搜索 → 选第一条结果等 | 已作为列表切换失败后的路径；可借鉴其**时序与 settle**，降低对「全列表 OCR」的依赖 |
| **其它** | 监听、独立会话窗口等 | 若 4.1+ 上可稳定复现再纳入；以**线程模型**（单线程操微信 UI）为约束 |

**原则**：上述能力作为 **并联模块或策略链中的一环**，**读聊天正文**在 UIA 不可靠时仍以 **截图 + PaddleOCR + 布局** 为主；避免「整库替换」带来的版本与许可风险。

### 1.3 与 pyweixin POC 结论对齐（资源优先级）

对 `pyweixin` / `pywechat` 仓库的本地 POC 验证结论（强依赖 `mmui::` 与控件树、与 4.1.8 实测偏差、不宜作为主依赖等）已**集中写入**《[微信RPA实现方案](./微信RPA实现方案.md)》**§1「补充：pyweixin POC」**。  
**工程优先级**：在合规与安全前提下，**持续打磨截图 + OCR + 自有策略链**优于长期追逐上游 UIA 选择器；微信版本为**不可控外生变量**，主路径应以**视觉与配置化 ROI** 吸收漂移，开源 UIA 方案仅作**可选并联/对照**。

## 2. Inspect 下「进不了微信窗口内部」的原因说明

你在 **Microsoft Inspect** 中观察到：顶层「微信」窗口下往往只有少量子节点（例如 **「Weixin」窗格**、**`MMUIRenderSubWindowHW` 窗格**），**无法再展开出**聊天列表、输入框、消息气泡等细粒度控件。这与下列因素一致（不同微信版本会略有差异）：

### 2.1 主界面大量为自绘 / 专用渲染，而非标准 Win32 控件树

- 常见现象：主客户区由 **Qt**（截图中 `ClassName` 如 `Qt51514QWindowIcon` 等）与 **自绘 / 合成层**（如 `MMUIRenderSubWindowHW` 一类子窗）承载，**内部交互元素未必向系统注册为可枚举的 UIA 子节点**。
- 结果：**UI Automation 的「控件树」在 Inspect 里表现为「大块 Pane + 无子级」**，并非你操作有误，而是**可访问性视图本身不完整**。

### 2.2 未暴露或仅暴露有限模式

- 若各节点 **Text / Invoke / Scroll** 等 Pattern 多为 **不可用**，自动化工具**无法**像操作传统 WinForm/WPF 那样按「控件名/类型」稳定读写。

### 2.3 与「能否自动化」不是一回事

- **看不到树 ≠ 绝对不能控窗口**：仍可对**顶层或子 HWND** 做 **坐标点击、键盘输入、部分 Win32 消息**；也可对**像素区域**做 **OCR/图像匹配**（即当前项目路线）。
- **看不到树 = 很难做「纯 UIA 树驱动」的全套 RPA**：除非应用或某版本**补充暴露**可访问性，或换**非 UIA**手段。

## 3. 「pywechat 为什么能做到」——需在相同环境下验证的几种可能

开源项目 [Hello-Mr-Crab/pywechat](https://github.com/Hello-Mr-Crab/pywechat) 文档写明基于 **pywinauto**、**UI Automation** 等，并声明**不涉及逆向 Hook**。若在你当前 **微信版本 + Inspect 截图** 下，**UIA 树同样很浅**，则「能做到」通常可能来自以下几类机制（**彼此不互斥，需 POC 验证**）：

| 可能机制 | 说明 |
|----------|------|
| **多后端与降级策略** | pywinauto 除 **UIA** 外还有 **win32** 等后端；部分逻辑可能走 **HWND / 坐标 / 键鼠序列**，并不依赖完整控件树。 |
| **版本与分支差异** | **3.x / 4.x**、安装渠道不同，**可访问性暴露程度**可能不同；文档中的示例可能在**与你不完全一致的版本**上录制。 |
| **不读「内部树」也能完成部分功能** | 例如依赖 **全局快捷键、搜索框、固定布局点击** 完成「打开会话 / 发送」，对树依赖较弱；**监听类**能力若依赖独立子窗口或导出接口，路径也可能与主窗 Inspect 所见不同。 |
| **实现与宣传口径偏差** | 实际代码可能混合 **图像/坐标** 等辅助手段；应以 **依赖与关键调用** 为准，而非仅看 README 一句话。 |
| **维护成本** | 微信更新频繁，**UIA 选择器漂移**与 **OCR 漂移**一样是长期风险，需同等看待。 |

**结论（设计原则）**：  
- 在**你方目标微信版本**上，用 **pywechat/pyweixin 做最小 POC**（打开会话、读一条消息、监听回调），**与 Inspect 现象交叉验证**；  
- **若 POC 证明 UIA 仍不可用**，则不宜把「纯 pywechat 树驱动」作为唯一依赖，而应 **与现有截图/OCR 路线并列或混合**。

## 4. 优化方向总览（与现有文档对齐）

以下与《微信RPA链路耗时分析与优化方向》中的条目一致，此处从**方案设计**角度归纳优先级。

### 4.1 低成本、见效快（优先）

- **Reader 节拍与队列**：`poll_interval_sec`、未读队列是否批处理、冷却时间等。
- **OCR 与 ROI**：`max_side`、`chat_region`、列表全量刷新频率、标题 OCR 降频；生产关闭重调试截图。
- **C++ inbox 消费**：已实现微信适配器 **空闲/有流量动态轮询**（见 `wechatrp_adapter.cpp`），可按负载微调间隔。

### 4.2 中等成本（结构性）

- **Python 与 Qt 之间**：事件通知（减少纯 DB 轮询）、批量提交与连接策略（注意锁与一致性）。
- **会话切换长尾**：减少「搜索框回退」概率（校准、行高、重试），缩短 settle 的实测调参。

### 4.3 路线级（需单独评审，面向 4.1+）

- **混合架构（推荐演进方向）**：在 **4.1+** 上，「发现会话 / 切会话」优先尝试 **坐标 / 快捷键 / 搜索框** 等与 **Inspect 无关** 的手段；「读聊天内容」仍以 **OCR** 为主，直至 POC 证明有更低延迟且合规的替代。
- **UIA / pyweixin 类库**：仅当 POC 在**目标微信版本**上证明**稳定且收益明确**时，再考虑并联；**必须**解决与 **Reader/Writer 多线程**、**窗口锁** 的串行约束，且**不得**假设存在完整控件树。
- **版本锁定**：对外发布时声明**已验证的微信主版本/小版本区间**，大版本升级后重复 Inspect + POC。

## 5. 最小 POC（概念验证）怎么做

POC 的目的：**在与你目标用户一致的环境（优先微信 4.1+）下，用最小成本验证**「外部自动化库 / 多后端 / 坐标键鼠」**是否比「仅依赖 Inspect 能看到的 UIA 树」更可行**，并量化**成功率与耗时**。  
**不要求**首轮就接入本仓库主分支：可在**独立虚拟环境 + 独立脚本目录**完成，避免污染现有 `python/rpa` 依赖。

### 5.1 边界说明（POC 做什么、不做什么）

| 做 | 不做 |
|----|------|
| 验证 **能否** 打开指定会话、拿到可见文本或监听回调（按所选库的 API） | 不要求与 `rpa_inbox_messages`、Qt 联调 |
| 记录 **微信精确版本号、系统 DPI、单/多显示器** | 不要求打包安装器、不要求覆盖全部会话类型 |
| 与 **Inspect 基线**对照，判断依赖的是树还是坐标/快捷键 | 不假设 POC 代码可直接合并进生产 |

### 5.2 环境准备（必须记录）

1. **微信 PC**：升级或固定到 **4.1.x（或你拟支持的最低 4.1 小版本）**，在「设置 → 关于微信」记下 **完整版本号**。团队当前 POC 自用机试点版本：**4.1.8.29**；模块选用 **pyweixin**（见 `poc/pywechat-poc/README.md`）。  
2. **操作系统**：Windows 10/11 及**显示缩放**（100%/125%/150%），POC 机与日后用户环境尽量接近。  
3. **Python**：单独 **venv**（建议与主项目 `python/` 分离，例如 `poc/pyweixin-venv/`），避免与 Paddle 等依赖冲突。  
4. **微信状态**：主窗口已登录、**勿最小化到托盘**（与现网 Reader 假设一致）；测试账号仅用**可控的测试好友/群**，避免骚扰真实用户。

### 5.3 基线 A：Inspect（5 分钟内完成）

- 打开 **Microsoft Inspect**，选中微信主窗口及子节点，**截图或文字记录**：能否展开子控件、`ClassName`、`FrameworkId`、主要 Pattern 是否可用。  
- 目的：与下文「库能否跑通」**交叉对照**——若树很浅但库仍能自动化，说明对方**多半不依赖完整树**。

### 5.4 基线 B：本仓库现有链路（可选，用于对比）

- 在 `python/` 下按现有文档**仅启动 Reader**（或完整 `rpa.main --platform wechat`），用**同一微信窗口**收一条测试消息，记录从发送到 **控制台出现「写入 inbox」类日志** 的**大致秒数**（与链路分析文档方法一致）。  
- 目的：POC 若引入「切会话」新手段，将来可对比 **端到端是否变快**。

### 5.5 试验：外部项目（以 pywechat / pyweixin 为例，4.1+）

以下步骤**随上游仓库更新可能变化**，以 [Hello-Mr-Crab/pywechat](https://github.com/Hello-Mr-Crab/pywechat) 的 **README / Weixin4.0.md** 为准。

1. **获取代码**：`git clone` 到 POC 目录（不要与业务仓库子模块混用，除非刻意 submodule）。  
2. **安装依赖**：在独立 venv 中按对方 `requirements.txt` 安装；若提供 **pip 包名**（如文档中的 `pywechat127`），优先用**与 4.1 匹配**的分支说明。  
3. **最小脚本（逻辑步骤，非强制代码）**：  
   - **仅导入**：`from pyweixin import Navigator`（或文档当前推荐入口），执行 **打开微信 / 聚焦主窗**（若 API 要求）。  
   - **单会话**：对**一个测试联系人**执行文档中的 **「打开独立聊天窗口」或「搜索并进入会话」** 的最短示例。  
   - **读文本**：调用文档中的 **监听 / 读消息** 的最小 API（如 `Monitor.listen_on_chat` 类），持续时间 **30s～1min**，向该会话发 1～3 条已知内容。  
4. **观察**：控制台是否报错；返回结构中**文本是否与发送一致**；是否偶现卡死、焦点被抢（与「单线程操微信」约束是否冲突）。  
5. **重复次数**：同一脚本 **连续跑 5 次**（可隔天再跑 5 次），记录**成功次数 / 失败类型**（找不到窗口、超时、文本为空等）。

### 5.6 试验：纯 pywinauto（可选，用于归因）

若需区分「库魔法」与「通用 win32/uia」：

- 在同一 venv 安装 `pywinauto`，用 **Application(backend="uia")** 与 **backend="win32"** 分别连接微信主窗，打印**子控件数量与深度**（仅打印，不做业务）。  
- 若 **uia 极浅、win32 略多**，说明后续若自研脚本，可能更多依赖 **HWND + 坐标** 而非树。

### 5.7 与主工程集成（仅当 5.5 通过「通过标准」后）

- 在 `python/rpa` **新建模块占位**（例如 `poc_bridge_wechat.py`）或独立分支，只做 **一次**「调用 POC 函数 → 写 stdout / 临时表」，**仍不默认启用**。  
- 与 **Reader 的 `window_lock`、线程模型**对齐：所有 pywinauto/ pyweixin 调用放在**同一线程**或与 Reader 相同的锁内（见实现方案中的锁设计）。  
- **回退策略**：POC API 失败时 **无条件回退** 到现有 OCR 路径，避免双失败。

### 5.8 记录模板（建议复制到会议纪要或 issue）

| 项目 | 填写示例 |
|------|----------|
| 日期 | |
| 微信版本 | 如 4.1.x.x |
| OS / 缩放 | Win11，125% |
| POC 类型 | pyweixin 最小监听 / pywinauto 探测 |
| 依赖版本 | pyweixin / pywechat / pywinauto 版本号 |
| 基线 Inspect | 树深度：仅 2 层 Pane / 其它 |
| 5 次运行结果 | 成功 4/5；失败 1：超时 |
| 单次耗时量级 | 打开会话 ~Xs；读到首条 ~Ys |
| 结论 | 可并联 / 仅作参考 / 放弃 |

### 5.9 通过 / 暂缓标准（建议）

- **建议进入「小规模工程化」**：连续两天内、**≥8/10 次**成功完成「打开测试会话 + 读到已知文本」；失败原因可解释且可规避（如必须先点主窗）。  
- **建议暂缓并联**：成功率 **低于 50%**，或强依赖**固定分辨率/独占机器**；或引入后**与现有 Reader 并发必现死锁**。  
- **法律与合规**：仅使用对方 **License 允许** 的集成方式；禁止把 POC 用于骚扰、爬取非授权数据。

### 5.10 借鉴 pywechat 仓库做 POC（4.1+ 落地讨论）

以下把「讨论」落成**可执行清单**：优先 **微信 4.1+**、优先 **pyweixin**（与仓库内偏 3.9 的 **pywechat** 老入口区分），避免选错文档与包名。

#### 仓库内两个名字，不要混用

| 模块/名字 | 典型适用 | POC 建议 |
|-----------|----------|----------|
| **pyweixin** | README 中面向 **4.1+** 的 API（`Navigator`、`Monitor` 等） | **主 POC 只跟这一条线**，示例多在 README「pyweixin」小节 |
| **pywechat**（包内旧模块） / **pip pywechat127** | 偏 **3.9+**、部分环境仅 32 位等限制 | 若你明确不做 3.9，可**暂不安装**，减少变量 |

上游以 [Hello-Mr-Crab/pywechat](https://github.com/Hello-Mr-Crab/pywechat) 的 **README、`Weixin4.0.md`、`requirements.txt`** 为准；若与下表冲突，**以上游为准**。

#### 建议目录（与主工程 `python/rpa` 隔离）

在本仓库根下使用独立目录，避免与 Paddle、主依赖混装（仓库内已提供占位说明：`poc/pywechat-poc/README.md`）。

```
poc/pywechat-poc/
  README.md          # 指向本文 §5.10
  setup.ps1          # 创建 venv + pip install git+https://github.com/Hello-Mr-Crab/pywechat.git
  smoke_poc.py       # 最小 Navigator + Monitor 试验
  upstream/          # 可选：git clone，仅便于读 Weixin4.0.md / 源码
  venv/                # 本地虚拟环境（勿提交）
```

#### 落地步骤（建议严格按序）

1. **安装 POC 环境**：在 `poc/pywechat-poc` 执行 **`.\setup.ps1`**（或手动：`py -3 -m venv venv`，再 `pip install git+https://github.com/Hello-Mr-Crab/pywechat.git`）。安装包名为 **pywechat127**，内含 **pyweixin**，**无需**先 git clone 即可运行。
2. **（可选）读上游文档**：若已 `git clone` 到 `upstream/`，再读 `Weixin4.0.md`、核对 API；与 **`smoke_poc.py`** 对照即可。
3. **试验零（导入自检）**：`.\venv\Scripts\python.exe -c "from pyweixin import Navigator, Monitor"` 应无报错。
4. **试验一（最小闭环，不读消息）**  
   - 目标：仅用 **pyweixin** 的 **`Navigator.open_seperate_dialog_window`** 打开**一个固定昵称**的会话（「文件传输助手」或专用测试号）。可直接运行仓库内 **`smoke_poc.py --no-listen --friend "…"`**。  
   - 成功判据：肉眼可见**已进入该会话**（主窗切换或独立聊天窗出现，以文档行为为准）。  
   - **单线程执行**，不要一上来就用 README 里多线程监听示例。  
5. **试验二（读可见文本）**  
   - 在试验一稳定后，调用 **`Monitor.listen_on_chat`**；或使用 **`smoke_poc.py --friend "…" --duration 30s`**，向该会话发送 **1～3 条固定字符串**。  
   - 检查返回值中是否**包含**这些字符串；记录**首条文本出现耗时**、是否抢焦点、是否需微信前台。  
6. **试验三（归因，可选）**  
   - 在 `upstream` 源码中 **grep** `backend`、`Application`、`uia`、`win32`，记录 pyweixin **默认如何连接微信窗口**；对照 §2 Inspect 结论，判断其是否**主要不依赖深层 UIA 树**。  
7. **对照本仓库（为将来并联准备）**  

| pywechat / pyweixin 能力 | 我方现网（概念） | POC 要回答的问题 |
|--------------------------|------------------|------------------|
| 打开/切换会话 | `wechat_session` 列表点击、搜索兜底 | 对方路径是否更稳或更省 OCR |
| `listen_on_chat` 拿文本 | `wechat_reader` 聊天区 OCR | 能否减少或校验 OCR（或仅作辅证） |
| `scan_for_newMessages` 等 | 红点 + 列表区域 OCR | 能否减少全列表 OCR 频率 |

#### 可借鉴「思想」、谨慎「搬运」

- **宜借鉴**：**时序与 settle**（搜索→点击→等待）、**快捷键顺序**、**失败后的回退顺序**（与你们 `list_switch_settle_sec`、搜索路由对照调参）。  
- **宜对照读源码**：`utils` 里与**新消息数、会话扫描**相关的函数名（不必引入依赖，只确认**能否**在你们 ROI 策略下复现逻辑）。  
- **谨慎**：整文件复制需符合对方 **LICENSE**；生产代码更宜「自研实现 + 流程对齐」，避免许可证与维护双风险。

#### 与本仓库 Reader/Writer 并跑时的约束（POC 通过后工程化前牢记）

- 微信侧操作尽量 **单线程串行**；若并入主工程，须与现有 **`window_lock`** 同一套互斥，避免与 `wechat_reader` / `wechat_writer` 同时抢窗口。  
- 若 pyweixin 依赖**独立聊天子窗口**，而现网假设**单主窗 ROI**，需在方案上二选一或做**焦点/窗口状态**统一。

## 6. 建议的推进顺序

1. **基线**：固定微信版本号 + 当前配置，用现有链路测 **P50/P95 端到端耗时**（见链路分析文档中的测量建议）。  
2. **最小 POC**：按 **§5** 完成 Inspect 记录 + 外部库或 pywinauto 试验，填写记录模板并做通过/暂缓判断。  
3. **决策**：  
   - POC 达到 §5.9「建议进入工程化」→ 在《实现方案》中增「**并联模块**」设计（接口、回退到 OCR）。  
   - 未达标 → 收敛资源到 **OCR 链路与 DB/轮询优化**，不强行并联外部 UI 库。  
4. **文档维护**：任何微信大版本升级后，重复 **Inspect + §5 最小 POC** 快照，更新版本结论。
