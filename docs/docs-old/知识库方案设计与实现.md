# 千牛店铺「知识库」方案设计与实现

本文档描述在 **主窗口侧栏「管理后台」** 下新增 **「知识库管理」** 能力的设计定案与实现要点：**支持用户导入店铺话术与说明类文档**，经 **本地处理与可选 AI 加工** 后 **持久化保存**，并在 **聚合接待的 AI 辅助 / AI 自动回复** 链路中，将 **与当前客户问题相关的知识片段** 注入模型请求，替代或增强程序内 **固定短 system 话术**。

**首版范围**：**仅针对千牛店铺维度**（与当前聚合会话 `platform = qianniu`、店铺/账号隔离策略对齐）；其他平台可后续按同一数据模型扩展。

**文件名**：与标题一致，`知识库方案设计与实现.md`。

**关联文档**：

| 文档 | 关系 |
|------|------|
| 《AI辅助回复方案设计与实现.md》 | 聚合 **「生成本条回复」**；知识注入后仍 **仅写输入框、不自动发送**。 |
| 《AI自动回复方案设计与实现.md》 | 千牛 **自动生成并发送**；知识注入与 **§5.2 生成层** 衔接，**责任与风控** 仍以该文档为准。 |
| 《千牛RPA优化方案设计与实现.md》 | 客户侧 **文本 / 聊天区截图** 入站；知识库多为 **文本片段**，与 **多模态 user 消息** 并行拼装。 |
| 《AI助手集成方案设计与实现.md》 | `OpenAiCompatClient`、豆包线路；**内置帮助助手不读取客户会话**，与 **店铺知识库** 数据隔离。 |
| [火山引擎 · 文件输入（File API）](https://www.volcengine.com/docs/82379/1885708?lang=zh) | 方舟 **Files API** 与 **Responses API** 联用；**本文 §5.4** 摘要。 |

## 1. 背景与目标

### 1.1 背景

- 各 **店铺** 的 **商品、政策、话术** 差异大，仅靠程序内置的 **「客服话术 + 产品说明」** 固定字符串，无法覆盖真实接待需求，且 **发版成本高**。
- 已接入的 **豆包（火山方舟）** 等模型支持 **文本、图片** 作为对话多模态输入；**客户消息侧** 已具备 **千牛截图 + 文本** 的组装经验（见《AI辅助回复方案设计与实现》）。
- **文档文件（PDF 等）** 与 **对话多模态「图片 URL」** 不是同一概念：**PDF 可先本地解析为文本**，也可在 **火山方舟** 侧经 **Files API 上传并预处理**（官方对 PDF 会 **分页成图** 再供模型理解，见 **§5.4**）。设计上 **不将任一路径作为唯一依赖**：**本地文本 + RAG** 与 **Files API + Responses API** 可按场景 **并存或分阶段采用**。

### 1.2 目标（分层）

| 层级 | 目标 |
|------|------|
| **产品** | 在 **「管理后台」分组** 下增加 **「知识库管理」** 入口；用户可为 **当前千牛店铺（或绑定 scope，见 §4.1）** 维护一份 **可更新的知识素材**。 |
| **数据** | 支持 **导入** 话术类文档（**阶段化**：先 **纯文本 / Markdown**，再 **PDF 等**）；内容 **仅存本地**（路径与格式见 §6）。 |
| **智能** | 支持对素材做 **自动分析 / 摘要 / 结构化**（可选，由 **已配置的豆包或其他线路** 完成），产出 **便于检索与拼 prompt 的单元**（chunk、摘要、标签等，见 §5）。 |
| **消费** | 在 **AI 辅助** 与 **AI 自动回复** 调用模型前，将 **与客户当前入站相关的知识片段** 写入 **system 或固定 user 前置块**（与现有 `aggregateAiMvpSystemPrompt` 类逻辑 **合并或替换**，见 §7）。 |

### 1.3 设计原则

- **店铺隔离**：知识 **必须** 与 **店铺/租户维度** 绑定，避免 **A 店知识回答 B 店客户**。
- **本地优先**：默认 **原文与派生数据落本地**；外发仅 **调用模型 API** 时传输 **必要片段**（需在设置中延续 **隐私提示** 表述）。
- **分阶段交付**：**不**在第一版同时承诺 **全格式 PDF 完美版式**、**向量检索**、**多语言 OCR**；每一阶段 **可验收、可回滚**。
- **可解释**：后台可展示 **「本次回复参考了哪些片段」**（至少开发期日志；产品期可简化为「已引用知识库」开关提示）。

## 2. 范围与非目标

### 2.1 范围内（按阶段滚动纳入）

| 阶段 | 内容 |
|------|------|
| **MVP** | 侧栏入口 **「知识库管理」**；**手动粘贴 / 导入 `.txt`、`.md`**；**本地存储**；生成侧 **「全文或单文件摘要 + 长度截断」** 注入 prompt（**不做向量检索**），与千牛聚合 **AI 辅助 / 自动回复** 打通 **一条** 注入路径。 |
| **P1** | **PDF** 导入：依赖 **成熟 PDF 文本抽取**（无文字层时 **OCR 可选/降级**）；**分块（chunk）** 与元数据（文件名、导入时间）。 |
| **P2** | **检索**：**关键词 / BM25** 或 **向量检索（RAG）**（需 **嵌入模型** 与 **向量索引**，嵌入可本地或 API）；**仅 Top-K 片段** 注入，控制 token。 |
| **P3** | **多文档版本**、**生效时间**、**类目标签**；与 **自动回复** 的 **频率限制、熔断** 策略联动（见《AI自动回复方案设计与实现》§6、§7）。 |

### 2.2 非目标（本文档阶段不承诺）

- **实时协同编辑**、**云端团队知识库**（除非另立同步与账号体系）。
- **自动从千牛后台爬取商品详情**（可与 RPA 采集 **分列项**，避免与知识库混为一谈）。
- **以「方舟 Files API 为唯一知识来源路径」**：Files API 已正式发布（见 **§5.4**），但 **知识库仍宜保留本地副本与检索**（合规、离线、成本控制）；**「仅上传、不落地」** 可作为部分场景的简化选项，而非全局替代。
- **在未完成联调前承诺全格式生产可用**：大文件、长视频、复杂 PDF 的 **预处理超时（官方说明 5 min 级）** 与 **存储有效期** 需实测后再写死 SLA。

## 3. 业务概念

### 3.1 知识单元（建议数据语义）

| 概念 | 说明 |
|------|------|
| **知识库（KB）** | 隶属于某一 **店铺 scope**（见 §4.1）的逻辑容器；MVP 可 **一对一**「一店一库」。 |
| **文档（Document）** | 一次导入产生一条记录：源文件名、类型、存储路径、导入时间、处理状态（已解析 / 失败原因）。 |
| **片段（Chunk）** | 用于注入模型或检索的 **文本块**；MVP 可 **整文件即一块**；P1 起 **固定大小或语义分块**。 |
| **派生（Derived）** | AI 生成的 **摘要、关键词、问答对草稿** 等，**依附于文档或 Chunk**，持久化便于调试与复用。 |

### 3.2 与客户消息的关系

- **客户文本 / 千牛截图**：仍由 **入站消息模型** 提供（见《千牛RPA优化方案设计与实现》）。
- **知识库**：提供 **店铺事实与话术依据**；拼装时 **典型结构** 为：  
  **system** = 角色与安全约束 + **检索到的知识片段**；  
  **user** = 客户文字 +（可选）**图片多模态块**。  
  **不宜** 用单张客户截图替代 **PDF 说明书** 的细粒度文字（除非产品明确走「视觉问答」路径）。

## 4. 店铺维度与隔离（千牛）

### 4.1 Scope 定义（实现期须定案）

候选绑定键（择一或组合）：

- 当前登录 **客服账号**（与 `UserDao` / 会话归属一致）；
- **千牛店铺标识**（若 RPA/配置中已有稳定 `shop_id` 或等价字段）；
- **本软件内「店铺配置」主键**（若存在）。

**要求**：同一 scope 下 **知识库读写一致**；切换店铺或账号时 **UI 展示对应知识库**，且 **聚合生成** 时 **只拉当前 scope**。

### 4.2 与「仅千牛」的关系

- **知识库管理入口** 可对 **所有用户可见**，但 **「应用到聚合生成」** 在首版 **仅当会话 `platform == qianniu`** 时注入（与《AI自动回复方案设计与实现》范围一致）；其他平台会话 **可忽略知识库或提示未开放**（产品定）。

## 5. 文档处理管线

### 5.1 导入与解析

| 格式 | MVP | P1 及以后 |
|------|-----|-----------|
| `.txt` / `.md` | **UTF-8 读取**，失败时 BOM/编码探测 | 同左 |
| PDF | **不支持或提示「敬请期待」** | **文本层抽取**；扫描件走 **OCR**（可选依赖 Tesseract 等，另立依赖说明） |

### 5.2 分块与摘要

- **MVP**：单文件总长度 **低于阈值** 时 **整体注入**；超过则 **头部截断 + 告警** 或 **仅调用一次模型做摘要** 再注入（摘要长度上限配置化）。
- **P1+**：**固定字符 / 段落分块**，块间 **overlap** 可配置；每块可选 **离线摘要**（费用与耗时需提示）。
- **摘要的定位**：**导航与二次检索** 有价值；**不宜** 作为唯一依据替代全文检索（易丢细节）。

### 5.3 检索策略（P2）

- **关键词 / BM25**：实现轻量、无嵌入依赖；适合 **专有名词**。
- **向量 RAG**：**chunk → embedding → 向量库**；查询 = **客户当前入站文本**（+ 可选 **会话最近几句**）→ **相似度 Top-K** → 注入 prompt。
- **混合检索**：关键词 + 向量 **重排**（产品后期）。

### 5.4 火山方舟 Files API（文件输入）摘要

以下依据火山引擎《文件输入（File API）》公开说明整理，**细节与配额以控制台与最新文档为准**；在线文档：  
[https://www.volcengine.com/docs/82379/1885708?lang=zh](https://www.volcengine.com/docs/82379/1885708?lang=zh)

| 要点 | 说明 |
|------|------|
| **定位** | **Files API** 提供文件 **上传、按 ID 查询、列表、删除**；与 **Responses API** 组合，用于 **大文件 / 重复引用文件** 的多模态理解（避免每次请求重复传完整二进制）。 |
| **上传端点** | 与方舟 OpenAPI 同基址，例如 **`POST .../api/v3/files`**，`purpose=user_data`，**multipart 上传**；成功返回 **`file_id`**、**`mime_type`**、**`status`** 等。 |
| **与推理的关系** | 官方说明：在 **Responses API** 请求中，以 **`input_file`**（及视频等类型的 `input_video` 等）引用 **`file_id`**，并配合 **`input_text`** 给出用户问题；**上传后需待 `status` 为 `active`** 方可用于推理（处理中需轮询或 SDK `wait_for_processing` 一类能力）。 |
| **容量与保留** | 公开材料载明：**单文件最大约 512 MB**、**总存储约 20 GB**；**默认保留约 7 天**，可通过 **`expire_at` 在约 1～30 天范围内调整**；高频场景建议 **缩短保留 + 主动删除**。 |
| **预处理** | **视频**：可按 **`preprocess_configs.video.fps`** 等抽帧；**PDF**：官方说明为 **分页转成多图** 再供模型，**预处理不对拆分图做分辨率缩放** 以保留信息；**预处理超时** 官方提示约 **5 min 级**，大像素视频/长 PDF 可能触发，文档给出压缩与排查建议。 |
| **支持类型（摘录）** | 含 **图片**（多类 MIME）、**视频**（如 mp4/avi/mov）、**PDF**（`application/pdf`）等，详见官方表格。 |

**与本文「知识库」的关系**：

- **路径 A（本地为主）**：导入文件 **落盘** → 本地分块/摘要/检索 → Chat Completions 注入文本——**不依赖 Files API**，与现有 `OpenAiCompatClient` 一致。
- **路径 B（方舟预处理）**：用户文件经 **Files API** 上传 → **`file_id`** → **Responses API** 做 **文档理解** → 再将会话结果 **写回本地索引或仅展示**——适合 **省本地解析开发量**，但依赖 **外网、额度与保留策略**。
- **工程注意**：当前工程 **内置助手与聚合** 广泛使用的是 **Chat Completions**（`/chat/completions`）封装；**Files API 官方示例与 File ID 消费主要在 Responses API（`/responses`）**。接入 Files API **通常需新增 Responses 客户端（或扩展现有网络层）**，与「仅改 prompt 文本」工作量不同，宜 **先在低风险界面验证**（见 **§5.6**）。

### 5.5 与现有 `OpenAiCompatClient` 的衔接（评估）

- **现状**：`OpenAiCompatClient` 面向 **流式 Chat Completions**，请求体为 **messages 数组**，与 **Responses + `input_file`** 的 JSON 结构 **不同**。
- **建议**：将 **「方舟文件上传 + Responses 调用」** 作为 **独立模块**（如 `ArkFilesClient` + `ArkResponsesClient` 或合并为 **VolcengineArkMultimodalChannel**），与 **`OpenAiCompatClient` 并行**；待协议稳定后再考虑 **内部复用 QNetworkAccessManager / 鉴权**。
- **知识库产品阶段**：MVP 仍可 **只做路径 A**；路径 B 在 **内置助手验证通过** 后，再评估是否用于 **知识库「一键上传摘要」** 或 **聚合侧**。

### 5.6 内置 AI 助手对话页：文件能力先行验证（占位已就绪）

**产品动机**：在 **不接触千牛客户会话数据** 的前提下，先跑通 **选文件 → 方舟上传 →（可选）等待处理完成 → Responses 引用 `file_id` → 流式输出**，降低 **聚合 / 知识库** 首接时的联调风险。

**界面现状（与工程一致）**：

- **入口**：**机器人管理** 内 **内置 AI 助手** 对话页，输入框 **左上方** 与「添加图片」并列的 **文件夹形按钮**（`RobotAssistantWidget::m_fileBtn`）。
- **当前行为**：工具提示为 **「添加文件（即将支持）」**；点击走 **`onPickFilePlaceholder`**，**仅占位**，尚未接 Files API。

**建议实施顺序**：

1. **仅豆包（方舟）预设**：复用 **同一 `base_url`（如 `https://ark.cn-beijing.volces.com/api/v3`）与 API Key**。
2. **实现文件选择**（`QFileDialog`）→ **multipart 调用 Files API** → 解析 **`file_id`**，轮询或使用 SDK 直至 **`active`**（失败则提示预处理超时或格式不支持）。
3. **发起 Responses API**（流式与非流式按官方示例），将 **用户输入文字 + `input_file`** 一并发送；助手气泡展示结果。
4. **会话结束后**：可按需 **调用删除文件** 以节省 **20 GB 配额**；或在 **expire 前** 依赖自动过期。
5. **验收通过** 后，再将 **同一套上传/引用封装** 抽到 **公共库**，供 **知识库管理**（例如：上传店铺 PDF → 摘要入库）与 **后续聚合** 按需引用。

**数据边界**：内置助手对话 **仍不读取客户会话**；此处验证的是 **「本机用户选中的文件」** 与方舟之间的链路，与 **§4 店铺知识库 scope** 无关——后续知识库接入时需 **显式绑定 scope 与文件元数据**。

## 6. 本地存储（建议）

### 6.1 存放位置

- **用户数据目录** 下子目录，例如 `knowledge_base/<scopeId>/...`（具体名与 `QStandardPaths` 对齐）。
- **原文档**：保留用户上传副本（便于重新解析）。
- **索引与派生**：SQLite 表 或 **侧车 JSON**（MVP 可 **单表 documents + chunks**）。

### 6.2 元数据字段（示例）

- `id`, `scope_id`, `source_filename`, `mime`, `imported_at`, `status`, `error_message`, `content_hash`（去重可选）。

### 6.3 备份与删除

- 提供 **「删除文档 / 清空知识库」**；**物理删文件 + 删索引行**。
- 备份策略：**随本软件整体用户数据备份**（文档不另述）。

## 7. 与聚合 AI 链路的集成

### 7.1 注入点

- 在 **`buildAggregateAiCompletionRequest`**（或后续抽出的 **AggregateAiContextBuilder**）中：
  - 读取 **当前 scope** 的知识内容或 **检索结果**；
  - 与现有 **`aggregateAiMvpSystemPrompt()`** 合并：  
    **建议**：**安全与角色约束** 仍放 **system**；**店铺事实** 以 **「【店铺知识】」** 块置于 **system 尾部** 或 **单独 user 前置消息**（以 **模型表现与 token 计数** 实测为准）。

### 7.2 Token 与费用

- **硬上限**：注入总字数 / 估算 token **上限**（配置项），超出 **截断** 并 **日志**。
- **RAG 生效后**：以 **K 个 chunk** 为主控，而非全文。

### 7.3 日志与调试

- 记录 **scope、文档 id、chunk id（若有）、注入长度**；**不**在默认日志打印 **全文客户消息**（脱敏策略与现有聚合日志对齐）。

## 9. 风险与对策

| 风险 | 对策 |
|------|------|
| **PDF 版式错乱 / 扫描件无字** | P1 明确 **降级策略**（提示用户换 **可复制文本的 PDF** 或 **导出 txt**）。 |
| **检索不准 / 答非所问** | RAG 后加 **「不知则不说」** 的 system 约束；可选 **置信度阈值** 不注入。 |
| **token 爆量 / 费用** | 注入上限 + RAG Top-K + **冷却**（与自动回复联动）。 |
