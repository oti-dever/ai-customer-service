# 聚合会话「AI 辅助回复」方案设计与实现

本文档描述在**聚合接待界面**中，为客服提供 **AI 起草回复草稿** 能力的产品定案与实现要点：模型**仅写入输入框**，**不自动发送**给客户；客服确认或修改后**手动发送**。与 **「机器人管理」内置 AI 助手**（软件使用帮助）在场景、数据与配置上**相互独立**，可复用同一套 **OpenAI 兼容 HTTP 客户端**（如 `OpenAiCompatClient`）。

文件名建议与本文标题一致：`AI辅助回复方案设计与实现.md`。

**关联文档**：《AI助手集成方案设计与实现.md》（内置帮助助手；**豆包多模态与 `OpenAiCompatClient` 已与聚合共用网络层**）、《千牛RPA优化方案设计与实现.md》（聊天区 **截图入站** 与 PNG 保留策略；与本文 **多模态「生成本条回复」** 衔接）、**《AI自动回复方案设计与实现.md》（自动生成并发送；与本文辅助模式互补，详述千牛首版规则与触发，避免本文过长）**、《RPA-DB-协议.md》（聚合消息与 SQLite）、《需求分析与技术方案.md》（总体架构）。

## 1. 背景与目标

### 1.1 背景

- 业务主线为 **多平台会话聚合 + RPA 读写**；各平台侧通常无官方消息 API，会话内容经 RPA 进入本地库后在 **聚合界面** 展示与发送。
- 当前 RPA 消息链路以 **纯文本入站** 为主；**千牛** 已可按《千牛RPA优化方案设计与实现》配置 **`chat_content_mode=image_only`**，以 **聊天区 PNG 路径入站**（无聊天区 OCR）。聚合侧通过 **`content_image_path`** 区分 **纯文本** 与 **带截图**，「生成本条回复」据此走 **纯文本** 或 **多模态（豆包）**。
- **内置 AI 助手**（`RobotAssistantWidget`）服务「用户 ↔ 本软件」的使用问答，**不读取**当前客户会话；本方案针对 **客服 ↔ 客户** 的接待场景，二者 **模块与配置宜分离**，避免把客户上下文混进帮助助手会话。

### 1.2 第一版目标

在 **聚合接待模式**下，当操作者选择 **「AI 辅助」** 相关模式后：

1. 通过 **显式按钮**「生成本条回复」触发一次模型调用，将 **客户侧待回复的一条消息**与 **短知识块** 送入模型；
2. 模型输出以 **流式** 方式写入 **聚合输入框**，**不**触发发送、**不**写入出站 `messages`、**不**驱动 RPA 自动发话；
3. 客服对草稿 **可编辑**，再 **手动点击发送**，从而降低「从零打字」成本，同时 **责任仍在人**。

### 1.3 设计原则

- **人在回路**：默认不承诺全自动回复；是否上线「全自动」须待功能与风控稳定后 **单独立项**。
- **意图清晰**：不在切换会话、切换 Tab 时静默请求模型；仅在用户点击「生成本条回复」时发起调用，节省流量与费用。
- **失败可恢复**：请求失败或流中断时，**保留输入框内客服已输入、尚未发送的文本**，不得整段覆盖清空；可在状态栏或短提示中说明错误。
- **数据边界**：客户消息与知识片段会发往所选模型服务商；需在 UI 层做 **简要隐私说明**（与内置助手设置页风格一致）。

## 2. 范围与非目标

### 2.1 范围内（第一版 / MVP）

| 项 | 说明 |
|----|------|
| 入口形态 | 聚合界面在 **模式类下拉框** 中可选 **「AI 辅助」**；选中后，在 **输入区上方** 显示 **「模型：豆包 / DeepSeek」** 下拉（与机器人管理 **`ai/presets/*`** 预设一致），在 **发送按钮旁** 显示 **「生成本条回复」**（二者均仅在 **AI 辅助 + 已选会话 + 中心区为聊天页** 时可见）。左侧原 **「模拟消息」** 已改为 **「停止自动回复」**（与 **《AI自动回复方案设计与实现》§4.1** 一致，与 AI 辅助按钮独立）。 |
| 触发 | **仅**在点击「生成本条回复」时拉取上下文并调用模型；**不**自动轮询、**不**在切会话时预生成。 |
| 上下文 | **仅一条**：**客户侧、作为待回复对象的最近一条入站**——**文本** 或 **图片（本地路径，千牛截图方案）**（见 **§3**）。 |
| 知识库 | **MVP**：一篇短文档（客服话术 + 产品说明），以 **system 或固定 user 前置块** 注入，长度可控（见 **§4**）。 |
| 输出 | 流式写入 **聚合输入框**；**不**自动发送。 |
| 协议 | OpenAI 兼容 `chat/completions`，`stream: true` 时 SSE（与内置助手一致）。**若上下文为图片**：`user.content` 须为 **多段结构**（`image_url` + 可选 `text`），与《AI助手集成方案设计与实现》**§5.2** 一致；**须使用支持视觉的模型（如豆包/方舟接入点）**，纯文本模型（如 DeepSeek 默认 chat）不适用。 |

### 2.2 非目标（第一版不做或明确延后）

- **不**默认自动向客户发送模型生成内容；**不**在未点击「生成本条回复」时后台调用模型。
- **不**在第一版做 RAG / 向量检索（可放在后续演进）。
- **不**强制与内置助手共用同一 `model_key` 会话表；聚合辅助可不落库「草稿」行，除非后续要审计（若落库需单独表与保留策略）。
- **千牛聊天区截图入站 + 聚合多模态**：已与《千牛RPA优化方案设计与实现》**阶段 A～D** 衔接（**§3.4、§5.5～§5.6**）；**其他平台** 仍可按纯文本处理，不强制图片进模型。

## 3. 上下文：仅「客户一条消息」

### 3.1 定案

- **只取一条**：从当前选中会话中，取 **客户（对方）侧、入站方向** 的 **最近一条纯文本消息** 作为「待回复对象」。
- **原因**：客户对话碎片多、多条拼接难以统一质量与长度控制；先 **一条** 降低产品与技术复杂度，**后续若需要**再扩展为「最近 N 条」或 token 预算截断（单独立项）。

### 3.2 取值规则（建议实现时写死）

- 在本地 `messages` 中，按会话 `conversation_id` 过滤后，取 **时间序上最后一条** 满足以下条件的记录：
  - **方向为入站**（如 `direction = 'in'`）；
  - **有效载荷**：**非空文本**，或 **非空 `content_image_path`**（千牛聊天区截图路径）；二者满足其一即可（见 **`MessageDao::latestInboundSnapshot`**）。
- **若最后一条入站是客服自己发出的**（例如连续两条均为出站）：应 **向前查找最后一条入站客户消息**；若不存在，则 **提示「暂无客户消息可辅助」**，不发起请求。  
  - **当前实现**：**「生成本条回复」** 使用 **`latestInboundSnapshot`**（`content` + `content_image_path`）；**仅需要纯文本入站** 的场景仍可用 **`latestInboundContent`**（在 **有非空文本** 时返回正文，与 **§3.2** 纯文本表述兼容）。

### 3.3 与内置助手「十轮帽」的关系

- **无直接关系**：内置助手请求侧 **10 轮 user↔assistant** 约束见《AI助手集成方案设计与实现》**§10.2**；本方案 **仅一条客户消息 + 知识块 + 生成任务**，不参与该会话表。

### 3.4 图片入站（千牛聊天区截图方案）

**前提**：Reader 在 `chat_content_mode=image_only`（名称以《千牛RPA优化方案设计与实现》为准）下，将 **PNG 路径** 写入入站消息；`MessageDao` / 消息模型可 **用类型字段** 或 **约定 `content` 前缀** 区分文本与图片路径。

**定案**

- **取待回复对象**仍为 **时间序上最后一条客户入站**；若该条为 **图片**，则「生成本条回复」**不得**再走纯字符串 `user`。
- **请求体**：`system` 同 **§4**；`user` 使用 **多模态 `content` 数组**（至少含 **聊天区截图** 的 `image_url`，建议 **Base64 data URL** 与内置助手一致；可选附一句固定说明文字，如「图为客户最后一条消息截图，请生成回复草稿」）。
- **模型**：须为 **支持图片理解** 的端点（**火山方舟豆包接入点** 等）；若用户 `aggregateAi` 仍指向 **仅文本模型**，UI **提示切换模型** 或 **禁用按钮**（见 **§7**）。
- **与内置助手关系**：网络层、SSE、`content` 数组组装 **与 `RobotAssistantWidget` 豆包路径复用同一套能力**（见《AI助手集成方案设计与实现》**§5.6**），避免两套协议。

---

## 4. 知识库（MVP）

- **形态**：一份 **短**、**可控** 的「客服话术 + 产品说明」文本（可随版本放在 `:/` 资源、独立 txt 或配置中）。
- **注入方式**：作为 **`role: system`** 的一段，或 **`role: user`** 的固定前置说明块（二选一，以实现与模型表现为准）；**须与「客户那条消息」在 prompt 中角色区分清楚**，避免模型混淆「谁在说」。
- **后续**：多文档、版本管理、**RAG** 检索后再注入片段（与《技术选型》中长期规划对齐）。

## 5. 交互与 UI

### 5.1 模式与按钮

1. 用户在聚合界面 **模式类下拉框** 中选择 **「AI 辅助」**。
2. 选中后，在 **输入框上方** 显示 **「模型：…」**（`QToolButton` 下拉：**豆包（火山方舟）** / **DeepSeek**），所选值持久化在 **`QSettings` 键 `aggregateAi/sessionModelKey`**；API 配置与 **「机器人管理」→「设置」** 中对应预设一致（**`ai/presets/doubao_ark`**、**`ai/presets/deepseek_deepseek-chat`** 的 `baseUrl` / `model` / `apiKey`）。若预设 Key 为空，仍回退 **`aggregateAi/apiKey`、`ai/apiKey`** 等（与旧版单一路径兼容）。
3. 在 **发送按钮旁边** 显示 **「生成本条回复」**；未选 **AI 辅助** 或未打开会话时 **隐藏** 模型按钮与生成按钮。
4. **仅当**用户点击「生成本条回复」时，才组装请求（§3 + §4）并调用模型。
5. **输入区布局**：消息列表与底部 **输入条** 为 **叠放**（输入条 **半透明**），聊天记录可滚至输入条区域 **下方**，形成 **透过输入条看到消息** 的效果；列表底部为输入条高度预留 **底边距**，避免最后一条气泡被完全遮挡。

### 5.2 输入框行为

- 生成内容 **流式追加或覆盖策略**（建议：**从光标处替换整段草稿区域** 或 **追加到末尾**，产品可二选一并写死；首版可采用「清空本次 AI 草稿区再流式写入」但 **失败时保留人工已输入内容**，见 **§6**）。
- 客服可 **任意编辑** 输入框后再点 **发送**；发送路径与现有聚合发送、RPA 出站 **完全一致**，不新增「AI 专用发送」接口。

### 5.3 与「内置 AI 助手」的 UI 隔离

- 内置助手在 **「机器人管理」** Tab 内；本方案仅在 **聚合接待** 区域出现按钮与提示，避免两处入口混用。

### 5.4 模式下拉选项（与实现对齐）

| 顺序 | 文案 | 行为 |
|------|------|------|
| 0 | 人工接待 | 默认；不显示「生成本条回复」。 |
| 1 | AI 辅助 | 已选会话且中心区为聊天页时，显示「生成本条回复」；逻辑见 **§5.5**。 |
| 2 | AI 自动回复 | **全自动发送待实现**；文案为「AI自动回复」。首次从其他模式切入时会 **弹窗确认**（说明将接管千牛等平台会话的自动回复风险）；确认前 **不切换模式**。当前行为与「人工接待」类似，不显示「生成本条回复」。 |

> 已移除原「人工休息」项（与产品现状重复、使用少）。

### 5.5「生成本条回复」执行流程（与代码一致）

以下为点击 **「生成本条回复」** 后 `AggregateChatForm` 中的逻辑梳理，便于评审与排障。

**前置（不必点按钮即成立）**

1. 模式下拉当前为 **「AI 辅助」**（内部下标 `1`）。
2. 已选中会话且中心区展示 **聊天区**（`m_centerStack` 当前页为 `m_chatArea`），`m_currentConvId > 0`。
3. 上述条件同时满足时，按钮 **可见**；若正在生成中，按钮与发送、模式下拉等均按忙碌态处理（见下）。

**点击「生成本条回复」**

1. **防重入**：若已在生成中（`m_aggregateAiGenerating`），直接返回。
2. **读 API 配置**：按当前 **「模型：…」** 所选 **`aggregateAi/sessionModelKey`**，读取 **`ai/presets/<slug>`**（`slug` 为 `doubao:ark` → `doubao_ark` 等）中的 **`baseUrl`、`model`、`apiKey`**；若某字段为空，再回退 **`aggregateAi/*`、`ai/*`**（与仅填一套全局 Key 的旧用法兼容）。
3. **校验 Key**：若最终 API Key 仍为空 → **弹窗提示**（机器人管理预设或 `aggregateAi/apiKey`），**不发起请求**。
4. **取客户一条入站**：调用 **`MessageDao::latestInboundSnapshot(conversationId)`**（`direction='in'`，且 **文本 trim 非空** 或 **`content_image_path` 非空**，按 **`id DESC` 取 1 条）。  
   - 无记录，或 **既无文本又无本地可读图片** → **弹窗提示**「暂无可辅助的客户入站消息」，**不发起请求**。  
   - 若仅有 **图片路径** 但文件已删 → **提示截图文件不存在**。
5. **截图入站 + 非视觉线路**：若存在 **可用本地图**，且当前配置 **不像方舟多模态**（如 **DeepSeek**）→ **弹窗提示切换到豆包（火山方舟）** 并完成预设配置，**不发起请求**。
6. **冻结输入草稿基准**：`m_aggregateAiBaseline = m_inputEdit->toPlainText()`；`m_aggregateAiAccumulated` 清空。
7. **忙碌 UI**：`setAggregateAiBusy(true)` — 禁用 **发送**、**模式下拉**、**生成本条回复**、**模型选择**；输入框 **只读**。
8. **组装 `messages`**：  
   - `system`：`aggregateAiMvpSystemPrompt()`（含 **短答**、订单/商品类篇幅说明）；  
   - **仅有文本入站**：`user` 为单字符串（**【客户最后一条入站消息】** + 正文）；  
   - **含可读 PNG**：`user.content` 为 **数组**：**`image_url`（data URL，由 `imageFileToDataUrl` 读文件）** + **`text` 段**（占位文案或 OCR 文本说明）；**勿**把路径当纯文本塞进单段 `user`。  
9. **HTTP**：`OpenAiCompatClient::buildCompletionsUrl(baseUrl)` + **`requestChatCompletion(..., stream: true, extraRootFields)`**；`extraRootFields` 含 **`max_tokens`**（如 512）；对 **方舟 URL** 客户端自动附加 **`thinking: { type: "disabled" }`**（可被 `extraRootFields` 覆盖）。走 **SSE**。

**流式进行中**

- 每段 `streamDelta`：`m_aggregateAiAccumulated += delta`，`m_inputEdit` 显示为 **`baseline + accumulated`**（客服先前已打字内容保留在 `baseline` 前段，模型输出接在后面）。

**正常结束**

- `completed`：`setAggregateAiBusy(false)`，恢复发送/模式/按钮与输入可编辑。

**失败**（HTTP 错误、非 2xx、解析失败等）

- `failed(reason)`：在 **`baseline + 已生成片段`** 后 **追加**「（AI 未能完成 / 后续内容未能完成：…）」说明，再 `setAggregateAiBusy(false)`。  
- **不**清空 `baseline` 中用户原有内容。

**切换会话 / 进入空会话区 / 主动中断**

- **`abortAggregateAiRequest()`**：对 `OpenAiCompatClient::abortActive()`；若此前处于生成中，**强制结束忙碌态**（取消时网络层可能不发 `failed`，由 abort 路径统一收尾）。  
- 在 **`showConversation`**、**`showCenterEmptyState`** 开头调用，避免会话 A 的请求写到会话 B 的输入框。

### 5.6「生成本条回复」：最后一条为 **图片**（千牛截图入站）时

**前置**：千牛 RPA 已写入 **`content_image_path`**；聚合能取到 **`latestInboundSnapshot`** 且本地 **PNG 可读**。

**在 §5.5 通用步骤基础上**，图片分支要点如下：

1. **读取入站**：对 **`content_image_path`** 做 **`QFileInfo::isFile()`**；失败则提示 **截图文件不存在或无法读取**，**不发起请求**。
2. **模型校验**：依赖 **当前所选预设**（**豆包** vs **DeepSeek**）及 **Base URL / model 名启发式判断**（如 **DeepSeek** 视为不支持看图）；不满足则 **弹窗引导切换到「豆包（火山方舟）」** 并在机器人管理中配置方舟 **Base URL、接入点、Key**。
3. **组装 `messages`**：`system` 同 §5.5；`user` 的 `content` 为 **数组**：**`image_url`（data URL）** + **`text`（含【客户最后一条入站】与占位/ OCR 说明）**；与 **`RobotAssistantWidget`** 豆包发图 **同一套多段结构**。
4. **HTTP**：同 §5.5（含 **`thinking` 关闭**、**`max_tokens`**）。

**流式 / 失败 / 切换会话**：与 §5.5、§6 相同；**`m_aggregateAiBaseline`** 仍为点击前输入框内容。

## 6. 流式、失败与体验

- **流式**：推荐与 `OpenAiCompatClient` 一致，将增量内容写入输入框（或专用草稿缓冲再一次性提交，以实现为准）。
- **失败**（网络错误、非 2xx、流中断等）：
  - **必须保留** 输入框内 **客服在点击生成前已填写、尚未发送** 的文本；
  - 可在输入框末尾 **追加** 简短错误说明，或使用 **状态栏 / 一行提示**，避免误以为草稿全部丢失。
- **首版不提供「停止生成」** 可与内置助手策略一致；若后续增加，再对 `abort` 与 UI 状态单独立项。

## 7. 配置、密钥与模型

- **已实现**：聚合内 **「模型：豆包 / DeepSeek」** 与机器人管理 **`ai/presets/doubao_ark`、`deepseek_deepseek-chat`** **共用同一套 URL / model / Key**；另存 **`aggregateAi/sessionModelKey`** 记住用户在聚合中的选择。
- **兼容**：仍支持仅配置全局 **`aggregateAi/*` 或 `ai/*`**（未在预设中填写时回退）。
- **禁止**将 Key 写入源码与版本库。
- **多模态（千牛截图入站）**：处理图片时须在聚合中选 **豆包（火山方舟）** 并在预设中填写 **方舟 Base URL + 接入点 ID**；选 **DeepSeek** 且入站含图时会 **弹窗提示切换**。

## 8. 实现侧要点（概要，非排期承诺）

| 模块 | 说明 |
|------|------|
| 上下文查询 | **`MessageDao::latestInboundSnapshot(conversationId)`**：最后一条入站（文本和/或 **`content_image_path`**）；**`latestInboundContent`** 仍用于「仅非空文本」场景。 |
| 网络 | **`OpenAiCompatClient`**：`requestChatCompletion` 支持 **`extraRootFields`**（如 **`max_tokens`**）；方舟域名自动 **`thinking: { type: "disabled" }`**。 |
| 读图 | **`imageFileToDataUrl`**（`src/utils/imagedataurl.*`），与内置助手读图逻辑一致。 |
| 配置 | **`aggregateAi/sessionModelKey`** + **`ai/presets/*`**；回退 **`aggregateAi/*`、`ai/*`**。 |
| UI | **`AggregateChatForm`**：模式 **人工接待 / AI 辅助 / AI 自动回复**；后两者在「生成本条回复」上见 **§5.1**；**AI 自动回复** 切入时 **确认弹窗**；底部输入区 **半透明叠层**。 |
| 数据 | AI 草稿 **不**写入 `messages`（仍以输入框为准）。 |
| 多模态 | 入站含 **`content_image_path`** 且文件可读 → **`user.content` 数组**（**§3.4、§5.6**）。 |
