## 一、文档定位

本文档用于定义 `yy-ai-customer-service` 下一阶段的 UI 收敛方向：从“多主题、多强调、页面局部样式较多”的现状，过渡到“单一默认主题、简约克制、信息优先”的统一视觉体系。

它是对现有 `docs/UI设计规范.md` 的补充，不直接替代原文档。原规范更偏“全量界面能力描述”，本文档更偏“视觉收敛与产品化统一”。

参考灵感来源：

- 微信内容页/信息卡片的轻量信息层级与留白感：[微信文章参考](https://mp.weixin.qq.com/s/piyZWbyU-VLEeZm22ZxK2A)

---

## 二、改造目标

### 2.1 总目标

将当前界面从“功能表达充分但视觉元素偏多”的后台工具风格，收敛为：

- 只保留一个默认主题
- 更少颜色、更少装饰、更少视觉干扰
- 更统一的页面层级、控件语言和状态表达
- 更突出主工作区，而不是突出控件本身

### 2.2 本轮不追求的内容

以下内容不属于本规范的目标：

- 不改业务信息架构
- 不新增复杂动效
- 不引入新的 UI 技术栈（仍以 `Qt Widgets + QSS` 为主）
- 不为了“高级感”增加渐变、玻璃拟态、厚重阴影等高维护成本效果

---

## 三、核心设计原则

### 3.1 单主题

系统只保留一个默认主题，不再提供：

- `Cool`
- `Warm`
- 主题切换按钮
- 同一组件在不同主题下的多套色值分支

收益：

- 降低维护复杂度
- 降低页面之间的风格漂移
- 降低测试和后续重构成本

### 3.2 单强调

同一屏内只允许一个主强调点。

表现方式：

- 当前选中导航项
- 当前主操作按钮
- 当前关键状态

不应同时使用多种高饱和色来争夺注意力。

### 3.3 内容优先

视觉层级应服务于信息阅读和主任务完成，而不是服务于“界面丰富度”。

优先级应始终是：

1. 主内容
2. 当前操作
3. 上下文信息
4. 装饰性元素

### 3.4 控件统一

按钮、输入框、标签、弹窗、列表项、空状态等基础控件必须遵循同一套视觉规则，不允许各页面“各修各的样式补丁”。

### 3.5 低噪音

收敛以下视觉噪音来源：

- 多余边框
- 过重阴影
- 过多块状背景
- 过多图标
- 过多颜色类型
- 非高频操作常驻首屏

---

## 四、默认主题设计语言

### 4.1 整体气质

默认主题应体现如下关键词：

- 简洁
- 克制
- 安静
- 稳定
- 专业

目标感受应更接近“产品化工作台”，而不是“堆满功能按钮的管理后台”。

### 4.2 基础视觉模型

建议采用：

- 浅灰主背景
- 白色内容容器
- 深灰文字
- 极浅描边
- 一套低饱和蓝色主强调

视觉上遵循：

- 大面统一
- 局部点亮
- 靠留白分层
- 靠文字权重和间距分层
- 少靠颜色和装饰分层

## 五、设计 Token（默认主题）

以下为建议值，后续可在实现前做一次微调，但不应偏离本结构。

### 5.1 颜色

| Token | 值 | 用途 |
| --- | --- | --- |
| `bg_app` | `#F5F5F7` | 应用全局背景 |
| `bg_panel` | `#FFFFFF` | 主内容面板/卡片背景 |
| `bg_panel_soft` | `#FAFAFB` | 次级容器、浅层区块 |
| `bg_hover` | `#F0F2F5` | 列表项/按钮 hover |
| `bg_selected` | `#EAF2FF` | 当前选中项 |
| `text_primary` | `#18181B` | 主文字 |
| `text_secondary` | `#71717A` | 次级文字 |
| `text_tertiary` | `#A1A1AA` | 更弱辅助信息 |
| `border_default` | `#E4E4E7` | 默认边框/分割线 |
| `border_strong` | `#D4D4D8` | 需要强调的边界 |
| `accent_primary` | `#2563EB` | 主强调色 |
| `accent_primary_hover` | `#1D4ED8` | 主强调 hover |
| `success` | `#16A34A` | 成功状态 |
| `warning` | `#D97706` | 警告状态 |
| `danger` | `#DC2626` | 错误/危险状态 |

### 5.2 间距

统一采用以下节奏：

- `4`
- `8`
- `12`
- `16`
- `24`
- `32`

使用原则：

- 控件内边距优先用 `8 / 12`
- 模块内元素间距优先用 `12 / 16`
- 模块之间优先用 `24 / 32`

### 5.3 圆角

统一只保留两档：

- `radius_sm = 8px`
- `radius_md = 12px`

使用原则：

- 按钮、输入框、小容器：`8px`
- 卡片、面板、大容器：`12px`

避免在同一页面出现过多圆角档位。

### 5.4 字号

统一为三层主层级：

| Token | 值 | 用途 |
| --- | --- | --- |
| `font_title` | `18px / 20px` | 页面标题、核心标题 |
| `font_base` | `14px` | 正文、按钮、输入框 |
| `font_meta` | `12px` | 辅助文案、说明、标签 |

额外约束：

- 页面中不要同时出现过多字号
- 避免为了“强调”同时放大字号和加粗和上色

## 六、基础控件规范

### 6.1 按钮

按钮只保留三类：

#### 主按钮

- 背景：`accent_primary`
- 文字：白色
- 圆角：`8px`
- 用途：当前页面唯一主操作

#### 次按钮

- 背景：白色
- 边框：`border_default`
- 文字：`text_primary`
- 用途：取消、返回、次要动作

#### 文字按钮 / 轻按钮

- 无重背景
- 用于低频辅助动作

禁止事项：

- 同一页面并列多个主按钮
- 过多彩色按钮并列
- 用不同颜色表达“不同功能模块”

### 6.2 输入框

- 高度保持统一
- 白底
- 1px 细边框
- 聚焦时仅边框变为主色
- 不使用重投影

### 6.3 标签与状态

标签应更像“辅助说明”，而不是“迷你按钮”。

建议：

- 使用浅底 + 低饱和文字
- 限制颜色种类
- 不做强立体感

### 6.4 弹窗

所有弹窗统一为：

- 浅背景
- 深色文字
- 标题、正文、按钮层级统一
- 不做页面专属弹窗皮肤

## 七、主窗口规范

对应文件：

- `src/ui/mainwindow.cpp`
- `src/utils/ApplyStyle.cpp`

### 7.1 顶栏

目标：瘦身、降噪、保留核心操作。

建议保留：

- 应用标题
- 添加窗口
- 帮助
- 用户相关入口

建议后置或合并：

- 主题切换
- Bug 日志
- 置顶
- 低频管理类入口

视觉规则：

- 顶栏高度适中
- 不堆叠太多独立按钮
- 状态信息不要做成强视觉中心

### 7.2 左侧导航

目标：从“重后台侧栏”变为“轻目录导航”。

建议：

- 背景弱化，不做过重深色压迫
- 分组标题更轻
- 当前项高亮即可，其余项保持安静
- 图标仅作辅助，不作为主识别信息

选中态建议：

- 浅色选中底
- 左侧细强调条或轻描边
- 不再使用过多重色块

### 7.3 首页 / 就绪页

目标：从“视觉展示页”收敛为“工作台欢迎态”。

建议：

- 保留欢迎与状态信息
- 卡片数量控制
- 减少装饰性图标/大面积包装
- 首屏把“下一步可以做什么”讲清楚，而不是堆样式

### 7.4 状态栏

目标：存在感降低，但信息清晰。

建议：

- 做成细条，不要厚重
- 只展示必要状态
- 不使用太多颜色分类

## 八、聚合接待页规范

对应文件：

- `src/ui/aggregatechatform.cpp`

### 8.1 页面定位

聚合接待页是核心业务页，视觉中心必须是：

- 会话列表
- 当前聊天内容
- 当前输入与发送动作

其它元素都必须降级。

### 8.2 三栏权重

建议权重：

- 左：会话导航
- 中：主聊天区
- 右：辅助详情

其中中间聊天区必须最强，右侧详情只能作为补充。

### 8.3 会话列表

建议：

- 列表项更平、更简洁
- 只保留必要信息：客户名、摘要、时间、未读
- 待处理/全部切换做轻量化表达
- 搜索框不做重装饰

### 8.4 聊天区

建议：

- 聊天气泡、输入区、状态提示保持统一风格
- 输入区清晰但不过度突出
- 主发送按钮是唯一强按钮

### 8.5 AI 相关控件

建议：

- AI 辅助、模型选择、自动回复是“能力层”，不是“视觉主角”
- 不应让 AI 控件比消息内容更显眼
- 模型选择更适合做轻入口或次级菜单
- 发送轨迹、调试信息默认折叠或弱展示

---

## 九、机器人助手页规范

对应文件：

- `src/ui/robotassistantwidget.cpp`

### 9.1 页面定位

机器人助手页应呈现为“单线程对话产品”，而不是“设置面板 + 功能集合页”。

### 9.2 结构原则

默认焦点应放在：

1. 历史对话
2. 当前输入
3. 当前回复状态

设置、模型、附件能力应后置。

### 9.3 设置入口

建议方向：

- 不让设置页抢主界面注意力
- 可以继续保留设置能力，但视觉层级降低
- 对话页与设置页的切换不应破坏整体简洁感

### 9.4 对话体验

建议：

- 气泡样式与聚合接待保持统一语言
- 附件、多模态只作为增强能力呈现
- 输入区和发送按钮语义与聚合接待一致

## 十、实现层约束

### 10.1 `ApplyStyle` 收敛方向

后续代码实现时，应将：

- `Default / Cool / Warm`

收敛为：

- 单一 `Default`

并逐步删减：

- 主题保存/读取逻辑
- 主题切换按钮
- 多套 token 分支
- 页面级 theme 参数透传

### 10.2 禁止新增的模式

在进入代码实现后，应避免继续增加：

- 页面局部临时 QSS 补丁
- 单页私有颜色常量泛滥
- 同一种控件多套视觉定义
- 为了“好看”叠加新的颜色分支

### 10.3 推荐方式

推荐：

- 设计 token 集中定义
- 页面级组件复用统一风格
- 颜色和字号先归一，再做局部微调

不推荐：

- 先改单页，再补整体规则
- 通过大量局部 if/else 维持多个视觉模式

---

## 十一、分阶段落地建议

### 阶段 A：视觉收敛

- 删除多主题设计决策
- 确认默认主题 token
- 确认按钮/输入框/弹窗统一规则

### 阶段 B：主窗口减法

- 顶栏入口瘦身
- 左侧导航减重
- 首页卡片减法
- 状态栏弱化

### 阶段 C：业务页统一

- 聚合接待与机器人助手统一为一套内容容器风格
- AI 控件降级为次级视觉层
- 统一弹窗与提示样式

### 阶段 D：清理遗留

- 删除不再使用的主题分支
- 删除旧样式常量
- 清理重复 QSS 片段

---

## 十二、验收标准

完成默认主题收敛后，应达到：

- 用户只能看到一个默认主题
- 页面中的颜色数量明显减少
- 顶栏和侧栏不再“抢戏”
- 聚合接待与机器人助手在视觉语言上显著统一
- AI 相关控件不再压过聊天内容
- 样式修改主要集中在统一 token 和基础组件层，而不是散落在页面局部

## 十四、默认主题色板与组件表

本节将前文的视觉原则进一步细化为可直接映射到 `ApplyStyle` 与页面控件的组件级规则。

### 14.1 页面背景分层

| 层级 | 建议值 | 用途 |
| --- | --- | --- |
| 应用底色 | `#F5F5F7` | 整体窗口底色 |
| 一级容器 | `#FFFFFF` | 主工作区、主卡片 |
| 二级容器 | `#FAFAFB` | 次级面板、辅助区 |
| hover 层 | `#F0F2F5` | 列表项、轻按钮悬停 |
| 选中层 | `#EAF2FF` | 当前导航、当前切换项 |

规则：

- 同一页面中，容器背景层级最多保留三层。
- 不再为不同业务模块赋予不同底色。
- 用浅层级差异替代重边框或深色块。

### 14.2 文字系统

| 类型 | 建议值 | 场景 |
| --- | --- | --- |
| 页面标题 | `18px` / `600` | 页标题、主标题 |
| 区块标题 | `14px` / `600` | 面板标题、分组标题 |
| 正文 | `14px` / `400` | 会话摘要、正文、按钮 |
| 辅助文字 | `12px` / `400` | 时间、说明、状态补充 |

颜色建议：

- 主文字：`#18181B`
- 次文字：`#71717A`
- 弱文字：`#A1A1AA`

规则：

- 不通过大量不同颜色来区分信息优先级。
- 优先使用字号、粗细、留白来建立层次。

### 14.3 按钮系统

#### 主按钮

| 项 | 规则 |
| --- | --- |
| 背景 | `#2563EB` |
| hover | `#1D4ED8` |
| 文字 | `#FFFFFF` |
| 圆角 | `8px` |
| 高度 | 统一 |

使用约束：

- 一屏只允许一个主按钮承担主操作。
- 发送、确认、保存等才可使用主按钮。

#### 次按钮

| 项 | 规则 |
| --- | --- |
| 背景 | `#FFFFFF` |
| 边框 | `#E4E4E7` |
| 文字 | `#18181B` |
| hover | `#F5F5F7` |

适用场景：

- 返回
- 取消
- 次要切换
- 辅助动作

#### 轻按钮 / 图标按钮

规则：

- 默认透明或浅底
- 只在 hover 时出现轻背景
- 不做重描边
- 不作为页面主视觉点

### 14.4 输入框系统

| 项 | 规则 |
| --- | --- |
| 背景 | `#FFFFFF` |
| 边框 | `1px solid #E4E4E7` |
| 聚焦 | 边框变为 `#2563EB` |
| 圆角 | `8px` |
| 文本 | `#18181B` |
| 占位符 | `#A1A1AA` |

规则：

- 搜索框和普通输入框遵循同一结构。
- 不单独设计“花样搜索框”。
- 视觉焦点来自输入内容，而不是装饰性图标。

### 14.5 列表项系统

适用范围：

- 平台树
- 会话列表
- 菜单项
- 设置项

规则：

- 默认背景透明或接近透明
- hover 使用轻浅底色
- 选中仅使用一套选中态
- 未选中项不做重边框

建议选中态：

- 背景：`#EAF2FF`
- 左侧强调条：`#2563EB`
- 文字：`#18181B`

### 14.6 标签与状态块

建议将现有“彩色业务标签”收敛为两类：

1. 信息标签
2. 状态标签

信息标签：

- 浅灰底
- 深灰字
- 不抢注意力

状态标签：

- 成功、警告、错误三类
- 仅在确有语义时使用颜色
- 不作为装饰元素滥用

### 14.7 弹窗与确认框

统一规则：

- 背景统一为浅色
- 标题与正文保持统一排版
- 底部按钮最多一个主按钮 + 一个次按钮
- 警告场景依然保持简洁，不做强烈视觉冲击

不建议：

- 每个页面自己定义一套 `QMessageBox` 局部 QSS
- 用弹窗颜色去区分业务线

## 十五、线框稿说明

本节用于把“简约化”落到页面结构层面，明确哪些区域保留、哪些区域弱化、哪些入口后置。

### 15.1 主窗口线框

```plain
+--------------------------------------------------------------+
| Logo | 应用标题 |                         帮助 | 用户入口      |
+--------------------------------------------------------------+
| 轻量导航区 |                                             |
| - 工作台   |               主工作区                      |
| - 聚合接待 |                                             |
| - 机器人助理|                                             |
| - 平台管理 |                                             |
|            |                                             |
+--------------------------------------------------------------+
| 状态栏：简短状态 / 时间 / 运行提示                            |
+--------------------------------------------------------------+
```

#### 主窗口收敛规则

保留：

- 应用标题
- 核心导航
- 主工作区
- 状态栏

弱化：

- 顶栏图标密度
- 系统就绪卡片的装饰性表达
- 低频功能的常驻入口

后置：

- 主题切换
- Bug 日志
- 置顶等偏系统级操作

### 15.2 聚合接待页线框

```plain
+------------------+--------------------------------+------------------+
| 会话筛选与列表   | 当前会话内容                   | 辅助详情         |
| - 搜索           | - 顶部会话标题                 | - 客户信息       |
| - 待处理/全部    | - 消息区                       | - 当前状态       |
| - 会话项列表     | - 输入区                       | - 次级操作       |
|                  | - 主发送按钮                   |                  |
+------------------+--------------------------------+------------------+
```

#### 聚合接待页收敛规则

左栏：

- 保留搜索、筛选、会话列表
- 会话项只展示核心信息
- 减少大色块与多装饰图标

中栏：

- 聊天内容始终为视觉中心
- 输入框区域清晰但不过度突出
- AI 草稿/模型选择为次级操作

右栏：

- 仅承载辅助信息
- 发送轨迹、调试信息默认收敛
- 不与聊天内容争视觉层级

### 15.3 机器人助手页线框

```plain
+--------------------------------------------------------------+
| 助手标题                                      设置 / 模型入口 |
+--------------------------------------------------------------+
|                                                              |
|                        对话消息区                            |
|                                                              |
|--------------------------------------------------------------|
| 附件区（按需出现）                                           |
| 输入框                                              发送按钮  |
+--------------------------------------------------------------+
```

#### 机器人助手页收敛规则

保留：

- 对话历史
- 输入区
- 必要的模型入口
- 按需出现的附件区

弱化：

- 设置页存在感
- 模型与参数的视觉权重
- 非当前任务相关的提示块

建议：

- 将设置视为辅助入口，而不是主页面的一半
- 让用户进入页面后首先看到对话，而不是配置项

### 15.4 页面之间的统一关系

主窗口、聚合接待、机器人助手应共享以下视觉语言：

- 相同的背景层级
- 相同的按钮体系
- 相同的输入框风格
- 相同的弹窗风格
- 相同的标题与辅助文字节奏

这样用户在页面切换时，感受到的是“同一产品下的不同工作流”，而不是“多个子系统拼接”。

## 十六、进入实现前的确认清单

在进入代码改造前，建议先确认以下事项：

### 16.1 主窗口

- 顶栏最终保留哪些按钮

- 设置入口是否继续用 tab

## 十七、交互细节与状态反馈规范

本节用于补充“简约化”之外的另一条主线：通过统一的交互反馈和视觉线索，提升可用性与操作确定感。

参考方向：

- QSS 伪状态在交互反馈中的应用：[QSS 深度解析](https://www.cnblogs.com/yangykaifa/p/19671266)
- QSS 工程化与统一样式组织：[QSS 实战指南](https://cloud.tencent.com/developer/article/2618853)
- 动态属性选择器用于状态表达：[动态属性与交互式样式切换](https://blog.csdn.net/mac99/article/details/151383827)

### 17.1 基本原则

交互反馈应遵循以下原则：

- 反馈必须即时
- 反馈必须克制
- 反馈必须一致
- 反馈必须服务于任务完成，而不是制造视觉热闹

本项目后续不追求“动画感很强”的界面，而追求：

- 用户一看就知道哪里能点
- 点下去后立刻知道是否生效
- 当前选中、当前焦点、当前忙碌、当前错误都表达清楚

### 17.2 可点击元素状态规范

适用范围：

- 普通按钮
- 图标按钮
- 左侧导航项
- 会话列表项
- Tab 项
- 菜单项

所有可点击元素至少应定义以下状态：

#### 默认态

- 保持简洁
- 不用重边框或高饱和背景
- 通过版式和位置表达“这是操作元素”

#### 悬停态 `:hover`

- 仅做轻度强化
- 建议通过浅底色、轻微边框变化、文字色略增强体现
- 不应在 hover 时产生大幅度跳变

#### 按下态 `:pressed`

- 颜色或明度略深于 hover
- 让用户确认“已按下”
- 不做夸张位移或重阴影

#### 禁用态 `:disabled`

- 明确不可点击
- 降低对比度
- 禁止与“普通未选中态”视觉上太接近

### 17.3 输入焦点规范

适用范围：

- 搜索框
- 单行输入框
- 多行文本框
- 聊天输入区

焦点态应统一表达为：

- 边框高亮
- 背景保持稳定
- 不使用花哨 glow 或多重颜色变化

建议：

- 默认边框：`border_default`
- 焦点边框：`accent_primary`
- 禁止每个页面单独定义不同焦点风格

输入体验目标：

- 用户始终知道“当前键盘输入会落到哪里”
- 焦点变化不打断阅读

### 17.4 选中态规范

选中态是本项目最重要的视觉线索之一，应重点统一。

适用范围：

- 左侧平台树
- 会话列表
- 当前功能页
- 模型选择项
- 轻量切换项

统一建议：

- 使用浅色选中底
- 搭配单一主强调色
- 必要时加左侧细条
- 不再对不同页面使用不同选中语义

不推荐：

- 某些选中项用蓝底
- 某些选中项用深灰底
- 某些选中项只加粗
- 某些选中项再单独加图标高亮

### 17.5 忙碌态与进行中状态

本项目有较多“等待系统完成”的操作，必须建立统一忙碌态规范。

适用场景：

- AI 回复生成中
- 自动回复执行中
- RPA 启动/停止中
- 文件上传与解析中
- 数据加载中

建议规则：

- 按钮禁用并显示明确状态文案
- 输入区在必要时只读
- 页面内使用轻量状态提示，而不是大面积遮挡
- 若耗时超过短阈值，应展示持续中的文案或微型加载标识

原则：

- 不要让用户猜“系统是不是卡住了”
- 也不要用过于强烈的 loading 破坏页面安静感

### 17.6 错误、警告与成功反馈

反馈层级建议统一为三档：

#### 成功反馈

- 轻量提示即可
- 不建议大面积绿色块
- 对已完成操作给出简短确认

#### 警告反馈

- 用于提醒用户注意，但不阻断任务
- 颜色使用应克制
- 文案应强调下一步怎么做

#### 错误反馈

- 用于操作失败或配置缺失
- 重点应放在“失败原因 + 如何修正”
- 不依赖大红色背景制造压力

统一规则：

- 提示语气简短直接
- 同类问题使用统一文案结构
- 弹窗、内联提示、状态栏提示各自职责明确

### 17.7 视觉线索层级

界面中的视觉线索建议统一成以下优先顺序：

1. 当前选中
2. 当前焦点
3. 当前主操作
4. 当前异常
5. 次级辅助信息

目的：

- 避免一屏里同时出现多个“最重要”
- 避免主内容、错误提示、AI 状态、按钮都在抢注意力

### 17.8 动态属性建议

在 Qt/QSS 实现层，建议逐步把业务状态映射为动态属性，而不是散落在页面局部 if/else 中手工拼样式。

推荐的属性语义示例：

- `selected`
- `active`
- `pending`
- `unread`
- `busy`
- `danger`
- `warning`
- `compact`

建议用途：

- 左侧导航选中项
- 会话列表未读与当前会话
- AI 生成中按钮或状态块
- 错误态输入项或提示块

收益：

- 样式跟状态语义绑定，而不是跟页面绑定
- 后续统一替换样式时，变更面更集中

### 17.9 页面级交互建议

#### 主窗口

- 顶栏图标 hover/pressed 必须一致
- 左侧树当前项与 hover 项清晰区分
- 低频按钮不应和主按钮同等级强调

#### 聚合接待

- 当前会话必须一眼可识别
- 输入框聚焦态明确
- 发送按钮、AI 草稿按钮、停止按钮应有明确层级差
- 发送失败、自动回复中、模型不可用等状态应有统一反馈方式

#### 机器人助手

- 对话输入、发送、附件选择应形成连续操作流
- 生成中状态应明确，但不打断阅读
- 设置入口与当前会话流不应互相抢焦点

### 17.10 验收标准

交互细节规范落地后，应达到：

- 所有主要可点击元素都具有统一的 hover/pressed/disabled 表达
- 用户能明确辨认当前选中项与当前输入焦点
- AI/RPA 等耗时操作都有清晰但克制的忙碌反馈
- 错误提示具有统一结构，不再依赖页面局部补丁
- QSS 与动态属性更多承担状态表达职责，减少页面内零散样式判断

- 不同页面各自定义提示外观

2. 统一成功/警告/错误文案结构
3. 尽量减少页面私有 `QMessageBox` QSS
4. 优先建立公共反馈规则，而不是继续加页面补丁

建议后续抽象方向：

- 统一轻量提示函数
- 统一确认弹窗规则
- 统一状态栏提示与内联提示职责边界

### 18.6 动态属性实施建议

建议从以下区域开始逐步引入动态属性：

| 区域 | 属性建议 | 作用 |
| --- | --- | --- |
| 左侧导航项 | `selected`, `active` | 当前页、当前平台 |
| 会话列表项 | `selected`, `unread`, `pending` | 当前会话、未读、待处理 |
| AI 控件 | `busy`, `disabled` | 生成中、不可用 |
| 状态块 | `warning`, `danger`, `success` | 当前状态表达 |
| 输入区 | `focused`, `invalid` | 焦点、输入异常 |

实施原则：

- 属性名体现语义，而不是体现页面来源
- 样式由属性驱动，不由页面局部硬编码驱动

### 18.7 推荐改造顺序

为避免返工，建议严格按下面顺序实施：

1. `ApplyStyle`

   - 固定默认 token
   - 统一状态样式
2. `MainWindow`
   - 顶栏瘦身
   - 左侧导航减重
   - 首页/状态栏降噪
3. `AggregateChatForm`
   - 会话列表与聊天区层级重构
   - AI 控件降级
   - 右栏辅助化
4. `RobotAssistantWidget`
   - 对话主流程强化
   - 设置与附件能力后置
   - 状态反馈统一
5. 提示系统与遗留补丁清理
   - 移除局部消息框样式分叉
