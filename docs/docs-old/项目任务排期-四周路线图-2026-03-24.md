# 项目任务排期（四周路线图）

> 日期：2026-03-24  
> 目标：将“多平台会话聚合 + 统一收发 + 可接入 AI 自动化”拆成可执行、可验收的短周期任务。

---

## 0. 当前基线（截至 2026-03-24）

- 千牛 RPA：单会话读写链路已打通（Reader + Writer 可用）。
- 文档已收敛：`千牛RPA实现方案` 已精简并记录当前实现状态。
- 当前阶段重点：从“能跑通”进入“可稳定使用”。

---

## 1. 总体节奏（4 周）

- 第 1 周：千牛链路稳定化（发送确认、失败恢复、日志可追踪）
- 第 2 周：单平台多会话能力（调度、防串会话、优先级）
- 第 3 周：多平台统一编排（抽象统一、流程打通）
- 第 4 周：AI 自动回复接入准备（风控闸门 + 人工接管）

---

## 2. 第 1 周（千牛稳定化）

### 2.1 周目标

- 将千牛收发从“可用”提升到“稳定可复现”。

### 2.2 任务清单

- 增加发送后回执校验（输入框清空或末尾气泡变化）。
- 完善失败重试策略（重试次数、重试间隔、失败分类）。
- 增加死信记录机制（超过重试上限后保留失败上下文）。
- 统一日志字段：`msg_id / conv_id / target / attempt / error_code`。
- 固化一份可回滚配置快照（千牛窗口、OCR 区域、writer 参数）。

### 2.3 验收标准

- 连续 50 条消息发送成功率 >= 95%。
- 失败消息 100% 有明确 `ERR_*` 原因。
- 不出现“误发到错误会话”的问题。

---

## 3. 第 2 周（单平台多会话）

### 3.1 周目标

- 支持千牛多会话连续处理，保证不串会话、不漏会话。

### 3.2 任务清单

- 设计会话调度规则（轮询、优先级、超时让出）。
- 增加会话切换状态机（切换前校验、切换后确认、失败回退）。
- 优化 Reader 多会话识别（标题 OCR 稳定器、去抖动）。
- 增加“会话繁忙/锁定”机制，防止读写冲突。

### 3.3 验收标准

- 3~5 个会话交替收发，30 分钟内无串会话。
- 会话切换失败可自动恢复，不出现卡死循环。

---

## 4. 第 3 周（多平台统一编排）

### 4.1 周目标

- 打通“千牛 + 微信（或第二平台）”统一收发流程，建立平台无关的调度层。

### 4.2 任务清单

- 统一消息模型（平台字段、会话字段、状态字段）。
- 抽象平台能力接口（读取、发送、会话校验、错误码映射）。
- 统一任务队列与重试策略（跨平台一致）。
- 增加平台级健康状态（Reader/Writer 心跳、最近成功时间）。

### 4.3 验收标准

- 至少 2 个平台在同一聚合界面可稳定收发。
- 跨平台日志可按 `platform + conv_id + msg_id` 一键追踪。

---

## 5. 第 4 周（AI 自动化接入准备）

### 5.1 周目标

- 建立 AI 自动回复的“安全可控”接入框架，不追求一步到位全自动。

### 5.2 任务清单

- 设计 AI 回复触发条件（关键词、置信度、会话类型）。
- 增加风控闸门（敏感词、金额、售后争议场景转人工）。
- 增加人工接管机制（一键暂停自动回复、手动优先）。
- 定义 AI 回复审计字段（prompt、response、decision、operator）。

### 5.3 验收标准

- AI 仅在白名单场景自动回复，风险场景强制人工确认。
- 每条自动回复可追溯“为何发送、谁批准、何时发送”。

---

## 6. 跨周持续任务（每周都做）

- 每周一次稳定性回归（收发、切会话、异常恢复）。
- 每周更新一次“问题清单与修复状态”。
- 每周冻结一个可回滚版本（代码 + 配置 + 文档）。

---

## 7. 风险与应对

- 无 API 依赖 OCR：通过“校准 + 回执 + 重试 + 监控”降低风险。
- Windows 前台限制：保持后台发送优先，前台回退兜底。
- UI 波动导致识别偏差：保留手动校准入口与配置版本管理。
